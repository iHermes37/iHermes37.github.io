Solana合约重点

重点 核心原理与解释
程序与状态分离 根本性差异。在 Solana 上，可执行代码（程序）和它管理的数据（状态）存储在独立的账户中。一个程序可以管理成千上万个数据账户。必须忘掉“合约包含自身状态”的EVM模型。
账户是核心抽象 一切皆是账户。程序是账户，数据是账户，代币也是账户。开发就是定义账户结构和管理账户之间交互规则的过程。
交易是指令清单 用户发送的交易包含一个或多个指令，每个指令指定要调用的程序和所需的账户。这允许单个交易跨多个程序组合操作，是实现高性能可组合性的基础。
  1. 设计阶段：首先问自己：“我的程序账户是什么？它要管理哪些数据账户？这些数据账户的结构如何？”
  2. 开发阶段：
    • 用 Anchor：从第一天起就使用 Anchor 框架。
    • 验证一切：在 #[derive(Accounts)] 结构体中，为每个账户写上最严格的约束。
    • PDA 优先：默认使用 PDA 来存储程序状态，它更安全。
    • 考虑租金：为每个数据账户设计创建和销毁的生命周期。
  3. 测试阶段：编写覆盖所有指令和边缘情况的集成测试，特别是针对恶意账户输入的测试。

安全基石

重点 核心原理与解释 安全关联
账户验证 安全生命线！ 程序必须对传入的每一个账户进行严格验证，包括： • 签名检查：调用者是否授权 (is_signer)。 • 程序所有权：账户是否由正确的程序拥有 (owner)。 • 数据类型：账户数据是否符合预期结构。 绝大多数漏洞源于此。 未经验证的账户是黑客的入口。
PDA Solana 的超级功能。 程序派生地址是由程序（而非私钥）控制的地址。用于： • 安全存储状态：作为程序专属的数据账户。 •权限控制：代表程序对其它账户进行签名（如转移代币）。 必须精通。 错误使用 PDA 会导致资金丢失或权限漏洞。
租金的考量 账户需要支付租金才能永久存储数据。程序设计必须包含： • 租金豁免计算：创建账户时存入足够 SOL。 • 账户关闭逻辑：允许用户关闭账户并收回租金。 没有关闭指令会导致用户资金被永久锁定，体验极差。
计算预算 每个交易有计算上限。必须优化逻辑，避免复杂循环，并使用 sol_log_compute_units() 监控用量，防止交易失败。 超出预算会导致交易回滚。
跨程序调用 程序可以调用其他程序，形成乐高式的组合。需要正确构造被调用程序所需的账户列表和指令数据。 错误的 CPI 是主要漏洞来源，尤其是权限传递。

PDA

PDA（Program Derived Address）是 Solana 中最独特和重要的概念之一。

传统账户 PDA（程序派生地址）
需要私钥签名 不需要私钥
用户控制 程序控制
类似：个人银行账户 类似：智能合约托管账户
  1. 无密钥账户 - 不需要私钥就能控制的地址
  2. 程序控制 - 只有创建程序可以签名
  3. 确定性 - 通过种子确定性生成
  4. 安全隔离 - 完美实现数据隔离和权限控制

解决了”谁来控制程序数据”的问题，实现了真正意义上的程序自治并提供了安全可控的状态管理方案

核心特性

确定性生成

PDA 通过程序ID和种子确定性生成

无私钥控制

PDA 没有对应的私钥，只有创建它的程序可以签名，这通过 CPI（跨程序调用）中的 PDA 签名实现

程序专属

每个 PDA 都绑定到特定的程序，只有该程序可以为 PDA 签名

工作原理

生成算法

PDA = hash(种子 + 程序ID) 但不在椭圆曲线上

特点：不在 ed25519 曲线上（所以没有私钥），通过 bump（一个字节）来确保不在曲线上

基本构件

要点 说明
#[program] 这是你的智能合约。在这里定义所有供用户调用的指令函数。Anchor 会自动处理路由，将传入的指令分发到对应的函数。
Accounts Struct 这是核心！ 每个指令函数第一个参数都是一个自定义的结构体，它显式声明了该指令所需的所有账户。Anchor 会代表你自动完成这些账户的安全验证。
#[account] 这是一个属性宏，用于定义你的数据账户的结构。它会自动为结构体添加序列化/反序列化逻辑、账户 discriminator（用于标识账户类型），并计算所需空间。
#[derive(Accounts)] 这是一个派生宏，用于生成你指令的 Accounts 结构体的实现。在这里，你为每个账户指定约束条件，Anchor 会在指令执行前自动验证这些约束。

部署过程

在 Solana 中，智能合约并不直接存储在账户中，而是作为程序存储在 Solana 区块链上。这些程序由交易发送到 Solana 节点进行部署，部署后可以通过程序 ID 与 Solana 网络交互。部署过程的步骤如下：

编译和部署程序

  • 编写智能合约：首先使用 Rust 编写智能合约代码，Solana 的智能合约需要使用 Rust 编程语言（可以使用 Anchor 框架来简化开发）。
  • 编译程序：通过 cargo build-bpf 命令将程序编译为 BPF（Berkeley Packet Filter）格式，以便在 Solana 网络上运行。
  • 部署程序：通过 anchor deploy 或 solana program deploy 命令将编译好的智能合约（程序）部署到 Solana 网络。部署过程实际上是将合约的代码上传到 Solana 网络，并生成一个 程序 ID。

或者，如果直接使用 Solana CLI 部署：

部署完成后，Solana 网络会生成一个 程序 ID，这是程序在区块链上的唯一标识符。此后，任何对该程序的调用都需要使用这个程序 ID。

程序 ID 和账户

  • 程序 ID：每个部署的 Solana 程序都会分配一个唯一的程序 ID，作为该程序在链上的标识。
  • 数据账户：程序和账户是分离的，智能合约的数据存储在 Solana 网络上的 数据账户 中。程序可以对这些账户执行操作，但程序本身并不持有这些数据，数据由普通的 Solana 账户（例如用户账户）持有。

CPI

CPI 的核心价值：实现程序间的互操作性和组合性，重用经过审计的安全代码，构建复杂的去中心化应用生态系统

CPI = 一个程序调用另一个程序，类似于以太坊中的合约间调用，但有重要区别。
