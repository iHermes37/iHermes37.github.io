深度理解EVM系统

背景

Gas优化+合约安全

以太坊 ≈ 分布式数据库 + 分布式计算机

以太坊 ≈ 一台操作系统里运行的“收费版虚拟机”

  • 硬盘 = Storage
  • 内存 = Memory
  • CPU寄存器 = Stack
  • OS调度/一致性 = 共识协议
  • 电表/时间片 = Gas

Storage（存储，永久保存到链上）

  • SLOAD（读存储）：贵，但比写便宜。
    • 固定消耗 100 gas。
  • SSTORE（写存储）：非常贵，因为要修改链上状态。
    • 从 0 → 非 0：20,000 gas
    • 从 非 0 → 0：5,000 gas 退还（清理存储有 refund）
    • 从 非 0 → 非 0：5,000 gas

所以 EVM 程序里能用 memory 的地方绝对别用 storage，因为存储修改是全链共识的数据写入。

Memory（临时内存，交易级别 scratchpad）

  • 访问（MLOAD / MSTORE）本身的指令是 相对便宜的，几十 gas。
  • 真正消耗 gas 的是扩展内存：
    • EVM 的 memory 一开始是空的。
    • 如果你访问到还没分配的字节（比如 MSTORE 到 1000 字节位置），EVM 会自动扩展内存。
    • 扩展内存的费用 = 线性费用 + 二次方费用，大约公式：

也就是说：

  • 小范围内存操作便宜。
  • 但如果一次性开辟几 KB ~ MB 的内存，gas 成本会急剧上升。

上下文不同

  • 普通函数调用通常在同一对象/进程上下文中
  • 合约调用有 独立 Storage、Memory、操作数栈，跨合约访问需要显式调用

外部调用 = 控制权交给另一合约

  • 对方合约可能再次调用调用者（递归重入）
  • 普通函数调用通常不会被调用的函数“劫持”再调用自己

状态更新顺序重要

  • 正常函数调用中，操作是原子顺序执行
  • EVM 中，如果外部调用在状态更新前，容易被利用（重入攻击）

EVM 是单实例、单线程执行的

  • 一个节点在执行一笔交易时，只有 一个 EVM 实例在运行
  • 这个 EVM 在处理合约调用时，会 递归维护调用上下文，类似函数调用压栈

调用上下文（Call Context）

每次合约调用会创建一个 新的调用上下文，包括：

资源 说明
Storage 每个合约的 Storage 是独立的，存放链上状态（永久存储）
Memory 临时内存，交易/调用生命周期内有效
操作数栈（Stack） 当前合约执行的 1024-depth 栈，用于计算和指令执行

注意：Memory 和 Stack 是 上下文局部的，不会影响其他合约调用
Storage 是跨上下文共享的，但每个合约都有自己的 Storage 区域

  • EVM 是 单实例解释器，所有调用都在同一个 EVM 里顺序执行
  • 当 A 调用 B：
    1. EVM 压入 B 的调用上下文（包括 B 的 Memory + Stack）
    2. 执行 B 的指令
    3. B 执行完，弹出上下文，继续执行 A

所以看起来像 “独立运行”，其实是 上下文切换 + 递归栈管理

  • 合约调用 ≠ 两个 EVM 同时运行
  • 合约调用 = 单 EVM 在同一调用栈上递归执行
  • “独立 Storage/Memory/Stack” 只是 上下文隔离，保证被调用合约的局部状态不会污染调

EVM 架构

区块链安全101-深入EVM-先知社区

  • EVM 是 基于栈的计算模型，而不是基于寄存器的 CPU 模型。
  • 它的操作数栈最多存 1024 个元素。
  • 每个元素固定是 256 位（32字节）。
  • 运算过程全靠出栈、入栈，遵循后进先出（LIFO）规则。

栈最多能放 1024 × 32 = 32768 字节 = 32 KB 的数据。

  • 如果 PUSH 太多而栈已满，会抛出 stack overflow 异常。

  • 如果 POP/取数时栈不够，会抛出 stack underflow 异常。

EVM 的运算栈（操作数栈）是 固定上限 1024 个槽位，每个槽位存一个 256 位数值，执行合约时所有算术/逻辑/控制流都只能依赖这 1024 个栈元素来完成。

类比

  • 在 CPU 里，栈大小可以动态增长（受限于内存）。
  • 在 EVM 里，栈是 固定最大深度 1024，类似给虚拟机规定了“最多 1024 个寄存器”。

内存布局

  • 线性字节数组：EVM 的内存是一个从地址 0 开始的 无限长字节数组（理论无限，实际受 gas 限制）。
  • 临时性：内存生命周期只在 一次消息调用（message call / transaction） 内有效，执行完就销毁。
  • 扩展消耗 gas：内存最初为空，访问更高地址时需要“扩展”，扩展的大小和平方成正比增加 gas。
  • 字节寻址：按字节寻址，但大多数操作要求 32 字节对齐。

虽然 EVM 本身没有“段寄存器”这样的硬隔离，但 Solidity 编译器、Yul IR 都会遵循一种 约定式布局：

内存地址段 大小 作用/含义 Solidity 常见用法
0x00 – 0x3f 64 字节 Scratch space，临时区，用于哈希计算或中间变量 keccak256 时存放输入数据；也可作为临时存放区，执行完随时可覆盖
0x40 – 0x5f 32 字节 Free memory pointer，当前空闲内存的起始位置（初始值 0x80） 动态分配内存时，从这里取指针并更新，比如存储返回值、构造动态数组
0x60 – 0x7f 32 字节 Zero slot，零值槽，用于数组/映射初始化 Solidity 在初始化动态数组/映射时引用这里的零，不会被写入数据
0x80 – ∞ 动态扩展 动态分配区，实际存放局部变量、动态数组、字符串、返回值 编译器生成的代码会把数组、字符串数据写在这里；随着分配增加而扩展

存储布局

  • 持久化：存储的数据会永久写入区块链（直到被更新或删除）。
  • 键值对：每个合约都有一个独立的 Storage，它是一个 256-bit → 256-bit 的映射（key → value）。
  • Sparse（稀疏）：理论上可以存无限多的 key，但未写入的 key 默认为 0。
  • 操作消耗 Gas：写入非常昂贵，读取相对便宜。

键值映射

  • key：256-bit (32 字节)，通常通过 Solidity 编译器计算偏移地址或者通过哈希（keccak256）生成。
  • value：256-bit (32 字节)，存储实际数据。

示例：

  • 编译器会把 data[key] 的存储位置计算为：

slot=keccak256(key.baseslot)slot = keccak256(key . base_slot)slot=keccak256(key.baseslot)

  • base_slot 是映射在合约 Storage 中的起始槽位（slot）。
