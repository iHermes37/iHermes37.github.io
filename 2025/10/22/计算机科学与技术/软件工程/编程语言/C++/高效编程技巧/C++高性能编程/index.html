引用计数

引用计数是一种内存管理技术，通过跟踪对象被引用的次数来决定何时释放其资源。

基本思想:

  • 每个对象附带一个计数器
  • 当新引用指向对象时，计数器递增
  • 当引用失效时，计数器递减
  • 当计数器归零时，自动释放对象

实现细节

嵌入对象内部

优点：内存局部性好，一次分配
缺点：对象必须知情并配合，侵入式设计

独立控制块（std::shared_ptr采用）

优点：非侵入式，可管理任何类型
缺点：两次内存分配，缓存不友好

条件执行的优化

核心概念：CPU流水线

一个汽车装配流水线，它被分成多个阶段（安装发动机、安装车门、喷漆等）。当第一辆车在喷漆时，第二辆车已经在安装车门，第三辆车在安装发动机。这样同时处理多辆车，大大提高了效率。

CPU流水线也是类似的原理：

取指 -> 译码 -> 执行 -> 访存 -> 写回.当第N条指令在执行时，第N+1条指令在译码，第N+2条指令在取指。理想情况下，每个时钟周期都能完成一条指令。

编译器优化

虚函数

虚函数可能造成性能损失

  • 构造函数必须初始化Vptr
  • 虚函数是通过指针间接调用的，所以必须先得到指向虚函数表的指针，然后再获得正确的函数偏移量
  • 内联是编译时决定的，编译器不可能把运行时才解析的虚函数设置为内联

无法内联函数造成的损失

必须允许编译器在编译期间就解析函数的绑定。通过对类选择硬编码或者将它作为模板参数来传递，可以避免使用动态绑定。(模板把解析的步骤从运行期间提到编译期间)

模板与继承

这是一种基于编译期决议的替代方案，主要通过CRTP模式实现。

工作原理

  • 使用模板基类，派生类作为模板参数传入。
  • 通过静态转换将this指针转换为派生类指针。
  • 所有函数调用在编译时就确定，没有运行时查找。
  1. 零运行时开销：没有vptr/vtable查找。函数调用在编译时直接绑定。
  2. 可内联性：编译器能看到具体的函数实现，可以轻松内联。对于小型函数，这能带来巨大性能提升。
  3. 缓存友好：处理同质集合（如vector<Circle>），数据在内存中连续分布。更好的数据局部性，更高的缓存命中率。
特性 继承（动态多态） 模板（静态多态）
决议时间 运行时 编译时
性能开销 有（虚函数调用） 无（直接调用）
内联可能 几乎不可能 很可能
内存布局 异构，不连续 同质，连续
二进制大小 较小 较大（模板实例化）
灵活性 高（运行时切换） 低（编译时确定）
调试难度 相对容易 相对困难

临时对象的生命周期

基本规则：临时对象的生命周期持续到包含它的那个完整表达式结束时为止。

  • 完整表达式：通常不是一个子表达式。例如，在 int x = f1() + f2() * f3(); 中，整个初始化语句是一个完整表达式。当分号 ; 出现时，所有在这个语句中创建的临时对象（除了被绑定到引用上的，见下文）都会被销毁。

重要的特例：绑定到常量引用/右值引用

当将一个临时对象绑定到一个常量左值引用 或 右值引用 上时，它的生命周期会被延长，变得与这个引用的生命周期相同。

为什么不能绑定到非const左值引用？
因为从逻辑上讲，如果能修改一个临时对象（通过非const引用），但这个对象马上就会被销毁，那么你的修改就毫无意义，这很容易导致程序员犯错。所以C++语言标准禁止这样做。

临时对象的产生和销毁（调用构造函数和析构函数）是有成本的。在性能敏感的代码中（如循环、数值计算），大量的临时对象会严重影响效率。

现代C++的解决方案：移动语义

C++11引入的移动语义 极大地优化了临时对象的使用。因为临时对象是“将亡值”，它的资源可以被“转移”而不是“拷贝”。
