抓包工具原理简述

理解 Charles 的 HTTPS 原理，就是理解它作了两次 TLS 握手并在中间解密重放。对大多数开发和测试场景，Charles 足够高效。但面对 SSL Pinning、mTLS 或应用自建套接字时，Charles 会受限。工程化的做法是把工具链组合起来：

  • 日常制作：Charles / Proxyman 做第一线调试；
  • 自动化与批量修改：mitmproxy；
  • 底层握手/网络困难：tcpdump + Wireshark；
  • 无法改 App 或 Pinning 场景：USB 直连抓包工具（如 Sniffmaster）+ Wireshark 分析。

把这些应用按照“从高层到低层、从易到难”的流程使用，能把绝大多数 HTTPS 抓包问题工程化地解决。

背景

证书固定 是应用层对 TLS 握手证书验证的增强，不改变底层 TLS 协议，而是在握手完成前主动校验证书/公钥是否符合预期。

  • 网络通道被动抓包（Wireshark） → HTTPS 明文不可见
  • 中间人/代理解密（Charles/Fiddler/HttpCanary） → 需要客户端信任 CA
  • 客户端/应用内 hook → 绕过 SSL pinning，可抓明文
  • **VPN模式抓包 ** → 创建虚拟VPN隧道，重定向本机所有IP流量

结论先行：没有 100% 不可被攻破的客户端方案，目标是把绕过成本从“几分钟导入代理”提升到“需要 root + native hook + 高级技能+物理访问”，同时尽早发现并自动响应异常

被动网络嗅探器

主要通过将网卡设置为混杂模式，来监听流经其网卡的所有网络流量，而不仅仅是发送给本机的流量。它们本身不主动发送探测包（除非使用一些特定功能如主动解析域名）。

阶段一：流量捕获（驱动/内核层）

这是嗅探的基础。工具通过系统调用，与操作系统内核的网络子系统及网卡驱动进行交互。

  1. 设置混杂模式：工具向操作系统发出请求，将指定的网络接口（如以太网卡、Wi-Fi适配器）设置为混杂模式。在此模式下，网卡会丢弃“目标地址过滤”机制，将所有监听到的帧（包括目的地址不是本机的帧）都传递给操作系统内核。
  2. 数据包捕获：网卡驱动程序收到物理链路上的所有数据帧后，将其上交至内核的网络协议栈。嗅探工具通过捕获库（在Linux上是libpcap，在Windows上是Npcap/WinPcap）从协议栈的底层（通常在数据链路层，即“刚进内核，还未被上层协议处理的位置”）复制一份数据包的副本。
  3. 内核缓冲：复制的数据包被存入内核的一块环形缓冲区中。这个缓冲区用于平滑流量高峰，防止用户态程序处理不及时导致丢包。

阶段二：包过滤（BPF/过滤引擎）

为了避免处理所有流量带来的性能压力，工具在捕获时或捕获后可以进行高效过滤。

  1. 伯克利包过滤器：tcpdump 和 libpcap 的核心是 BPF。BPF 是一个基于寄存器的虚拟机，运行在内核态。用户输入的过滤表达式（如 host 192.168.1.1 and port 80）会被编译成高效的 BPF 字节码。
  2. 内核态过滤：BPF 字节码被加载到内核中，每个到达的数据包都会由 BPF 虚拟机快速执行这段代码进行判断。只有符合条件的数据包才会从内核缓冲区复制到用户空间。这极大地减少了不必要的内核态-用户态数据拷贝和上下文切换，是高性能嗅探的关键。

阶段三：协议解析与解码（核心解析层）

数据包从内核传递到用户空间的工具后，进入深度解析阶段。

  1. 链路层解析：工具首先根据网卡类型（以太网、802.11无线等）解析数据链路层头部（如以太网帧头），获取源/目的MAC地址和上层协议类型（如IPv4、IPv6、ARP）。
  2. 多层解封装：工具像剥洋葱一样，按照网络协议栈的顺序逐层解析。
    • 解析网络层（IP头）-> 获取源/目的IP地址、协议类型（TCP、UDP、ICMP）。
    • 解析传输层（TCP/UDP头）-> 获取源/目的端口、序列号、标志位等。
    • 解析应用层（HTTP、DNS、TLS等）-> 这是 Wireshark/tshark 最强大的地方，它们包含了成百上千个协议解析器，能够理解各种应用层协议的格式，并将二进制数据转换为人类可读的字段和信息。
  3. 字段重建：高级工具（如Wireshark）能进行更深入的分析，例如：
    • 流重组：将属于同一个TCP会话的多个数据包按序列号重组，还原完整的应用层数据流。
    • 协议偏好设置：允许用户自定义解析方式（如将特定端口解码为HTTP协议）。
    • 统计与智能分析：识别重传、乱序、计算吞吐量、标记潜在错误等。

中间人代理抓包

这一部分在 抓包对抗综述一文中已经解释的很明白了，这里不多解释，这里简要讲述两种代理方式

SOCKS 代理

从网络层次与设计哲学上看，SOCKS 代理和 HTTP 代理本质上是为解决不同层级问题而生的两种代理机制。

SOCKS 代理工作在传输层（TCP/UDP），设计理念是“建立一个通用的、协议无关的数据转发通道”，它像一条盲管道——不关心管道中流动的数据是什么协议、什么内容，只负责将客户端的数据包原封不动地转发给目标服务器，再将服务器的响应原路传回。

SOCKS 代理的连接建立过程遵循标准握手流程（以 SOCKS5 为例）：

  1. 客户端首先与代理服务器协商认证方式，
  2. 然后发送包含目标地址和端口连接请求，
  3. 代理服务器据此与目标建立连接，
  4. 成功后即进入纯数据转发模式。

由于这种“协议透明”的特性，SOCKS 代理可以承载 HTTP、HTTPS、FTP、SMTP、游戏协议、甚至 BT 下载流量等任何基于 TCP 或 UDP 的应用协议，这也是它常被用于全局代理、游戏加速和科学上网工具（如 Shadowsocks）的原因。

  • SOCKS4：只支持 TCP 连接，不支持身份验证，不支持域名解析（客户端需自己解析DNS）
  • SOCKS5（当前主流）:: 支持 TCP 和 UDP，支持多种身份验证（无验证、用户名密码、GSS-API等）， 支持远程DNS解析（代理服务器帮你查域名），支持 IPv6

HTTP 代理

相比之下，HTTP 代理工作在应用层，专门为 HTTP/HTTPS 协议优化设计，它像一位懂行的秘书——能够理解并处理 HTTP 协议语义。

当客户端（通常是浏览器）向 HTTP 代理发送请求时，发送的是完整的 HTTP 请求报文（如 GET http://example.com/path HTTP/1.1），代理服务器会解析这个请求，提取出目标主机、路径、头部等信息，然后以代理自身的身份重新构造一个新的 HTTP 请求发送给目标服务器。

这个过程允许代理进行深度干预：它可以缓存响应内容、过滤广告、修改请求头（如添加 X-Forwarded-For）、压缩数据、甚至拦截特定内容。

对于 HTTPS 流量，HTTP 代理通过 CONNECT 方法建立隧道，此时它退化为类似 SOCKS 的简单转发器（但仅在 TLS 握手后），这也是为什么 Charles、Fiddler 等调试工具能实施中间人攻击解密 HTTPS 的前提。

两者的核心区别主要体现在五个方面：

  • 一是协议支持范围，SOCKS 代理是通用的，HTTP 代理本质上是专用的；
  • 二是智能程度，HTTP 代理能理解并处理应用层协议，SOCKS 代理只是“笨”转发；
  • 三是性能特征，HTTP 代理可缓存内容提升重复访问速度，SOCKS 代理无此功能；
  • 四是使用场景，HTTP 代理主要用于 Web 调试、内容过滤和企业上网管理，SOCKS 代理更适合需要协议无关性的全局代理或特定应用代理；
  • 五是安全性，HTTP 代理通常内置于企业防火墙，侧重内容审查，而 SOCKS5 支持多种认证方式，更常用于需要认证的访问控制。

VPN抓包

其核心原理是利用Android系统内置的VPN功能，在用户空间创建一个虚拟的VPN隧道，强制本机流量流经该隧道，从而实现拦截和捕获。整个过程可以清晰地分为五个关键阶段，下图展示了流量从产生到被捕获的完整路径：

阶段一：启动与权限申请

  1. 用户安装并启动抓包App。
  2. App请求 android.permission.BIND_VPN_SERVICE 权限，并弹出一个系统级别的VPN连接请求对话框（这是由Android系统控制的，应用无法绕过）。用户必须点击“确认”或“允许”。
  3. 一旦用户授权，App就获得了创建VPN连接的能力。这不是Root权限，而是系统赋予的、受严格监管的VPN服务权限。

阶段二：建立虚拟VPN接口（tun/tap）

  1. App调用 VpnService.Builder().establish() 方法。
  2. Android系统内核为该App创建一个虚拟网络接口（通常是 tun0）。这是一个点对点的虚拟设备。
  3. App获得一个用于读取/写入该虚拟接口的文件描述符。所有发送到这个 tun0 接口的数据，都会变成App可以读取的字节流；反之，App写入这个文件描述符的字节，也会从 tun0 接口“发送”出去。

阶段三：配置系统路由（流量重定向）

这是实现全局抓包的关键。App通过 VpnService.Builder 进行配置：

  1. IP地址分配：为虚拟接口 tun0 分配一个本地IP地址（如 10.0.0.2）。
  2. 路由表控制：App可以指定哪些IP流量应该走这个VPN。为了实现全流量捕获，App通常会添加一条默认路由（0.0.0.0/0 或 ::/0），这意味着设备上几乎所有App的所有IPv4/IPv6流量，其下一跳都会被系统路由到 tun0 虚拟接口。
  3. DNS设置：App通常会设置一个DNS服务器（如 8.8.8.8 或指向自己的本地地址），以便能够解析DNS查询。

效果：从此刻起，除了被系统排除的少数流量（如VPN服务自身的连接、一些特殊的系统服务），设备上其他所有App的网络数据包，都会被Android网络栈自动重定向到 tun0 接口，进而被我们的抓包App接收。

阶段四：流量拦截与转发（用户空间数据包处理）

App进入一个主循环，不断地从 tun0 的文件描述符中读取和写入数据。这个过程在用户空间进行，无需内核特权。

对于出站流量（设备 -> 互联网）：

  1. App从 tun0 读取到原始的IP数据包（通常是IPv4或IPv6包，包含完整的IP头、TCP/UDP头和载荷）。
  2. App现在拥有了这个数据包的完整副本，可以：
    • 直接记录/捕获：将原始IP包（或根据需要组装的以太网帧）保存下来，这就是抓包的核心。
    • 解析与分析：在内存中解析IP、TCP、UDP甚至应用层协议（如HTTP），用于实时展示。
    • 选择性修改：部分高级App支持修改数据包内容后再转发（类似中间人，但操作在IP/TCP层，比较复杂）。
  3. App需要将数据包原样（或修改后）发送到真实网络，否则网络连接会中断。它通过标准的Socket API（DatagramSocket 用于UDP，Socket 用于TCP）重建连接，将数据包的载荷部分发送到目标服务器。它必须小心地维护TCP状态（如序列号、确认号），这比HTTP代理复杂得多。

对于入站流量（互联网 -> 设备）：

  1. App通过真实Socket接收到来自互联网的响应数据。
  2. App将这些数据封装成IP数据包，写入 tun0 的文件描述符。
  3. 系统内核从 tun0 接口读取到这个包，并根据目的IP地址将其递交给原始发起请求的App。

阶段五：数据包处理与输出

  1. 格式封装：捕获到的原始IP数据包通常会被封装成标准的 PCAP 或 PCAPNG 格式，这是Wireshark、tcpdump等工具通用的格式，便于导出和分析。
  2. 实时展示：App界面会实时显示流量概况，如源/目标IP、端口、协议、域名、流量大小等。一些App（如PCAPdroid）内置了简单的协议解析器（如HTTP）。
  3. 过滤与统计：提供基于IP、端口、协议或应用的过滤功能，并进行流量统计。
  4. 导出共享：可以将捕获的PCAP文件导出，用专业的Wireshark进行深度分析。

Hook抓包

Hook可以在不同层级发生，深度和难度不同：

  1. Java层Hook（Xposed、LSPosed主要领域）：
    • 目标：Android应用中使用Java/Kotlin编写的方法。
    • 关键函数：如 OkHttpClient.newCall().execute(), HttpURLConnection.connect(), 各种 Socket/SSLSocket 相关方法。
    • 原理：在方法调用前或后插入自定义代码，直接读取方法参数（包含请求数据）或返回值（包含响应数据）。
  2. Native层Hook（C/C++库，Frida强项）：
    • 目标：libssl.so、libcrypto.so（OpenSSL/BoringSSL）中的加解密函数，或自定义Native库的网络函数。
    • 关键函数：如 SSL_read, SSL_write, SSL_do_handshake。这些函数是TLS/SSL通信中数据加密前和解密后的最后/最前一站。
    • 原理：Hook这些底层函数，直接读取其缓冲区（buf 参数）中的明文数据。
  3. 系统框架层Hook：
    • 目标：Android系统自身的网络API实现。
    • 原理：通过修改系统框架（需要Magisk模块等），全局影响所有应用。
