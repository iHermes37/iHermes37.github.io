无锁编程概述

建议先阅读：存储 - The Peak Tower

在多线程程序中，访问共享数据通常需要同步机制来保证数据的一致性和线程安全。传统方法是使用互斥锁（mutex）或读写锁来保护数据结构。然而，锁虽然简单易用，但在高并发环境下存在明显的性能瓶颈：线程在等待锁的过程中会被挂起和调度，产生上下文切换；大量线程争用同一把锁时，会导致严重的阻塞和延迟，降低系统吞吐量。

无锁编程正是在这样的背景下应运而生，它代表了并发设计哲学的一次根本性转变。作为无锁技术的核心，CAS原子操作使我们能够实现一种非阻塞的同步策略：线程通过原子指令尝试直接修改共享状态，仅在检测到冲突时通过重试机制继续前进，而无需陷入内核态或让出CPU控制权。

CAS 能够保证对共享变量的修改在硬件层面上原子执行，避免线程阻塞，并允许多个线程同时尝试更新同一个数据结构而不会破坏其一致性。同时现代硬件架构为无锁编程提供了坚实的基础支撑。CPU提供的原子指令集与缓存一致性协议（如MESI）共同确保了CAS操作的内存可见性与顺序一致性。

这种机制带来的性能优势是多维度的——它消除了锁竞争带来的吞吐量瓶颈，在冲突较少的情况下提供近乎线性的扩展能力；它保证了系统整体的前进步伐，即使个别线程延迟或崩溃，也不会阻塞其他线程的执行；同时，它极大简化了高并发场景下的推理复杂度，避免了传统锁机制中常见的死锁陷阱。

指令重排

分类

指令重排是 编译器或 CPU 为了性能优化 对指令执行顺序进行调整的行为，只要不影响单线程的程序语义。

  • 编译器重排：为了优化寄存器使用和流水线，可能改变语句顺序
  • CPU 重排：流水线或 out-of-order 执行，可能改变内存操作在多核观察到的顺序
  • 单线程程序看不出来 → 多线程共享内存时才会产生可观察异常

CPU 重排

CPU 为了提高流水线吞吐量，会重新排序指令执行顺序，只要单核程序的结果不受影响，CPU 重排是 硬件级别的，与编译器重排不同（编译器重排是在生成指令前做的），在多核环境下，如果没有内存屏障，其他核心看到的顺序可能和程序顺序不一致。

假设有两个线程，两个变量 X 和 Y，初始值都为 0。

  • 写 Y 可以延迟到读 X 之后执行

顺序一致性

内存访问顺序是指多核/多线程环境下，其他线程观察到的内存操作顺序。在多线程下，你看到的不一定是代码顺序，因为：

  • CPU 可能乱序执行
  • cache write-back 可能延迟
  • MESI 协议会在核间传播写入

  • 所有线程都同意内存操作发生的顺序，
  • 并且该顺序与程序源代码中的操作顺序一致，

这就是所谓的顺序一致性。换句话说，多线程执行时，好像每个线程的操作都是按程序顺序依次完成的，即便它们实际上在多个核上并行执行。

实现顺序一致性的一种简单（但显然不切实际）的方法是禁用编译器优化并强制所有线程在单个处理器上运行。处理器永远不会看到它自己的内存效果出问题，即使线程在任意时间被抢占和调度。

一些编程语言即使对于在多处理器环境中运行的优化代码也提供顺序一致性。在 C++11 中，您可以将所有共享变量声明为具有默认内存排序约束的 C++11 原子类型。在 Java 中，可以将所有共享变量标记为volatile.

因为 std::atomic 默认保证 顺序一致性，不可能出现结果 r1 = r2 = 0，编译器和 CPU 在幕后会插入必要的 内存栅栏或原子操作，这些保证让多核环境下操作顺序符合程序逻辑，但可能会略微降低性能

注意：顺序一致性是一种语义约束，不是硬件自动提供的，每种语言或编译器可能通过原子类型或 volatile 实现。

内存排序工具

在现代 CPU 中，要强制正确的内存访问顺序，一般使用三类工具：

  1. 轻量级同步或栅栏指令：阻止编译器或 CPU 对特定操作重排
  2. 完整内存栅栏指令（full fence / mfence）：禁止屏障前后的所有内存操作乱序
  3. 获取（Acquire）与释放（Release）语义的原子操作
    • Acquire：阻止后续操作提前执行
    • Release：阻止前面操作延迟执行

内存序

atomic头文件中定义了6种内存序：

  • memory_order_relaxed：松散内存序，只用来保证对原子对象的操作是原子的
  • memory_order_consume：目前不鼓励使用，其内部实现可能同 memory_order_acquire
  • memory_order_acquire：获得操作，在读取某原子对象时，当前线程的任何后面的读写操作都不允许重排到这个操作的前面去，并且其他线程在对同一个原子对象释放之前的所有内存写入都在当前线程可见
  • memory_order_release：释放操作，在写入某原子对象时，当前线程的任何前面的读写操作都不允许重排到这个操作的后面去，并且当前线程的所有内存写入都在对同一个原子对象进行获取的其他线程可见
  • memory_order_acq_rel：获得释放操作，一个读‐修改‐写操作（RMW）同时具有获得语义和释放语义，即它前后的任何读写操作都不允许重排，并且其他线程在对同一个原子对象释放之前的所有内存写入都在当前线程可见，当前线程的所有内存写入都在对同一个原子对象进行获取的其他线程可见
  • memory_order_seq_cst：顺序一致性语义，对于读操作相当于获取，对于写操作相当于释放，对于读‐修改‐写操作相当于获得释放，是所有原子操作的默认内存序，对访问顺序的约束力最强

内存屏障

在前面我们可以看到：在单线程执行中，CPU 和编译器可能会对指令进行重排，以提高执行效率，只要重排不影响该线程的语义，程序行为就是可预期的。但在多线程环境中，一个线程对共享变量的读写可能与其他线程的操作存在同步依赖。如果没有正确的同步机制（比如内存屏障），重排可能导致其他线程观察到的状态出现异常。

内存屏障提供了一种手段，用于显式限制指令的重排范围，确保关键读写操作的执行顺序符合线程之间的同步语义。当然如果一个线程不依赖另一个线程的执行顺序可以不使用内存屏障。

C++11 提供了 原子操作和内存顺序约束，通过读屏障（Acquire）、写屏障（Release）和全屏障（SeqCst / Fence）来保证顺序一致性。

读屏障（Acquire Barrier）

防止屏障之后的读操作被 CPU 或编译器提前到屏障前执行。

典型用途：读取一个“标志位”或同步信号（生产者/消费者模式）。

  • 语义：
    • 读取操作前面的所有操作可以被乱序
    • 读取操作之后的操作不能提前跨过屏障

例子场景：线程 A 写入数据后更新标志位，线程 B 读取标志位（Acquire），保证看到最新数据。

  • ready.store(..., release) 是写屏障，保证之前的 data.store 对其他线程可见
  • ready.load(..., acquire) 是读屏障，保证之后读取 data 时能看到最新值

写屏障（Release Barrier）

防止屏障前的写操作被 CPU 或编译器延迟到屏障之后执行。

语义：

  • 屏障前的写操作必须在屏障后可见
  • 屏障后的写操作不能提前到屏障前

读屏障和写屏障常配合使用（Acquire/Release），保证多线程同步。

  • 与上例类似，但重点在 写屏障：release 保证 buffer.store 的写入在标志位更新之前对其他线程可见。

全屏障（Full Fence / SeqCst Barrier）

屏障前后的 所有内存操作（读/写） 都不能跨越屏障

语义：实现 顺序一致性

适用场景：

  • 多线程无锁算法需要严格顺序
  • CAS 操作默认在 x86 下就是 full fence
  • 防止复杂多核乱序导致 ABA 或数据可见性问题
  • memory_order_seq_cst：对每个线程而言，所有操作按全局顺序一致
  • 即使多核乱序执行，也能保证每个线程看到的顺序符合程序逻辑
屏障类型 作用 C++ 对应方式 典型用途
读屏障（Acquire） 屏障后读不能提前 load(..., memory_order_acquire) 消费者线程读取标志位
写屏障（Release） 屏障前写不能延后 store(..., memory_order_release) 生产者线程写完数据再更新标志位
全屏障（SeqCst） 前后所有操作顺序一致 load/store(..., memory_order_seq_cst) CAS、严格顺序一致性需求

ABA问题

ABA 问题是无锁编程中 CAS 操作的一个典型问题。发生在如下场景：当一个变量的值从 A 变为 B，再变回 A 时，系统可能会误认为该变量没有发生变化，但实际上它经历了中间状态 B 的改变。

  1. 线程 T1 读取一个共享变量 X，值为 A
  2. 线程 T2 将 X 从 A 改成 B，再改回 A
  3. 线程 T1 执行 CAS(&X, A, C) 时成功
  4. 问题：T1 无法察觉 X 中间被修改过，可能破坏数据结构（例如链表断裂、栈丢节点）

核心：CAS 只比较值，不比较修改历史

解决方法

版本号 + CAS

给每个共享变量附加一个 版本号 counter，每次修改变量时，版本号加 1，CAS 操作同时比较 值和版本号，确保即使值回到原值，版本号不同，CAS 也会失败

原子操作

在多线程编程中，为了避免锁带来的开销和死锁风险，可以使用 无锁编程。无锁编程的核心是 原子操作（Atomic Operation），它保证在多个线程同时访问同一共享变量时操作 不可分割，不会被中断。

特性 CAS RMW RCU
类型 原子比较并交换 原子读-改-写 读写分离同步
原子性 是 是 写操作保证原子性，读操作无需锁
适用场景 无锁栈、队列、计数器 原子计数器、标志位 读多写少的数据结构（链表、哈希表）
优点 灵活，底层构建无锁结构 简单高效，硬件直接支持 读操作无锁，性能极高
缺点 ABA 问题，需要重试 仅适合简单数据 写操作复杂，需要延迟释放旧数据
C++ 示例 compare_exchange_weak/strong fetch_add/fetch_sub 原子指针 + 延迟回收

C++11 提供了 <atomic> 库，实现对基本类型或自定义结构体的原子操作，同时可以指定内存顺序（memory order）来控制操作的可见性和顺序。

原子操作的特点：

  • 不可分割：操作完成前不会被线程切换或中断
  • 线程安全：多个线程同时对同一个原子变量操作，不会破坏数据
  • 可控制内存顺序：可使用 memory_order_relaxed / acquire / release / seq_cst 指定屏障类型
  • 基础操作：
    • 读取 / 写入：load() / store()
    • CAS：compare_exchange_strong() / compare_exchange_weak()
    • 加减/位操作：fetch_add() / fetch_sub() / fetch_or() / fetch_and()
