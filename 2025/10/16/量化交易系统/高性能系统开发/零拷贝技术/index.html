零拷贝技术

可以看到：在传统 IO 数据传输的复合路径中：

路径 CPU拷贝
磁盘 → 内核 0
内核 → 用户 1
用户 → 内核 1
内核 → 网卡 0
总计 2 次 CPU copy

再加上 2 次 DMA，经常被简化说成：“传统 IO 有 4 次拷贝”，但严格区分应是：2 次 DMA，2 次 CPU memcpy，同时存在4 次上下文切换：

  1. read 系统调用时：用户态切换到内核态；
  2. read 系统调用完毕：内核态切换回用户态；
  3. write 系统调用时：用户态切换到内核态；
  4. write 系统调用完毕：内核态切换回用户态。

这里简单的介绍DMA：

DMA（Direct Memory Access，直接内存访问）是一种由硬件控制器或外设自行完成数据传输的机制，使设备（如网卡、磁盘、GPU）能够在不经过 CPU 逐字节搬运的情况下，直接在设备与主存之间读写数据；CPU 只负责在传输前配置内存地址和长度、在完成后接收中断或轮询结果，因此极大降低了 CPU 占用并避免了 cache 污染，同时充分利用内存与总线带宽。

现代操作系统与高性能 I/O（如网络收发、磁盘读写、DPDK、RDMA、GPU 直通）几乎都以 DMA 为基础，其核心价值在于把“搬数据”的工作交给硬件，把 CPU 资源留给协议处理与业务计算。

注意：只要是 外设 ↔ 内存，就一定要有 DMA 或等价硬件通路。

DMA = 直接内存访问，意思是：外设（网卡/磁盘/显卡）可以不经过 CPU 指令参与，直接把数据写进内存或从内存读走。

换言之：“没有 DMA 搬运数据”的网络 I/O 基本不存在。

综上所述：

  • 读过程涉及2次空间切换、1次DMA拷贝、1次CPU拷贝；
  • 写过程涉及2次空间切换、1次DMA拷贝、1次CPU拷贝；

可见在传统 I/O 模型中，数据从磁盘或网卡进入内核缓冲区，再从内核缓冲区复制到用户态缓冲区，发送时又需要从用户态复制回内核缓冲区，然后通过设备 DMA 发出，这会产生多次 CPU 拷贝和多次用户态与内核态切换，严重影响性能。整个过程涉及多次空间切换和数据冗余拷贝，效率并不高，接下来就该零拷贝技术出场了。

零拷贝原理

零拷贝（Zero-Copy）技术是一种高性能数据传输优化方法，其核心思想是尽量消除 CPU 在内存中搬运数据的操作，尤其是应用程序缓冲区与内核缓冲区之间的拷贝，从而降低 CPU 占用、减少 cache 污染、提升 I/O 吞吐和降低延迟。

零拷贝技术通过机制如 sendfile、mmap、splice、io_uring、DPDK 或 RDMA，使数据可以直接从设备 DMA 的内存缓冲区映射到最终处理位置或直接发送到网络，完全或部分绕过用户态拷贝和内核态中转，从而实现 CPU 资源的释放和 I/O 性能的极大提升。

分类

文件 I/O 相关零拷贝

  1. mmap（内存映射）：将文件或设备直接映射到用户态虚拟内存空间，用户程序可以直接访问映射页，避免内核→用户 memcpy
  2. **sendfile**：内核直接将文件页缓存（page cache）通过内核网络栈发送到 socket，避免用户态缓冲区拷贝
  3. splice / tee / vmsplice（管道零拷贝）：将内核缓冲区数据直接在管道或 socket 之间移动，用户态不参与拷贝

网络 I/O 相关零拷贝

  1. mmap + PF_RING / AF_PACKET：通过 mmap 将网卡缓冲区映射到用户态，减少内核拷贝，用户态轮询方式提高高频数据包处理性能
  2. DPDK（Data Plane Development Kit）：用户态直接管理网卡 DMA 缓冲区，完全绕过内核协议栈、零 CPU 拷贝、零系统调用
  3. RDMA（Remote Direct Memory Access）：网卡 DMA 直接写入远程内存，无 CPU 拷贝，避免内核协议栈、减少延迟

共享内存 / 内存复用

  1. POSIX / System V 共享内存：多进程共享同一块物理页，无需拷贝即可直接访问
  2. 环形缓冲区 + 内存池：网络中高性能消息队列（如 DPDK mbuf），避免频繁 malloc/free 和数据拷贝

补充

RDMA

RDMA（Remote Direct Memory Access，远程直接内存访问）是一种由网卡在硬件层直接完成的数据传输技术，使一台主机的网卡可以直接读写另一台主机已注册的内存区域，绕过对端操作系统内核、协议栈和用户态拷贝，从而实现零 CPU 拷贝、零系统调用、极低时延和极高吞吐的网络通信；

其本质是把传统由 CPU 和内核完成的网络协议处理下沉到网卡中执行，并通过内存注册与地址映射机制保证 DMA 可直接定位远端物理内存，因此广泛应用于高性能计算、分布式存储、数据库复制和金融低延迟交易系统中，是当前数据中心内部高速互连的核心技术之一。

DPDK

DPDK是一套用于高性能网络数据平面处理的开源软件库，其核心目标是绕过操作系统内核网络栈，直接在用户态管理网卡的收发缓冲区，实现极低延迟和高吞吐的网络 I/O。

DPDK 利用用户态轮询（polling）机制和大页内存（hugepage）将网卡的 DMA 缓冲区映射到用户空间，使数据包可以直接由网卡 DMA 写入用户态内存，而无需经过内核协议栈和中间拷贝，从而避免了传统 I/O 中的 CPU 拷贝和频繁的用户态↔内核态切换。

通过这种方式，DPDK 能够支持百万级包/秒的高速网络处理，广泛应用于高频交易、金融行情系统、电信网络和分布式计算等对延迟和吞吐要求极高的场景，同时提供丰富的内存管理、队列调度、轮询接口和多核优化机制，使开发者能够在用户态实现完整、高效的网络协议处理和数据流转。

实现方式

mmap

mmap 是 Linux 提供的内存映射文件机制，它将内核页缓存映射到用户态，使用户程序直接访问文件数据，消除了用户态与内核态之间的 CPU 拷贝，

但当这些数据通过套接字发送时，内核仍需要将 page cache 拷贝到 socket buffer，这一步仍由 CPU 执行，因此在内核空间仍存在一次 CPU 拷贝。

mmap+write方式有一定改进，但是由系统调用引起的状态切换并没有减少。

sendfile

sendfile系统调用是在 Linux 内核2.1版本中被引入，它建立了两个文件之间的传输通道。

sendfile方式只使用一个函数就可以完成之前的read+write 和 mmap+write的功能，这样就少了2次状态切换，由于数据不经过用户缓冲区，因此该数据无法被修改。

从图中可以看到，应用程序只需要调用sendfile函数即可完成，只有2次状态切换、1次CPU拷贝、2次DMA拷贝。

但是sendfile在内核缓冲区和socket缓冲区仍然存在一次CPU拷贝，或许这个还可以优化。

splice

splice系统调用是Linux 在 2.6 版本引入的，其不需要硬件支持，并且不再限定于socket上，实现两个普通文件之间的数据零拷贝。

splice 系统调用可以在内核缓冲区和socket缓冲区之间建立管道来传输数据，避免了两者之间的 CPU 拷贝操作。

splice也有一些局限，它的两个文件描述符参数中有一个必须是管道设备。
