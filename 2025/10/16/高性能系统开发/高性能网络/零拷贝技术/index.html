无锁队列

你可以想象一个路径：

网卡 → 内核 buffer → 用户态 buffer → 解析器 → 内部队列 → 策略引擎

  • 传统做法：
    • 内核收包存一份
    • 用户态 recv 再拷一份
    • 解析 JSON/二进制时再生成对象（第三份）
      → 每次行情相当于拷贝好几次，性能浪费大
  • 高性能做法：
    • recv/readv 直接拿到 buffer 指针
    • 在 buffer 上原地解析，不再做一次完整拷贝
    • 内部用 MarketData* 指针丢到无锁队列
    • 上层模块消费完数据后，buffer 可以回收（类似内存池）

这样整个链路只发生一次拷贝（内核→用户），甚至可以用 DPDK 或 io_uring + registered buffer 做到零拷贝。

行情是 实时流数据，一般不会缓存所有数据，否则内存爆炸。通常有几种处理方式：

  1. 逐条处理：
    每来一条行情（trade / depth update），直接推送到下游策略 → 无缓存，只依赖队列。
  2. 滑动窗口缓存：
    • 对 orderbook，需要保留一个全量快照，然后根据增量 update 更新（类似币安深度订阅）。
    • 对 trade，可以保留最近 N 条（比如 100ms 内的 tick）用于策略回测或统计。
  3. 内存池管理：
    • 使用固定大小的 buffer pool（预先分配几万个 MarketData 对象），避免频繁 malloc/free。
    • 解析器复用这些 buffer，策略消费后再归还 pool。

默认网络栈下，行情消息确实是 先到内核，再拷贝到用户态。优化方向是：避免用户态的额外 memcpy，原地解析 + 指针传递 + 内存池复用；如果要极致性能，可以考虑 mmap/io_uring/DPDK 直接绕过内核的二次拷贝。

1. 网卡接收数据的存放位置

默认情况下（不使用 DPDK 这种用户态驱动）：

  1. 网卡 → DMA → 内核空间缓冲区（socket buffer, skb）
    • 网卡收到数据包，会通过 DMA 直接写到内核预分配的 ring buffer（称为 socket buffer，Linux 内核里的 sk_buff）。
    • 这一步是硬件→内核，用户态看不到。
  2. 应用层调用 recv() / read() / readv()
    • 内核会把 skb 中的数据拷贝一份到用户态 buffer。
    • 所以你应用拿到的数据，已经是在 用户空间 buffer 里了。
  3. 用户态解析 → 分发 → 策略
    • 如果你再 memcpy 一次，就多了一份拷贝，性能浪费。
    • 所以高性能做法是：直接在这块 buffer 上原地解析，不再复制。

2. 高性能优化手段

传统 recv() 有一次 内核 → 用户态的拷贝，这个无法避免。但可以进一步优化：

  • mmap/PACKET_MMAP (Linux TPACKET v3)
    • 把网卡接收环（RX ring）直接映射到用户态。
    • 用户态不再需要 recv() 拷贝，直接在 mmap 出来的区域解析。
  • io_uring + registered buffer
    • 预注册 buffer 给内核，减少拷贝，几乎接近零拷贝。
  • DPDK/Netmap
    • 直接绕过内核网络栈，用户态驱动直接 poll 网卡。
    • 完全零拷贝，但需要独占网卡。

3. 你的“零拷贝”理解的正确性

所以回答你的问题：

  • 是的，网卡收到的行情消息最初是放在 内核缓冲区。
  • 应用 recv 拿到的已经是 用户态的一份拷贝。
  • 高性能做法：
    • 接受到用户态 buffer 后，直接原地解析，不再复制。
    • 用指针丢给下游模块，配合内存池复用。

4. 后续不断的行情怎么办？

  • 每次 recv/readv 系统调用都会写入新的 buffer。
  • 这些 buffer 的生命周期是：
    1. I/O 线程收到数据 → wrap 成 MarketData* → push 到无锁队列
    2. 策略线程消费完 → 回收 buffer 到 内存池
  • 这样不会无限增长，而是循环利用固定数量的 buffer。

你这个问题非常关键，涉及 零拷贝/用户态直解析 的极限性能场景，特别是 高速行情、UDP/组播、I/O 线程和消费线程速率不匹配 的问题。我来逐条分析：

1️⃣ Ring Buffer 满了就丢包怎么办？

在零拷贝/用户态直解析下，Ring Buffer 本质上是用户态的队列，存储 指向内核 buffer 的指针或描述符。如果 I/O 线程 push 的速度 > 消费线程 pop 的速度，就会出现溢出。解决思路有几类：

方案 A：增加缓冲容量

  • 分配更大的 ring buffer（比如 64K slots），给消费线程更多时间处理。
  • 缺点：内存占用大，延迟可能更高（cache miss + NUMA 跨页）。

方案 B：多线程/多队列并行消费

  • 将 symbol 或 feed hash 到不同队列，每个消费线程只处理自己的一部分，充分利用 CPU 核心。
  • 适合行情高并发（BTC/ETH/多品种），避免单队列成为瓶颈。

方案 C：批量消费

  • 让 I/O 线程一次 push 一批指针，消费线程一次 pop 批量处理（减少 CAS 原子操作次数）。
  • 也可减少队列 lock-free 操作开销。

方案 D：丢包策略（可接受的情况下）

  • 对 UDP/组播行情，如果消费追不上，丢掉最旧的包，保证最新行情可处理。
  • 可以结合快照机制，周期性重建状态（orderbook snapshot）。

2️⃣ 为什么还需要内存池？

你提到“所有操作都直接操作内核 mmap 内存，为什么还需要内存池”，这里有两个点：

  1. 内核 mmap buffer 只是 raw packet data
    • 内核的 RX ring buffer 是固定大小，每个 packet 有指针（或 offset）。
    • 但是解析后的 MarketData 对象/结构体 需要放在用户态可用的内存中，供下游策略线程消费。
    • 这些 MarketData buffer 如果每次都 new/malloc，频繁分配/释放成本高，还会造成 CPU cache miss。
  2. 内存池作用：
    • 管理 MarketData 对象的生命周期
    • 循环复用，减少 malloc/free
    • 可以固定大小、对齐，优化 CPU cache 行
    • 保证高频行情下内存不会频繁碎片化或增长

✅ 注意：内核 RX ring 的 buffer 与用户态解析后的 MarketData buffer 是两套东西。零拷贝指的是 解析可以直接读取 mmap buffer，而不拷贝原始数据，但最终的 MarketData 对象还是需要有内存来存放结构化字段或 metadata。

3️⃣ 为什么需要 epoll？

即使零拷贝，我们仍然需要 事件驱动机制 来知道哪些 socket / fd 上有数据可读：

  • TCP/WebSocket：
    • fd 是 socket 文件描述符
    • epoll/kqueue/poll 用于多路复用
    • 当有数据到达内核 RX ring → epoll 通知用户态 I/O 线程 → I/O 线程解析
  • UDP 多播：
    • epoll 仍然可以用来管理多个组播 fd 或不同网卡接口
  • 优势：
    • 避免 busy-loop 消耗 CPU
    • 只在真正有数据时触发回调

即使你用 mmap/DPDK，也可能用 轮询（polling）模式 替代 epoll，但：

  • 对极低延迟场景，DPDK 轮询 CPU 消耗大，但可以更快处理包
  • epoll 适合一般场景，可同时处理 TCP/UDP/WebSocket，不占用 100% CPU

4️⃣ 高性能零拷贝 & 消费慢的对策总结

问题 解决方案
I/O > 消费（Ring buffer 满） 1. 增加队列容量2. 多线程并行消费3. 批量 push/pop4. 可接受丢包 + 快照恢复
内存池作用 管理 MarketData 对象，不是原始 RX ring buffer
epoll 的作用 多路复用 fd，事件驱动通知，避免 busy loop
UDP 丢包风险 增大 socket buffer，批量处理，快照补偿

📌 核心思想：

  • 零拷贝指减少原始数据拷贝，不意味着不需要用户态内存。
  • 内存池管理解析后的数据对象。
  • epoll管理事件通知，决定何时读取内核 RX ring。
  • Ring buffer 满是用户态设计问题，需要通过多线程/批量/丢包策略解决。

你这个问题非常关键，我们来把 WebSocket、TCP、UDP、零拷贝和行情消费 结合起来分析，弄清楚适用性和区别。

1️⃣ WebSocket 本质

  • WebSocket 是一种基于 TCP 的应用层协议：

    • 建立在 TCP 之上，所以它有可靠性、顺序保证。
    • 数据通过 TCP 流传输，并且会封装成 WebSocket frame。
    • 你不会遇到“丢包”的问题，TCP 内核负责重传。
    • 应用只需关心：什么时候有完整 frame 可读。

  • 与 TCP、UDP 的区别：

    特性 TCP UDP WebSocket
    协议类型 传输层 传输层 应用层（基于 TCP）
    是否可靠 是 否 是（继承 TCP）
    顺序保证 是 否 是
    丢包重传 内核负责 无 内核负责
    适用场景 交易/下单/Web服务 行情/组播/低延迟 交易所行情/实时推送

✅ 总结：WebSocket 本质是可靠 TCP 流，不会像 UDP 那样丢包。

2️⃣ 对币安 WebSocket 的适用性

a. 是否需要事件驱动机制？

  • 推荐使用：即使 WebSocket 是 TCP，你仍然需要 I/O 线程事件驱动（epoll/select/kqueue 或 libuv/boost asio）：
    • 不占用 CPU 的 busy-loop。
    • 同时可以管理多个 WebSocket 连接（币安、OKX 等）。
    • 当有 frame 到达时触发回调解析，而不是轮询。

b. 是否需要方案 D（丢包策略）？

  • 不需要：
    • WebSocket 是可靠传输，不会丢包。
    • 你不需要像 UDP 那样丢最旧的包。
    • 只需解析 frame 并入队。

c. 是否需要快照机制？

  • 仍然推荐：
    • 对 orderbook 增量推送（depth updates），策略层要有全量快照。
    • 即便 TCP 不丢包，网络延迟/策略处理延迟可能导致状态落后。
    • 你通常会结合：
      1. REST 快照（初始全量 orderbook）
      2. WebSocket 增量更新（逐条解析 frame）
    • 这样可以保证策略总是有最新、完整状态。

3️⃣ 消费线程和内存池

  • 零拷贝：
    • 对 WebSocket，你仍然可以在 I/O 线程解析 frame 到 MarketData。
    • 用内存池管理 MarketData 对象，提高解析/队列性能。
  • Ring buffer：
    • TCP 不会丢包，但 I/O 线程 push 速度 > 消费线程速度时，队列仍会积压 → 延迟增加。
    • 不会丢包，但会出现 延迟累积，可通过多线程、批量解析缓解。

4️⃣ 与 UDP/组播对比

特性 UDP/组播 WebSocket
丢包风险 高，消费慢会丢包 低，TCP保证可靠
快照机制 必须，用于恢复状态 推荐，用于初始化/保证状态完整
消费速度要求 极高 高速可缓解延迟累积，但不丢数据
事件驱动 可选/轮询可 推荐，用 epoll/async 处理

✅ 总结

  1. WebSocket ≈ TCP，不需要丢包策略，但仍需事件驱动。
  2. 快照机制仍推荐，尤其是 orderbook 增量推送场景。
  3. 零拷贝/内存池依然适用：
    • 避免每条 frame 都 new/malloc
    • 提高高频 frame 的处理效率
  4. Ring buffer 满了不会丢包，但会增加延迟，需要多线程或批量处理缓解。

  1. 从设计开始
    • HFT 系统不是随便打开编辑器写代码就行，而是从整体架构设计开始规划。
    • 核心路径（critical path）必须尽可能短、无状态（stateless）、数据全在 CPU cache 中。
  2. 核心路径（hot path）
    • Hot path = 当市场数据到达、策略需要下单时，系统立即做出决策的路径。
    • Hot path 不允许阻塞，不进行日志记录，不做不必要的判断或循环。
    • 所有必要的决策都是一次性完成，避免重复判断。

2️⃣ 数据流和市场数据处理

  1. 海量市场数据
    • 不是交易量，而是数据量大。比如衍生品（期权）会产生数百万个不同符号的数据。
    • HFT 系统需要接入多个数据源（OPRA aggregate、交易所直连 feed 等），每个 feed 都可能独立处理。
  2. 多通道处理
    • 通过多达几十个（如 96）数据通道分流，保证系统不会因为单通道堵塞而延迟。
    • 系统会在用户态解析和标准化数据，使策略层拿到一致的、可立即使用的 MarketData。
  3. 数据清洗与解析
    • 只解析必要字段，跳过无用信息。
    • 数据解析在内存中直接进行（零拷贝），减少 CPU cache miss 和内存分配开销。

3️⃣ 风险引擎与策略执行

  1. 分布式风险管理
    • 订单必须经过风险引擎判断：价格是否合理，执行地点选择。
    • 数据量太大，不能集中在单个风险引擎上，需要分段、分布式处理。
  2. 策略执行
    • 解析后数据进入策略线程，快速计算指标和信号。
    • 决策生成后，通过异步、高性能下单模块发送订单。
  3. 延迟优化
    • 数据和对象尽量在 CPU cache 中
    • 循环和条件判断尽量减少，代码路径尽可能短
    • 内核和硬件优化（如 FPGA/NIC）确保关键事件即时触发

“数据在 CPU Cache 中” = 数据紧凑、连续、易于 CPU 预取，访问速度极快，不必频繁访问慢速主内存。

1️⃣ mmap + PACKET_MMAP 的“零拷贝”是相对的

  • 使用 AF_PACKET + PACKET_MMAP，用户态通过 mmap() 映射内核缓冲区。

  • 原理：

    • 网卡通过 DMA 写入 内核环形缓冲区（kernel memory）
    • 用户态访问这段内存时，不会再做额外的 内核到用户拷贝

  • 特点：

    • 减少了一次内核 → 用户态的拷贝
    • 但是 数据仍然先写入内核空间（内核分配缓冲区，网卡 DMA 写入）

  • 换句话说：

    内核缓冲区依然存在，占用内核内存，也需要内核管理。

✅ 优势：普通服务器即可使用，延迟低
⚠️ 缺点：内核仍然参与数据管理，系统调用和中断仍有开销，微秒级延迟有瓶颈。

2️⃣ DPDK 的“完全无内核拷贝”

  • DPDK 的关键：绕过内核网络栈（Kernel Bypass）

  • 网卡 DMA 直接写入用户态预分配的 hugepage 内存

  • 原理：

    • 不经过内核缓冲区
    • 用户态直接操作数据
    • 不涉及内核网络栈处理、拷贝或中断驱动（使用轮询 Poll Mode Driver）

  • 这是真正的“零拷贝”，延迟可降到 亚微秒级

✅ 优势：

  • 完全绕过内核
  • 数据直接在用户态
  • 避免内核中断、中断处理延迟
  • 避免内核缓冲区拷贝

⚠️ 缺点：

  • 网卡和 CPU 要支持 DPDK
  • 编程复杂，系统资源管理需要自己做

网卡 (NIC) → 内核网络栈 → 内核缓冲区 → 系统调用 read()/recvfrom() → 用户态缓冲区

技术 原理 优势 缺点
PF_PACKET + PACKET_MMAP 用户态 mmap 内核缓冲区 零拷贝，低延迟 仅 Linux，复杂
DPDK 用户态直接 DMA 亚微秒延迟，高吞吐 网卡/系统限制高

recvmmsg 批量系统调用 减少上下文切换 不是真正零拷贝

技术 内核 involvement 数据流 特点

异步I/O+多线程

在线程同步代码中保持对共享变量/内存区的疯狂轮询，一旦有消息就立刻处理，之后继续轮询，这样是最快的处理方式。

对象是分配在堆上的，而堆的内存结构对我们来说是完全不透明的，没办法控制一个对象会具体分到内存的什么位置上，这直接导致的问题是本来连续收到的网络包，在内存里的分布是分散的，当处理线程需要读取数据时就会发生大量的cache miss，产生不可控的延迟。

　　所以对C++开发者来说，第一条需要谨记的应该是，不要随便使用堆（用关键字new）。核心的数据要保证分配在连续内存里。

另一个问题在于，市场数据和交易指令都是结构化的，包含了股票名称，价格，时间等一系列信息。如果使用C++ class来对数据进行建模和封装，同样会产生不可知的内存结构。为了严格控制内存结构，应该使用struct来封装。 一方面在对接收到的数据解析时可以直接定义名称，一方面在分配新对象（比如交易指令）时可以保证所有数据都分配在连续的内存区域。

正确性应该可以借助于C++的特性比如强类型，模板等来加强验证，这方面我不熟悉就不多说了。

普通的C++服务程序，内部延迟（从进程收到消息到进程发出消息）做到几百微秒（即亚毫秒级）是不需要特殊的努力的。

可取的做法：尽量使用vector或者array（初始化时分配足够的空间，之后每次使用都从里面取出来用）。尽量使用内存池。如果需要二叉树或者哈希表，尽量使用侵入式容器（boost::intrusive）。

上下文切换是非常耗时的，其中固定的消耗包括（cpu流水线被冲掉、各种寄存器需要被保存和恢复、内核中的调度算法要被执行），此外，缓存很有可能出现大量miss，这属于不固定的时间消耗。

可取的做法：使用带有内核bypass功能的网卡。每个进程或者线程都独占一个cpu核【isolcpus和irqbalance的细节见注释3】，并且不停地轮询，用以保证快速响应。尽量避免任何可能导致阻塞的事件（如mutex），某些注定很慢的活动（比如把log写到磁盘上）应该被独立出来放到别的cpu上，不能影响主线程。

系统调用只需要<100ns，线程/进程切换需要>1us（不包括缓存miss的时间）。

使用共享内存作为唯一的IPC机制相关的知识背景：共享内存只有在初始化的时候有一些系统调用，之后就可以像访问正常内存一样使用了。其他IPC机制（管道、消息队列、套接字）则是每次传输数据时都有系统调用，并且每次传输的数据都经历多次拷贝。因此共享内存是最快的IPC机制。可取的做法：使用共享内存作为唯一的IPC机制。

当然，可能需要手动实现一些东西来保证共享的数据在多进程下是安全，我们是自己实现了无锁内存池、无锁队列和顺序锁

  • 共享内存 + Ring Buffer 是高频多线程/多进程解耦的关键
  • uWebSockets 是便利和高性能结合的方案
  • 自己实现解析 只有在追求极限延迟时才值得

总结一句话：

  • 延迟可接受 → 用 uWebSockets，回调里写入共享内存 Ring Buffer
  • 极限微秒延迟 → 自行 TCP + WebSocket 帧解析 + 共享内存 Ring Buffer

虽然访问速度快，但需要注意：

(1) 数据一致性

  • 多进程同时访问共享内存可能导致读写冲突
  • 需要 原子操作 / 无锁队列 / 顺序锁 / RCU 保证数据一致性

(2) 内存管理

  • 高频消息更新，需要 无锁内存池 来避免频繁 malloc/free
  • 消费者和生产者不能同时释放同一块内存

(3) 缓冲区设计

  • 高频行情建议使用：
    • Ring Buffer（环形缓冲区）：最新 N 条行情缓存
    • 顺序锁（seqlock） 或 RCU：保证读者无锁读取最新数据
    • 无锁队列：多策略进程或线程消费
名称 定义 特性 高频行情用途
共享内存 多进程共享的物理内存 初始化一次，用户态可直接访问 IPC 高速通道
mmap 用户态映射共享内存的系统调用 把内核内存映射到进程虚拟空间 访问共享内存
Ring Buffer 循环缓冲队列 高速、可无锁、多线程/多进程安全 缓存最新 N 条行情，实现生产

  • 把内核缓冲区映射到用户态
  • NIC DMA 写入内核缓冲区 → 用户态直接访问内核缓冲区
  • 避免每次 read 的内核拷贝
  • 还是需要用户态解析 WebSocket 帧

共享内存只是提供多线程/多进程之间零拷贝交换数据的区域

  • TCP socket（用户态） → WebSocket帧解析（用户态） → 共享内存 Ring Buffer → 消费者线程/进程

WebSocket 帧解析和共享内存是两个独立的步骤：

  1. WebSocket 解析：把 TCP payload 拆成完整消息（JSON/Protobuf）
  2. 写入共享内存 Ring Buffer：保证多线程/多进程消费的低延迟、无锁访问

写到共享内存 Ring Buffer 的内容

  • WebSocket 接收线程拿到的是完整 payload，一般是 JSON 或 Protobuf
  • 写入共享内存 Ring Buffer前，你有两种选择：

方案 A：直接写 JSON / 原始字节

  • Ring Buffer 内部存放 原始消息字符串或二进制数据
  • 消费者线程需要：
    • 再解析 JSON → 内部结构体
    • 每次访问都解析，性能开销较大
  • 优点：
    • 实现简单
    • 可以兼容多种消费者，不要求固定结构
  • 缺点：
    • 高频行情，每条消息都解析 JSON → CPU 消耗大

方案 B：提前解析成结构体再写入

  • WebSocket 线程解析 JSON → 填充 结构体
  • 写入共享内存 Ring Buffer
  • 消费者线程直接访问结构体 → 零解析开销
  • 优点：
    • 高频访问零解析
    • 延迟最低
  • 缺点：
    • 结构体必须固定大小或预分配缓冲区
    • 需要考虑内存对齐、线程安全

✅ 高频行情系统常用 方案 B

高频消息，不要每条都 new / malloc → 会严重拖延 GC/堆管理

使用内存池（Memory Pool）：

  • 预先分配固定大小的内存块
  • WebSocket 线程写入时直接取内存块
  • 消费者线程用完归还
  • 无锁或 lock-free 内存池可以减少线程竞争

共享内存 Ring Buffer 存储的是结构化数据（结构体），而非 JSON 字符串

高频访问必须提前解析，避免消费者重复解析

使用内存池管理结构体，避免频繁分配/释放，保证低延迟

  • 共享内存本质上是内核分配的一块物理内存
  • 通过 mmap 映射到用户态后，用户态访问就像普通内存一样
  • 注意：
    • 它不在进程的堆上
    • 也不是栈内存
    • 多进程都可以映射同一块共享内存

示意：

所以，共享内存 Ring Buffer 不属于堆内存，访问方式和堆类似，但生命周期和堆不同（需要 shm_open / shm_unlink 管理）。

Ring Buffer本身只是一个缓冲区（下标 + 元素数组）

存放的数据是否用内存池取决于数据结构：

不需要每种结构体都建 Ring Buffer

推荐做法：

  1. 统一消息类型 MarketMsg + 类型字段 + payload指针
  2. 不同结构体用内存池管理
  3. 消费者根据 type 解析处理

这种方式：

  • Ring Buffer 一套即可
  • 多类型行情支持灵活
  • 内存管理高效，适合高频低延迟场景
  1. CPU 核心绑定 → 避免调度抖动，提高预测性延迟
  2. 异步 I/O → 不阻塞线程，提升吞吐和并发
  3. DPDK → 用户态零拷贝网络 I/O
  4. FGFK / FPGA → 硬件加速，极限延迟场景

FPGA 做硬件解析 → 结构化数据直接写入共享内存 / DMA

CPU 做策略计算 → 生成交易指令

PGA = 可编程硬件，硬件级并行处理

优点：极低延迟、零拷贝、并行处理

高频交易使用场景：

  1. 实时行情解析（UDP/TCP/WebSocket）
  2. 低延迟风控、策略前置计算
  3. 协议解析 / 下单指令生成

FPGA + DPDK/用户态网络 + 内存池/Ring Buffer → 构建极致低延迟交易系统

组合方式 数据流 优点
FPGA 前置，DPDK 后台 NIC → FPGA 硬件解析 → DMA 写入用户态内存 → DPDK 收取 → CPU 策略计算 延迟极低，CPU 不做解析，直接拿结构化数据
DPDK + FPGA DPDK 用户态收网络流 → FPGA 做协议卸载/硬件解析 → 写回共享内存 高吞吐量，减少 CPU 负担

核心思想

  1. FPGA：做硬件解析、协议拆分、过滤
  2. DPDK：做用户态网络 I/O、零拷贝收发
  3. CPU：只做策略计算和下单
  4. 共享内存 / Ring Buffer：多线程或多进程高效消费

  1. 减轻用户态线程负载
  2. 提前丢弃不需要的包 / 消息
  3. 简单字段提取，减少 JSON 解析压力
