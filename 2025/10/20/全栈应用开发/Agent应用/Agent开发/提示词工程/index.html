实现范式

‌ReAct 模式

注意ReAct只是一种agent设计思想，而不是一种框架

  • 思考（Reason）‌：代理基于当前上下文生成推理步骤（如是否需要调用工具、选择哪个工具）。借鉴“Chain-of-Thought（思维链）”，让模型先“自言自语”地分析问题。
  • ‌行动（Action）‌：根据推理结果，调用工具执行操作（如 API、计算器、搜索）或生成自然语言响应。
  • ‌观察（Observe）‌：获取工具执行结果或用户反馈，更新上下文。
  • 循环上述步骤直至任务完成‌。

系统提示词

由于每次工具调用都会将结果拼接到提示词上下文中中整体返回给大模型。LLM 没有记忆，它需要每轮都看到“过去做过什么”。拼接后的上下文能让 LLM 进行“链式推理”而非盲目重复调用工具。这也是 ReAct 比单步工具调用更智能的根本原因。

所以系统提示词里，不要写死 Thought / Action / Observation / Final Answer 的内容模板或范例中具体的推理结果。因为在 ReAct 框架中，这些字段（Thought: ... / Action: ...）会在每次 LLM 调用后自动拼接到输入上下文中。
如果系统提示词中“硬编码”了这些推理步骤，LLM 会被误导，以为要重复执行相同的工具调用，从而陷入死循环。

初次使用时我犯了一个：“教模型思考”的错误，错误地在系统提示词中放入具体范例

导致 不断循环执行 get_price → 拼接 → 再次执行 → 死循环。

所以在系统提示词中只需定义角色功能和思考模式即可

结构化输出

默认的 ReAct 是“纯文本推理链”，输出是自然语言字符串

  • return_direct: bool = True

提示词要素

指令：可以使用命令来指示模型执行各种简单任务，例如“写入”、“分类”、“总结”、“翻译”、“排序”等，从而为各种简单任务设计有效的提示。请记住，你还需要进行大量实验以找出最有效的方法。以不同的关键词（keywords），上下文（contexts）和数据（data）试验不同的指令（instruction），看看什么样是最适合你特定用例和任务的。通常，上下文越具体和跟任务越相关则效果越好。

具体性：要非常具体地说明希望模型执行的指令和任务。提示越具描述性和详细，结果越好。特别是当对生成的结果或风格有要求时，这一点尤为重要。不存在什么特定的词元（tokens）或关键词（tokens）能确定带来更好的结果。更重要的是要有一个具有良好格式和描述性的提示词。

避免不明确：给定上述关于详细描述和改进格式的建议，很容易陷入陷阱：想要在提示上过于聪明，从而可能创造出不明确的描述。通常来说，具体和直接会更好。这里的类比非常类似于有效沟通——越直接，信息传达得越有效。

做什么还是不做什么:设计提示时的另一个常见技巧是避免说不要做什么，而应该说要做什么。这样（说要做什么）更加的具体，并且聚焦于（有利于模型生成良好回复的）细节上。

此外还需要注意：

  • 上下文：包含外部信息或额外的上下文信息，引导语言模型更好地响应。
  • 输入数据：用户输入的内容或问题。
  • 输出指示：指定输出的类型或格式。

提示技术

经过大量数据训练并调整指令的LLM能够执行零样本任务。指令调整本质上是在通过指令描述的数据集上微调模型的概念。

当零样本不起作用时，建议在提示中提供演示或示例，这就引出了少样本提示。少样本提示可以作为一种技术，以启用上下文学习

以下是在进行少样本学习时关于演示/范例的一些额外提示：

  • “标签空间和演示指定的输入文本的分布都很重要（无论标签是否对单个输入正确）”
  • 使用的格式也对性能起着关键作用，即使只是使用随机标签，这也比没有标签好得多。
  • 其他结果表明，从真实标签分布（而不是均匀分布）中选择随机标签也有帮助。

标准的少样本提示对许多任务都有效，但仍然不是一种完美的技术，特别是在处理更复杂的推理任务时。

链式思考(CoT)

核心思想：在少样本示例中，不仅给出问题和答案，还展示得出答案的中间推理步骤。这引导模型“一步一步地思考”。

效果：模型会效仿这个过程，先找出奇数（15, 5, 13, 7, 1），再求和（41），最后判断（41是奇数），从而得出正确答案 False。

自动CoT & 自洽性

这是CoT的进阶技巧，旨在提升准确性和可靠性。

  • 自动CoT：在提示中不再提供人工编写的推理步骤，而是直接用一句神奇的指令（如 “让我们一步一步地思考”）触发模型自己的推理能力。这省去了构造示例的麻烦。
  • 自洽性：针对一个问题，让模型生成多条不同的推理链，然后从这些链得出的最终答案中进行多数投票。这有效减少了偶然的错误。

链式提示

注意：

  • 链式思考 是一种 提示设计技巧，用于改进模型单次推理的内容质量。
  • 链式提示 是一种 系统架构策略，用于将一个复杂任务拆分成多个模型调用步骤。

将一个宏大的、模型难以一步到位的任务，拆解成一系列有依赖关系的子任务，每个子任务由一个独立的提示和模型调用来完成。

  • 您提供的例子（文档问答） 是链式提示的经典案例：
    1. 提示1（子任务A：检索）：专门给模型下达指令：“从文档中提取相关引文”。模型执行并输出引文列表。
    2. 提示2（子任务B：生成）：将提示1的输出（引文） 和原始文档作为输入，再给模型下达新指令：“根据这些引文构建答案”。模型执行并输出最终答案。

萌发了后文的ReAct

思维树 (ToT)

核心思想：将一个复杂问题分解为多个步骤，在每一步，模型都生成多种可能的下一步，然后像在树结构中搜索一样，通过某种评分机制（如模型自我评分）选择最有希望的路径继续探索，直至找到最终答案。

ToT 维护着一棵思维树，思维由连贯的语言序列表示，这个序列就是解决问题的中间步骤。使用这种方法，LM 能够自己对严谨推理过程的中间思维进行评估。LM 将生成及评估思维的能力与搜索算法（如广度优先搜索和深度优先搜索）相结合，在系统性探索思维的时候可以向前验证和回溯。

思维分解与生成

  • 思维步骤：将一个大问题分解为多个、离散的中间推理步骤。例如，在“算24点”中，每一步就是一个中间方程。
  • 候选项：在每一步，不是只生成一个“想法”，而是并行生成多个（如5个）可能的下一步。这模拟了人类在遇到难题时会考虑多种可能性的过程。

状态评估与搜索

  • 启发式评估：使用大语言模型本身作为一个评估器，对每个候选思维进行前景判断（如 “sure/maybe/impossible”）。这为搜索提供了方向，可以优先探索最有希望的路径（“sure”），并果断剪枝无效路径（“impossible”）。
  • 搜索算法：通过经典的搜索算法（如 BFS、DFS）来系统地探索这棵“思维树”。这使得推理过程不再是一次性的盲目生成，而是一个有规划、可回溯的探索过程。

ReAct 框架

ReAct 是一种将 推理 和 行动 整合在一起的范式，旨在让大语言模型在解决复杂问题时更加可靠和可控。模仿人类解决问题的过程：我们不会盲目行动，而是先思考一步（Reason），然后采取行动（Act），根据行动结果再继续思考，如此循环，直到解决问题。

ReAct = Reason + Act

  • Reason：模型生成对当前情况的推理，解释它正在做什么、为什么这么做，以及下一步计划。这类似于 CoT，但更侧重于规划。
  • Act：模型执行一个具体的动作。这些动作通常是可执行的、受限制的.

将每次工具返回结果作为下一次思考的上下文+系统预设置的提示词输入给llm进行思考调用

自动提示工程师 (APE）

如果说 ReAct 和 CoT 是让人来设计更好的提示，那么 APE 的目标是 让模型自己来设计最好的提示。

核心思想：将“寻找最优提示”本身定义为一个优化问题，并使用大语言模型作为搜索算法的主体来解决它。

工作流程：

  1. 指令生成：给定一个任务描述和几个示例，让一个大语言模型（作为“指令生成器”）生成大量候选的指令/提示。
  2. 指令评估：让另一个大语言模型（作为“评分器”）在验证集上运行这些候选指令，并根据任务目标（如准确性）对它们的表现进行评分。
  3. 指令选择：选择得分最高的指令作为最终结果。

APE 示例：

  • 任务：让模型执行“情感转换”——将负面评论重写为正面评论。
  • 初始想法：人工可能会设计提示为：“请将以下负面评论改写为正面评论，同时保持原意。”
  • APE 过程：
    1. 生成：模型可能会生成多种候选指令，例如：
      • “用积极的语气重写这个。”
      • “把这个抱怨变成赞美。”
      • “进行情感转换：负面 -> 正面。”
      • “写出这个句子的反面，但说的是同一件事。”
    2. 评估：在一批负面评论上测试这些指令，看哪个指令产生的正面评论最准确、最自然。
    3. 选择：假设评估发现 “写出这个句子的反面，但说的是同一件事。” 得分最高，那么这个指令就被 APE 选定为最优提示。

APE 的价值：

  1. 超越人类直觉：模型可能会发现人类想不到的、更有效或更鲁棒的提示方式。
  2. 自动化与规模化：为大量任务自动生成高质量提示，节省了大量人工试错成本。
  3. 发现新范式：有助于发现新的、可推广的提示策略。

检索增强生成 (RAG)

要完成更复杂和知识密集型的任务，可以基于语言模型构建一个系统，访问外部知识源来做到。这样的实现与事实更加一性，生成的答案更可靠，还有助于缓解“幻觉”问题。

为什么需要 RAG？解决大模型的三大痛点:

痛点 问题描述 RAG 的解决方案
知识幻觉 模型会自信地编造看似合理但完全错误的信息。 强制模型基于检索到的文档片段生成答案，言之有物，大大减少胡编乱造。
知识过时 模型的训练数据有截止日期，无法知晓最新事件。 知识库可以随时更新，让模型能够回答关于最新新闻、政策或产品的问题。
事实溯源 你无法知道模型的答案来自哪里，难以验证其真实性。 答案来源于具体的文档，可以提供引用来源，方便用户核对，增强信任。

RAG 是一种将 信息检索 系统与 大语言模型 相结合的技术架构。它的核心思想是：在让大模型回答问题之前，先从一个外部知识库中检索相关的信息，然后让模型基于这些确凿的依据来生成答案。

RAG 会接受输入并检索出一组相关/支撑的文档，并给出文档的来源（例如维基百科）。这些文档作为上下文和输入的原始提示词组合，送给文本生成器得到最终的输出。这样 RAG 更加适应事实会随时间变化的情况。这非常有用，因为 LLM 的参数化知识是静态的。RAG 让语言模型不用重新训练就能够获取最新的信息，基于检索生成产生可靠的输出。

自我反思(Reflexion）

自我反思是一个通过语言反馈来强化基于语言的智能体的框架。

在高层次上，自我反思将来自环境的反馈（自由形式的语言或者标量）转换为语言反馈，也被称作 self-reflection，为下一轮中 LLM 智能体提供上下文。这有助于智能体快速有效地从之前的错误中学习，进而提升许多高级任务的性能。

传统的提示技术是单向的：输入 -> 模型 -> 输出。而 Reflexion 技术引入了一个 “自我反思循环”：

核心目标：通过让模型进行自我反馈，显著提升其在复杂任务（尤其是推理和编程任务）上的准确性和可靠性。

  1. 生成初始答案：模型根据问题和指令，生成第一个版本的答案。
  2. 自我评估与反思：这是最关键的一步。模型被要求扮演一个“审阅者”或“批评者”的角色，基于特定的标准来审视自己刚刚生成的答案。常见的反思指令包括：
    • “检查你的答案中是否有计算错误或逻辑错误。”
    • “你的推理过程是否完整？有没有跳过必要的步骤？”
    • “你的结论是否符合常识？”
    • “如果你是老师，你会给这个答案打几分？为什么？”
  3. 决策与迭代：
    • 如果反思后没有发现严重问题，模型就输出当前答案作为最终结果。
    • 如果反思后发现了错误或不足之处，模型需要将这次的“反思内容”作为新的上下文，重新生成一个改进后的答案。这个循环可以进行多次，直到模型对自己满意为止。
