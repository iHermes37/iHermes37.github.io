CAS和ABA问题

  • CAS：原子比较和交换，是无锁编程的核心工具
  • ABA：CAS 在多线程修改中可能看不到“中间变化”，导致逻辑错误

CAS 的设计就是“只在你看到的值没被别人改动时才写入，否则告诉你最新值”，保证并发安全且无需锁。

CAS 就像 银行取钱：

  1. 你看到账户余额 100 元（expected）
  2. 想取 20 元 → 新余额 80 元
  3. CAS 判断“余额还是 100 元吗？”
    • 是 → 扣款成功
    • 否 → 不扣，告诉你当前余额是多少，你可以重新计算

ABA 问题：

想象在超市排队买饮料：

  1. 你看到货架上有一瓶可乐（A），于是打算拿走。
  2. 在你走过去的过程中，别人把那瓶可乐拿走（B），又放回了一瓶新的可乐（A）。
  3. 你看到货架上还是 A，就以为没有人动过，直接拿走。

问题是：中间发生了变化，你却没察觉——拿到的可能已经不是原来的那瓶，可乐内容、价格或保质期都可能不同。

SPSC

原子变量（std::atomic）使用

std::memory_order - cppreference.cn - C++参考手册

C++ 的原子操作 (std::atomic<T>) 不仅要“线程安全”，还要考虑：

  • 操作间的顺序保证（happens-before）
  • 编译器优化的指令重排
  • CPU 的乱序执行
概念 含义 类比
原子性（Atomicity） 操作不会被中断或部分执行。要么全成功，要么完全不执行。 “门锁一次只能让一个人进”
顺序性（Ordering） 操作在多线程下的可见顺序如何。即：一个线程写入的结果，别的线程何时、以什么顺序看到。 “别人什么时候看到你进门”

C++原子操作的 memory_order 六种模式

内存序 名称 说明
memory_order_relaxed 放松（无序） 仅保证原子性，不保证任何顺序
memory_order_consume 消费（已废弃） 基于依赖的读屏障
memory_order_acquire 获取 保证此操作之后的读写不会被重排到此操作之前
memory_order_release 释放 保证此操作之前的读写不会被重排到此操作之后
memory_order_acq_rel 获取+释放 同时具备 acquire 和 release
memory_order_seq_cst 顺序一致 最严格，所有原子操作全局按顺序执行

compare_exchange_weak 与 compare_exchange_strong
都实现了同一个原子语义：

💡 如果当前原子变量的值等于预期值 (expected)，就把它替换为新值 (desired)，并返回 true；
否则，不修改并返回 false，同时 expected 被更新为当前实际值。

名称 含义 是否可能伪失败（spurious failure） 常见用途
compare_exchange_strong 强版本，严格比较 ❌ 不会伪失败（只有比较失败时才返回 false） 用于逻辑上必须“一次性成功”的地方
compare_exchange_weak 弱版本，可能伪失败 ✅ 即使值相等也可能返回 false 通常用于循环重试结构（无锁算法）
场景 队列类型 备注
单生产者 → 单消费者 SPSC 你当前版本完全适用
多生产者 → 单消费者 SPSC + 每生产者独立队列 推荐方式，避免多线程直接竞争队列
多生产者 → 多消费者 MPMC 队列 Michael-Scott 或 Vyukov，实现复杂但安全
多生产者 → 多消费者 + 低延迟 生产者独立 SPSC → 消费者轮询 最常见 HFT 架构

explicit是C++中的一个关键字，它用来修饰只有一个参数的类构造函数，以表明该构造函数是显式的，而非隐式的。当使用explicit修饰构造函数时，它将禁止类对象之间的隐式转换，以及禁止隐式调用拷贝构造函数。防止隐式类型转换，比如禁止写 LockFreeQueue q = 100;

HFT 性能优化原则：

  • 尽量保持单生产者单消费者（SPSC）
  • 避免 CAS 竞争
  • 数据复制要按批处理和 cache line 对齐
