React思想

Fiber 树是虚拟 DOM 吗？不完全等同，但 Fiber 树是 React 对虚拟 DOM 概念的新一代实现。

传统虚拟 DOM（React 16 之前）： 是一个普通的 JavaScript 对象树，表示 UI 的层级结构（类似 { type: 'div', props: { ... } }）。 每次渲染都会生成新的虚拟 DOM 树，并通过 递归 Diff 算法 对比新旧树，计算最小 DOM 操作。

Fiber 树（React 16+）：

  • 增强版的虚拟 DOM，仍然是 JavaScript 对象，但结构更复杂，包含额外的调度信息（如优先级、副作用标记等）。
  • 不再是简单的树结构，而是链表结构（child、sibling、return 指针），支持可中断的异步渲染。
  • 每个 Fiber 节点不仅存储 UI 信息，还存储组件状态、副作用、更新队列等（如 memoizedState、updateQueue）。

Fiber 树是 React 对虚拟 DOM 的优化升级，使其不仅能描述 UI，还能管理渲染调度和状态更新。 传统虚拟 DOM 仅用于 Diff 计算，而 Fiber 树还承担了 任务调度 和 状态管理 的职责。

Fiber 树 vs. 浏览器 DOM

对比维度 Fiber 树（虚拟 DOM） 浏览器 DOM
本质 JavaScript 对象（内存中） 浏览器渲染引擎管理的节点树（内存 + 渲染管线）
更新方式 异步可中断（React 调度控制） 同步更新（直接操作，可能触发重排/重绘）
结构 链表（child、sibling、return） 树形结构（parentNode、childNodes）
存储内容 组件状态、props、副作用标记、更新队列等 纯 UI 节点（元素、属性、样式、事件监听等）
更新策略 增量更新（Diff + 批量提交） 直接修改（立即生效）
性能影响 避免直接操作 DOM，减少重排/重绘 频繁操作可能导致性能问题
调度能力 支持优先级调度（如高优先级交互先渲染） 无调度能力，依赖事件循环

React的渲染机制

组件的本质

组件是什么？

声明式 UI 的单元：组件是状态（state/props）到 UI 的映射函数（UI = f(state)）。

  • 类组件：通过实例化 class 生成，生命周期方法与 Fiber 的 effectTag 和 flags 联动。
  • 函数组件：直接执行函数，依赖 Hooks 与 Fiber 的 memoizedState 链表绑定状态。

与 Fiber 的关系

组件实例的载体：每个组件对应一个 Fiber 节点，Fiber 存储了组件的：

  • 状态（stateNode：类实例或 Hook 链表）
  • props（pendingProps/memoizedProps）
  • 副作用（updateQueue：effect 链表）

更新调度的最小单元：Fiber 的 tag 标记组件类型（如 FunctionComponent、ClassComponent），决定如何调和（Reconcile）。

组件更新的本质

  • 触发更新：setState 或 props 变化时，创建更新（Update 对象）放入 Fiber 的 updateQueue。
  • 调和过程：React 对比新旧 Fiber 树，生成副作用列表（如 DOM 更新、effect 执行）。
  • 提交到 UI：根据副作用列表同步更新 DOM（Commit 阶段）。

为什么不能直接修改 State？

在 React 中，不能直接修改 State 的原因是 React 依赖于 State 的不可变性来优化组件的重新渲染。直接修改 State 会导致 React 无法检测到 State 的变化，从而无法触发组件的重新渲染。为了确保 React 能够正确跟踪 State 的变化，应该使用 setState 或 useState 的更新函数来更新 State。

  1. React 的渲染机制：React 通过对比前后两次渲染的 State 和 Props 来决定是否需要重新渲染组件。如果直接修改 State，React 无法检测到 State 的变化，因为 State 的引用没有改变，React 会认为 State 没有变化，从而不会触发重新渲染。

  2. 不可变性的重要性：React 的设计理念之一是保持数据的不可变性。不可变性意味着每次更新 State 时，都应该创建一个新的对象或数组，而不是直接修改现有的对象或数组。

如何正确更新 State：在 React 中，应该使用 setState 或 useState 的更新函数来更新 State。这些函数会确保 State 的更新是不可变的，并且会触发组件的重新渲染。

与 Fiber 的关系

  • Element 到 Fiber 的转换：createElement 返回的 React Element（普通对象）是 Fiber 节点的“蓝图”。
  • Fiber 的创建依据：
    • element.type 决定 Fiber 的 tag（如函数组件、原生 DOM 节点）。
    • element.props 转化为 Fiber 的 pendingProps。
  • 动态更新：每次渲染生成新的 Element 树，React 根据新旧 Element 树协调 Fiber 树。

JSX 的动态性

  • 条件渲染：JSX 中的逻辑（如 &&、map）会动态生成不同的 Element 结构，触发 Fiber 树的增删移动。
  • 插值能力：{expression} 在运行时求值，通过 props 或 state 驱动 UI 变化。

Fiber 架构的核心作用

  1. 协调（Reconciliation）：

    • 双缓存机制：当前屏幕显示的 Fiber 树（current）和正在构建的 Fiber 树（workInProgress）交替更新。
    • 可中断渲染：Fiber 的链表结构允许拆分渲染任务为小单元（时间切片）。

  2. 状态与副作用的统一管理：

    • 类组件的 this.state、函数组件的 Hook 状态，最终都存储在 Fiber 节点的 memoizedState 或 updateQueue 中。
    • 副作用（如 DOM 操作、useEffect）通过 Fiber 的 flags（如 Placement、Update、Deletion）标记并批量处理。
概念 本质 与 Fiber 的关系
组件开发 状态驱动 UI 的抽象单元 Fiber 是组件的运行时载体，管理状态、副作用和更新逻辑。
JSX 描述 UI 结构的语法糖 编译后的 Element 是 Fiber 节点的输入，决定 Fiber 的创建和更新策略。
Fiber 实现协调和调度的底层数据结构 连接组件逻辑（JSX/Hooks）与实际渲染（DOM/原生视图）的桥梁。

关键结论：

  1. Fiber 是 React 的“引擎”：组件和 JSX 是开发者友好的抽象层，而 Fiber 是实现其动态更新的底层机制。
  2. 响应式编程模型：JSX 声明 UI，组件管理状态，Fiber 负责将状态变化高效同步到 UI。
  3. 性能优化的基础：Fiber 的可中断调度和优先级机制（如并发模式）使得 React 能处理复杂应用的高性能需求。

通常 React 运行时会有两个 Fiber 树，一个是根据当前最新组件状态构建出来的，另一个则是上一次构建出来的 Fiber 树，当然如果是首次渲染就没有上一次的 Fiber 树，这时就只有一个了。简单来说，render 过程就是 React 「对比旧 Fiber 树和新的 element」 然后「为新的 element 生成新 Fiber 树」的一个过程。

props 的本质

在 React Fiber 架构中，props 的本质是组件间数据流动的载体

每个 Fiber 节点对应一个 React 元素（如组件、DOM 节点等），其 pendingProps 和 memoizedProps 属性分别存储了：

  • **pendingProps**：即将生效的新 props（来自父组件的最新渲染结果）。
  • **memoizedProps**：当前已生效的旧 props（上一次渲染时使用的值）。

Props 在 Fiber 树更新中的生命周期

  1. 调和（Reconciliation）阶段：
    • React 比较新旧 Fiber 树，生成更新队列（Update Queue）。
    • 若父组件传递的 props 变化，子 Fiber 节点的 pendingProps 会被更新，标记为需要处理（effectTag）。
  2. 提交（Commit）阶段：
    • 将 pendingProps 应用到组件实例，更新为 memoizedProps。
    • 对于 DOM 节点，props 会转换为实际的 HTML 属性（如 className、onClick 等）。

Hook的底层本质

React Hooks 的本质是通过 Fiber 节点的内部机制（如 memoizedState 链表和 queue）实现状态和副作用的逻辑复用，其核心原理与 Fiber 架构深度绑定。以下是各类 Hook 在 Fiber 树中的具体实现机制：

Hook 的通用存储结构

每个函数组件对应的 Fiber 节点中，Hooks 以单向链表的形式存储在 memoizedState 属性上，每个 Hook 节点包含：

  • 链表顺序严格依赖调用顺序（因此 Hook 不能条件调用）。
  • 每次组件渲染时，React 会遍历该链表，按顺序“消费” Hook。

常见 Hook 在 Fiber 中的工作原理

1. useState / useReducer

存储位置：hook.memoizedState 存储当前状态值。

更新机制：调用 setState 时，将 action 存入 hook.queue。在渲染阶段（如 beginWork），React 处理队列中的更新，计算出新状态，更新 memoizedState。

2. useEffect / useLayoutEffect

存储位置：hook.memoizedState 存储 effect 对象（包含回调函数、依赖项等）。

调度机制：

  • 渲染阶段：收集 effect 到 Fiber 的 updateQueue 中（区分 PassiveEffect 和 LayoutEffect）。
  • 提交阶段：依次执行 effect 的创建/销毁函数。

依赖比对：通过比对 deps 决定是否重新调度 effect。

3. useMemo / useCallback

存储位置：hook.memoizedState 存储缓存的值或函数。

更新条件：依赖项变化时重新计算（浅比较）。

4. useRef

  • 存储位置：hook.memoizedState 直接存储 { current: initialValue } 对象。
  • 特性：绕过 Fiber 的更新机制，修改 current 不会触发渲染。

Hook 与 Fiber 树的联动流程

  1. 挂载阶段（Mount）：首次渲染时，构建 Hook 链表并初始化状态（写入 memoizedState）。对于 effect，标记 Fiber 的 flags（如 PassiveEffect）以调度副作用。

  2. 更新阶段（Update）：按顺序读取 Hook 链表，处理更新队列（如 setState 的 action）。对比依赖项决定是否更新缓存（useMemo）或重新调度 effect。

  3. 提交阶段（Commit）：执行 useLayoutEffect 的回调（同步）。异步调度 useEffect 的回调（通过 scheduleCallback 放入微任务队列）。

React的优势

React 相比传统的直接操作 DOM 渲染，在开发效率、性能优化和可维护性上具有显著优势，其核心优势源于 虚拟 DOM和 Fiber 架构 的协同设计。以下是具体对比：

性能优化：减少不必要的 DOM 操作

传统 DOM 渲染的问题

  • 直接操作 DOM 成本高： 浏览器 DOM 操作（如 appendChild、style.color = 'red'）会触发 重排（Reflow）和重绘（Repaint），频繁操作会导致性能瓶颈。
  • 手动优化困难： 开发者需要自行计算最小 DOM 变更（如手动缓存 DOM 查询结果、批量操作），代码复杂且易出错。

React 的解决方案

  • 虚拟 DOM Diff 算法： React 先在内存中对比新旧虚拟 DOM（Fiber 树），计算出最小的 DOM 操作（如仅更新 className 而非整个 div）。

    批量更新（Batching）：将多个 setState 合并为一次 DOM 更新（减少重排/重绘次数）。

  • 时间切片（Time Slicing）： Fiber 架构将渲染任务拆分为可中断的微任务，避免长时间阻塞主线程（如大列表渲染不会卡死 UI）。

开发效率：声明式编程 vs. 命令式编程

传统 DOM 渲染（命令式）

  • 开发者需要手动描述“如何做”： 精确控制 DOM 的每一步操作（如创建节点、添加子节点、绑定事件）。
  • 代码冗长且易出错： 随着交互复杂度上升，代码会变得难以维护（如动态表单、动画联动）。

React（声明式）

  • 开发者只需描述“UI 应该是什么”： 通过 JSX 声明 UI 结构，由 React 负责 DOM 操作的具体实现。
  • 状态驱动 UI： 只需修改 state 或 props，React 自动处理 UI 更新（无需手动同步 DOM）。

跨平台能力

传统 DOM 渲染的局限

强依赖浏览器环境，难以直接迁移到其他平台（如移动端、桌面端）。

React 的抽象层优势

虚拟 DOM 是平台无关的：React 可以对接不同渲染器，将虚拟 DOM 转换为：

  • 浏览器 DOM（react-dom）
  • 原生移动组件（React Native）
  • 终端 UI（如 ink 控制台库）
  • 甚至 WebGL（如 react-three-fiber）
