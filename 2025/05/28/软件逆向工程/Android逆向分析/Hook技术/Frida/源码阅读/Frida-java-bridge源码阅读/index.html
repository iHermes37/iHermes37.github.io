从 Dalvik 到 ART 的演进

ArtMethod 是 Android Runtime (ART) 的核心数据结构之一，它的设计源于对 Dalvik 虚拟机 的优化和改进。要理解 ArtMethod 的由来，需要回顾 Android 运行时的历史：

运行时 执行方式 方法表示 问题
Dalvik 解释执行 + JIT（后期支持） Method 结构体 1. 解释执行慢 2. JIT 编译开销大
ART AOT 编译 + JIT（优化补充） ArtMethod 结构体 优化执行效率，减少运行时开销
  • Dalvik 的 Method 结构:在 Dalvik 中，Java 方法通过 Method 结构体表示，主要存储字节码信息，执行时依赖解释器逐条解释字节码，性能较低。
  • ART 的改进:ART 在 Android 5.0（Lollipop）引入，核心变化是 ​​AOT（Ahead-Of-Time）编译​​，即在安装时就将 Java 字节码编译成机器码。为了高效管理编译后的方法，ART 设计了 ArtMethod。

针对Method结构体，不得不谈谈xposed，还有第一代指令抽取加固方式。实际就是找到Method 中的 insns指令码字段（加固时是在dex文件，hook则直接获取函数指针得到该指令码字段），然后运行时替换。 而ArtMethod，是ART下的产物。其结构教于Method发生了变化，当dex2oat的时候，还会有多种方法内联优化。而方法内联会改变原本的方法分布和调用流程，对加固、热修复、hook，实际上都有一定的影响。

ArtMethod 的诞生主要为了解决以下问题：

  1. 快速方法调用
    • Dalvik 的方法调用需要经过解释器，而 ART 希望直接跳转到编译后的机器码。
    • ArtMethod 存储 入口地址（entry_point），使得调用可以直接定位到机器码。
  2. 支持 AOT 和 JIT: AOT 编译后的方法需要固定入口，而 JIT 可能动态优化方法，ArtMethod 的 entry_point 可以动态更新。
  3. 支持虚方法（多态）:通过 ArtMethod 的 方法表（vtable） 实现高效的多态调用。

所以我们说在Android的ART虚拟机中，ArtMethod是Java方法在Native层的表示形式，表示了 Java 方法 的核心数据结构，其存储一个 Java 方法的 元数据（如类名、方法名、签名）和 执行逻辑。在 ART 中，所有 Java 方法调用（包括虚方法、静态方法、构造函数）都通过 ArtMethod 实现。

示例：调用MyClass.foo()时，ART通过ArtMethod的入口地址跳转到执行代码。

在 ART 虚拟机中，对于方法的调用，大部分会调用到 ArtMethod::Invoke，因此我们可以以此为起点理解实际调用的过程。

源码过程分析

ArtMethod::Invoke

在之前的文章中，我们知道android APP会经过即时编译（JIT）和预先编译（AOT）机制 会被编译优化，那么当App运行java层函数就有了如下三种情况：

  • AOT编译代码（直接跳转）
  • JIT代码缓存（动态优化后的代码）
  • 解释器桥接（未编译时进入解释执行）

所以在ArtMethod::Invoke中主要分为两种情况

  • 一种是ART未初始化完成或者系统配置强制以解释模式运行，此时则进入解释器另一种情况是有native代码时，比如JNI代码、OAT提前编译过的代码或者JIT运行时编译过的代码以及代理方法等，此时则直接跳转到invoke_stub去执行，
  • 对于解释执行的情况，也细分为两种情况，一种是真正的解释执行，不断循环解析CodeItem中的每条指令并进行解析；另外一种是在当前解释执行遇到native方法时，这种情况一般是遇到了JNI函数，这时则通过method->GetEntryPointFromJni()获取对应地址进行跳转。

frida-java是如何连通到java世界？

Native API

  • 统一抽象层：封装不同 Java 虚拟机（Dalvik/ART/JVM）的底层操作（如方法调用、内存管理）。
  • 跨平台支持：通过动态检测运行环境（Android/Oracle JVM），自动适配对应的虚拟机实现。
  • 基础设施：为上层 Java Hook（如 Java.perform）提供底层能力。

虚拟机类型判断

java-bridge 的实现是基于 inline-hook (gum-js) 对目标 Java 虚拟机中的符号进行解析、调用、替换从而实现 Java Hook 的功能，其支持的三种虚拟机及其对应动态库分别是:

  • Dalvik: libdvm.so
  • ART: libart.so
  • JVM: jvm.(dll|dylib|so)

Frida通过JS与Java交互的实际流程是：

  1. JS → Native Bridge：Frida的JS代码通过Frida的核心引擎(用C编写的)与目标进程交互
  2. Native → JNI：在Android/JVM环境中，Frida通过JavaVM和JNIEnv接口与Java层交互
  3. 关键桥梁：
    a) Native层访问JNI能力：需要获取JavaVM指针
    b) 获取正确的JNIEnv：每个线程需要正确的JNIEnv指针

总的来说frida-java通过两步实现js世界到java世界的单向通道，首先利用frida-gum提供的js接口操作native世界，然后再基于jni连通到java世界。

主要步骤如下：

  1. 连通native世界，获得拟机的接口，如JNI_GetCreatedJavaVMs
  2. 获得JavaVM
  3. 获得JNIEnv
  4. 获得java类的class引用
  5. 操作该java类，如创建对象，调用方法等。

查找偏移

为什么要查找这些偏移？

ART 运行时（以及之前的 Dalvik）的核心功能（如类加载、方法调用、垃圾回收等）是由 C++ 对象（如 art::Runtime、art::ClassLinker、art::ThreadList 等）管理的。这些对象的成员变量（如 heap_、class_linker_、instrumentation_）的 内存布局（偏移） 在不同 Android 版本中可能不同，但它们的 逻辑功能 是相同的。
需要通过偏移来：

  • 定位关键数据结构（如修改 ClassLinker 以拦截类加载）。
  • Hook 关键函数（如替换 art::Instrumentation 的方法入口点）。
  • 直接操作运行时内存（如修改 ArtMethod 的字段或字节码）。

MethodMangler

MethodMangler 同样是 API 定义的一个工具类，主要功能是用于实现 Java 方法封装和 Hook 操作。

每次 hook 指定方法的时候都会使用对应方法的 methodId 构造一个 MethodMangler，然后使用 replace 函数来替换目标方法实现；revert 则用来恢复原来的方法。

通过指定方法的 implementation 属性赋值来实现函数替换的操作本质上就是通过 MethodMangler 完成的

当对目标方法赋值时，调用了 set 方法，首先会判断是否之前有替换过，如果有的话直接将原 hook 函数替换，从这里可以看出 frida 在 针对 Java 目前还没有实现多重 hook；对于未 hook 情况，会创建对应的 MethodMangler 并保存到 Method._r._m 中。

  • 标志位含义：如 kAccFastNative 在不同版本的行为不同

Frida 的解决方案是 动态探测字段布局，而非硬编码偏移量。

通过特征搜索在内存中定位字段：

关键字段目标:

字段 作用 探测依据
jniCode JNI 实现入口 匹配已知 JNI 桩代码地址
quickCode Quick 编译模式入口 匹配 AOT 编译的机器码特征
interpreterCode 解释器模式入口 匹配解释器分发函数地址
accessFlags 方法属性（如 native/static） 通过已知方法标志位反推

探测结果保存在 originalMethod 中，供后续操作复用：

方法替换的详细流程

(1) 创建方法副本
  • 在堆上分配新的 ArtMethod 结构体
  • 完全复制原始方法的所有字段
(2) 关键字段修改

通过 patchArtMethod 修改副本：

修改逻辑详解

字段 修改值 目的
jniCode JS 函数转换的 NativePointer 使方法调用跳转到 Frida 的代理逻辑
accessFlags 添加 kAccNative 标志 强制 ART 将其视为 JNI 方法，走 JNI 调用流程
quickCode Generic JNI 桥接跳板 Quick 模式调用时跳转到 jniCode
interpreterCode 解释器→编译代码桥接 解释器模式调用时跳转到 Quick 模式
(3) 原子替换

将原始方法的入口指针指向新副本：

jniCode 替换为用户指定的 js 函数封装而成的 NativeFunction，并将 accessFlags 设置为 kAccNative，即 JNI 方法。quickCode 和 interpreterCode 分别是 Quick 模式和解释器模式的入口，替换为上文中查找保存的 trampoline，令 Quick 模式跳转到 JNI 入口，解释器模式跳转到 Quick 代码，这样就实现了该方法的拦截，每次执行都会当做 JNI 函数执行到 jniCode 即我们替换的代码中。

如果待 hook 方法是解释执行，且使用了 nterp，那么 frida 会将其入口替换为 art_quick_to_interpreter_bridge 来跳出解释器，并进入到实际的 quickCode 去执行，从而让我们的方法替换功能保持完整。

原始调用流程:

flowchart TD
    A[Java调用] --> B{执行模式}
    B -->|Quick模式| C[entry_point_from_quick_compiled_code_]
    B -->|解释器模式| D[entry_point_from_interpreter_]
    B -->|JNI模式| E[direct_code_entry_]

Frida 修改后的流程

flowchart TD
    A[Java调用] --> B{执行模式}
    B -->|Quick模式| C["quick_generic_jni_trampoline"] --> F[修改后的jniCode]
    B -->|解释器模式| D["art_quick_to_interpreter_bridge"] --> C --> F
    B -->|JNI模式| F
    F --> G[Frida JS回调]
    G -->|可选| H[调用原方法]

虽然此时我们已经将目标 ArtMethod 改成了 Native 方法，且 JNI 的入口指向我们的 hook 函数，但如果该方法已经被 OAT 或者 JIT 优化成了二进制代码，此时在字节码层调用 invoke-xxx 时会通过方法的 entry_point_from_quick_compiled_code_ 直接跳转到 native 代码执行，而不是 quick_xxx_trampoline。

因此对于这种情况，我们可以将 entrypoint 的地址重新指向 trampoline，但如前文所说，对于系统函数而言，其地址已知，因此调用方被优化后很可能直接就调转到了对应的 native 地址，而不会通过 entrypoint 去查找。因此 frida 采用的方法是直接修改目标方法的 quickCode 内容，将其替换为一段跳板代码，然后再间接跳转到我们的劫持实现中。

quickCode 的替换使用使用 ArtQuickCodeInterceptor 完成，姑且理解为跳板管理器，保存在 MethodMangler 的 interceptor 属性中，这样在程序退出或者用户调用 method.implementation = null 时还会调用 interceptor.deactivate 恢复方法，防止残留 hook 代码导致的运行时崩溃。

sequenceDiagram
    participant JS as JavaScript
    participant ArtMethod
    participant Trampoline
    participant OriginalCode

    JS->>ArtMethod: 1. 修改 entry_point_ 为跳板地址
    ArtMethod->>Trampoline: 2. 调用跳板代码
    Trampoline->>JS: 3. 执行 JavaScript 回调
    JS->>Trampoline: 4. 返回控制权
    Trampoline->>OriginalCode: 5. 跳回原方法
    OriginalCode-->>ArtMethod: 6. 继续执行原逻辑
