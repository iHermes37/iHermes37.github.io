了解SVM

概述

SVM 是 Solana 的虚拟机的高层次框架，它定义了整个交易处理管道和程序执行流程，并且包括了特定的指令集架构（SVM ISA）和系统调用（syscalls）。

rBPF 是 Solana 实现的 eBPF 虚拟机，负责执行编译后的 eBPF 程序。在 Solana 中，rBPF 是用于执行智能合约的实际虚拟机。

SVM 和 rBPF 的关系是：SVM 是一个更广泛的概念，涉及 Solana 的整个交易和程序执行流程，而 rBPF 是实现这一流程的 虚拟机引擎。

Solana 程序通常使用 Rust 编写，然后通过 Solana 平台工具编译为 eBPF。然而，程序也可以用 Zig、C 或汇编语言编写。平台工具确保程序编译到适当的 eBPF 格式，同时遵守 Solana VM 施加的 eBPF 限制

rBPF 与传统 eBPF 虚拟机

1. eBPF 的基本概念

  • eBPF（Extended Berkeley Packet Filter） 是一种高效的 虚拟机，它最初是用于操作系统内核中进行网络数据包过滤。eBPF 程序可以在内核中执行，但它们受到严格的限制，不能执行某些可能危害系统稳定性的操作。
  • eBPF 虚拟机：eBPF 程序在执行时受到虚拟机的约束，执行的是一组指定的低级指令，保证了程序的安全性和效率。它通常运行在 内核空间，以执行网络监控、安全检查等任务。

2. rBPF 的介绍

  • rBPF（Rust Berkeley Packet Filter）是一个 用 Rust 编写的 eBPF 虚拟机实现，它遵循 eBPF 的基本设计原则，但并不局限于运行在内核中，而是可以在 用户空间 中运行。
  • 用户空间与内核空间的区别：
    • 内核空间 是操作系统内核的一部分，直接与硬件交互，具有更高的权限和能力。
    • 用户空间 是操作系统中的应用程序运行区域，相比内核空间，它的权限更有限，操作也受到更严格的控制。
    • rBPF 是在 用户空间 运行的虚拟机，这对于 Solana 来说非常合适，因为 Solana 的验证者节点程序（即执行程序的运行时）也运行在用户空间。

3. rBPF 在 Solana 中的应用

  • Solana 和 rBPF：Solana 使用 rBPF 作为其程序执行的核心虚拟机（VM）。在早期，Solana Labs 将 rBPF 从原始的实现中分叉出来，并对其做了 定制修改，以支持 Solana 特有的功能和要求。这个修改后的版本被称为 Solana rBPF，它被用于 Solana 验证者节点 上执行智能合约（程序）。
  • SVM（Solana Virtual Machine）：虽然在 Solana 的开发者社区中，大家习惯性地将整个交易处理管道或执行层称为 “SVM”，但实际上 SVM 指的是 Solana 虚拟机指令集架构（SVM ISA）所定义的 eBPF 虚拟机。也就是说，SVM 是 Solana 运行时的一个总体概念，指代了包括交易处理和程序执行的整个执行过程，而 rBPF 则是执行这些程序的具体虚拟机。

4. rBPF 与纯正 eBPF 的区别

  • 内核 vs 用户空间：

    传统 eBPF 运行在内核空间，主要用于网络数据包过滤、性能监控和安全性分析等任务。rBPF 运行在 用户空间，这使得它更适合于 Solana 的要求，因为 Solana 的节点（验证者）运行时是在用户空间中。

  • 针对 Solana 的定制：

    Solana 在 rBPF 的基础上进行了 定制化修改，添加了 Solana 特有的功能，以支持智能合约的执行。Solana 使用了 SVM ISA（Solana 虚拟机指令集架构）来约束 rBPF 程序的行为，使得它们符合 Solana 网络的需求，例如在网络上进行交易的处理和智能合约的执行。此外，Solana 的虚拟机还支持对一些 系统调用（syscalls） 的访问，这些调用是 Solana 协议定义的低级操作，用于与网络、存储等其他系统组件进行交互。

Firedancer 是 Solana 项目的另一个重要部分，它是一个 完全重新实现的虚拟机，遵循了 Solana 的 SVM ISA。与 rBPF 不同，Firedancer 是一个新的虚拟机实现，旨在提高性能和可扩展性，但它仍然遵循与 rBPF 类似的设计思想，并与 Solana 网络兼容。

**SVM **

Solana网络的节点，称为验证者，每个节点都运行自己的Solana虚拟机（SVM）独立环境，以在区块链上保持共识。当部署智能合约时，它会将所需的状态变化传达给运行时。Solana运行时将这些状态变化转发给每个验证者系统内运行的SVM实例。所有验证者节点都会接收到并翻译这些变化，更新区块链。

这些SVM实例可以看作是执行必要操作的小型计算机，根据交易提供的指令更新Solana网络的状态。

SVM（Solana Virtual Machine）

SVM 这个术语常常被用来描述 Solana 网络的虚拟机，即用于执行交易和智能合约的整个执行环境。它是 Solana 运行时的一部分，负责在节点上执行智能合约（Solana 程序），处理交易、验证交易的有效性以及更新区块链状态。

SVM 不仅仅是一个虚拟机，它实际上包括整个 交易处理管道 和 执行层，也就是说，SVM 涉及了如何将交易从接收、验证到最终执行和区块生成的所有处理步骤。

  • 实际上，SVM 是基于 eBPF（扩展伯克利数据包过滤器） 的虚拟机架构，其目的是提供一个高效且安全的环境来执行程序。这个虚拟机执行的是符合 Solana 协议约束的 eBPF 程序。

SVM 和 rBPF 的关系

  1. SVM 和 rBPF 都是基于 eBPF 的执行环境：

    • SVM 作为 Solana 的虚拟机体系，实际上是建立在 eBPF 虚拟机（rBPF）的基础上的。SVM 通过 rBPF 虚拟机执行 Solana 程序（智能合约），并受到 Solana 自定义的虚拟机指令集架构（SVM ISA）的约束。
    • rBPF 是实现 eBPF 执行的工具，而 SVM 是使用 rBPF 来完成智能合约执行的整体框架。

  2. rBPF 是 SVM 的核心执行引擎：

    • rBPF 是一个 eBPF 虚拟机，负责执行 Solana 网络中的 eBPF 程序（即 Solana 智能合约）。这些程序可以用 Rust、C、Zig 等语言编写，然后通过 Solana 的工具编译成 eBPF 字节码。
    • SVM 的实际操作和执行是由 rBPF 虚拟机 执行的。SVM 定义了一个程序的执行过程和网络的协议约束，而 rBPF 实际上是运行这些程序的虚拟机。

  3. SVM 和 rBPF 的协同工作：

    SVM 是一个 高层次概念，它定义了 Solana 中的程序执行环境，包括交易处理管道、区块链状态的更新和程序执行的所有步骤。而 rBPF 则是 SVM 实现的一个具体的虚拟机，它负责在 Solana 节点上执行 eBPF 程序，即执行智能合约的具体指令集。

    SVM 负责确保程序符合 Solana 的运行时规范，并控制程序的执行流，而 rBPF 则负责实际的指令执行和资源管理。

  4. 系统调用（syscalls）：

    在 rBPF 虚拟机中，执行的程序会依赖于 Solana 协议定义的系统调用（syscalls）。这些系统调用为程序提供访问外部资源和执行协议层操作的能力。

    SVM 指定了这些系统调用，并通过 rBPF 执行它们。这使得 rBPF 能够在虚拟机中执行更复杂的操作，而不仅仅是简单的计算。

  5. Firedancer 与 rBPF：

    Firedancer 是 Solana 的另一个虚拟机实现，与 rBPF 的区别在于，Firedancer 是 完全重新实现的虚拟机，它也遵循了 SVM ISA，但它采用了不同的设计和优化。

    • Firedancer 是为 Solana 提供更高性能的虚拟机，它与 rBPF 类似，但不完全相同，并且专注于提升执行效率。

Sealevel引擎

SVM的关键组件是Sealevel引擎。Sealevel引擎通过允许多个智能合约同时运行而不影响彼此的性能，实现了Solana执行环境内的“水平”扩展，即并行处理。是 Solana 网络中用于处理并行交易执行的 并发引擎，它是 Solana 提供高吞吐量和低延迟的关键技术之一。Sealevel 使得 Solana 能够在同一时间并行处理多个交易，从而极大地提高了交易处理能力。

Sealevel 的核心概念

  1. 并行交易执行：

    在传统区块链系统（如以太坊）中，所有的交易必须按照先后顺序依次执行，这限制了交易的吞吐量。每个交易都必须等到前一个交易执行完才能继续执行，这就导致了处理效率低下。

    Sealevel 引入了 并行交易执行，它允许多个交易同时进行处理，显著提升了系统的吞吐量。Solana 通过分析交易之间的依赖关系，能够决定哪些交易可以并行执行，哪些交易必须顺序执行。

  2. 事务的并行性：

    Sealevel 引擎的工作原理基于事务的 数据依赖性分析。每个交易涉及的账户和数据可以被视为一组资源。Sealevel 通过分析这些资源的访问模式，确定哪些交易是独立的（即可以并行执行），哪些交易之间存在依赖关系（需要顺序执行）。如果多个交易不访问同一账户或同一数据，那么它们可以同时执行（并行）。如果交易访问了相同的账户或数据，它们则必须按照顺序执行。

  3. 状态和账户锁定机制：

    在 Sealevel 中，交易的执行是基于 账户 或 数据 的锁定机制的。当交易需要访问某个账户的数据时，Sealevel 会 锁定 该账户，确保同一时刻只有一个交易可以修改它。通过智能的账户锁定机制，Sealevel 可以保证交易的一致性和完整性，同时避免竞争条件（race conditions），并确保并行执行不引入错误。

  4. 优化并行度：

    Sealevel 对多个交易之间的依赖关系进行 静态分析，通过标识哪些交易之间相互独立，哪些需要按照顺序执行，进一步提高了并行度。这种优化减少了无效的等待时间，从而提升了吞吐量。

Sealevel 的工作原理

  1. 交易执行流程：

    每个交易都会被解析并确定它所涉及的账户和数据。如果交易中涉及多个账户，Sealevel 会检查这些账户是否有冲突或是否可以并行执行。在 Solana 中，交易的执行并不依赖于传统的单一事务执行模型，而是通过并行化计算来提升性能。Sealevel 会通过对 资源访问冲突的检查 来决定哪些交易可以并行，哪些交易需要排队等待。

  2. 并行性分析：

    Sealevel 使用的 并行性分析 是基于 账户访问分析，也就是说它在执行前通过检查交易所访问的账户是否重叠来判断是否可以并行执行。

    • 比如，如果两个交易都修改不同的账户（没有交集），则这两个交易可以并行执行；但如果两个交易都修改相同账户的数据，那么它们就必须顺序执行。

  3. 并行事务处理：

    Sealevel 不仅可以识别不同交易之间的独立性，还能根据节点的硬件资源进行优化，在多核 CPU 上并行执行交易，进一步提高吞吐量。

    集群中的并行化执行：Solana 网络中的多个验证者节点可以并行执行多个交易，通过高效的通信和共识机制，最终合并每个节点处理的结果。

SVM是多线程运行环境，利用验证者机器的所有核心并行处理多笔交易，使Solana随着硬件改进而更有效地扩展。SVM架构还改进了交易费用管理，开发了本地化费用市场，每个智能合约单独分配费用

概念

在 Solana 中，程序执行的存储空间（即智能合约的执行环境）与以太坊的 EVM 不同，主要通过 账户模型 进行管理。Solana 的存储体系不再细分为栈、内存和存储，而是通过以下几种存储结构来管理链上数据：

1. 账户（Account）

  • 账户 是 Solana 中存储数据的基本单位。Solana 网络的所有数据（包括智能合约状态、交易数据、资产余额等）都存储在账户中。

    每个账户都有一个 数据字段，这通常是智能合约的状态数据。Solana 的程序会使用这些账户来存储执行状态、用户数据、交易结果等信息。

    账户中存储的数据结构可以是任何类型的应用程序数据，通常是以字节数组的形式存储，智能合约程序会根据自己的需求对这些字节进行解析。

2. 程序账户（Program Account）

  • 程序账户 存储 Solana 程序（即智能合约）的执行代码。这些程序账户包含了编译后的可执行字节码（即智能合约代码），并通过 Solana 网络中的 BPF（Berkeley Packet Filter）虚拟机 执行。

    这个账户本质上是智能合约的部署位置，程序账户的地址通常被称为 Program ID，它用来标识程序并执行合约逻辑。

3. 数据账户（Data Account）

Solana 的 数据账户 存储程序状态。这些账户由程序控制，并且可以通过 程序账户 来修改。智能合约需要额外的账户来存储它的状态信息，通常在智能合约执行时通过 System Program 创建数据账户并将其与智能合约绑定。

这些账户通常有固定的存储空间，以适应程序的状态需求。不同的智能合约可能需要不同的数据账户，每个账户存储与合约相关的不同类型数据。

4. 内存（Memory）

在 Solana 中，内存 并不像 EVM 中那样是一个单独的抽象层。Solana 程序在执行时，使用 Solana 的 Rust 或其他语言的运行时内存来处理局部计算和临时数据存储。该内存是运行时的一部分，且由执行环境本身管理。需要注意的是，Solana 的程序执行是 无状态 的，程序本身并不直接管理永久性的内存数据，所有状态数据必须存储在外部的 账户 中。

5. 交易（Transaction）

Solana 的交易本身也包含了可执行的程序调用和数据传输，交易是由多个账户参与的，交易内的数据是程序执行时的输入（包括 用户账户 和 程序账户 中的数据）及输出。执行的结果会写入相关的账户（数据账户）。

6. 程序状态（Program State）

每个智能合约会通过 数据账户 来存储它的状态。这些状态是由程序账户创建和管理的，并且可能会被多个交易修改。状态存储 是由程序账户定义的，当某个智能合约需要读取或修改状态时，程序账户会确定哪些数据账户可以访问或更新。

Solana 存储模型的特点

  1. 无状态程序：

    Solana 的智能合约本质上是 无状态 的。也就是说，Solana 程序本身不保存自己的状态，所有的状态数据都保存在外部的账户中。这与以太坊的 EVM 存储模型有所不同，以太坊中的合约是有状态的，它们的状态直接存储在合约内。

  2. 高效的存储访问：

    Solana 的账户模型使得数据存储访问非常高效。智能合约（程序）可以通过操作账户来管理链上的状态，而这些操作不依赖于复杂的栈或内存结构。

  3. 灵活的存储设计：

    每个账户都是一个独立的存储单位，程序开发者可以根据需求定义每个账户的数据结构。这种模型灵活性较高，可以适应不同类型的应用需求。

Solana 程序的内存资源管理实际上是由节点的 操作系统 负责的。Solana 程序运行时在节点的内存空间中进行计算和临时数据存储。并且，这些内存资源不是由 eBPF 虚拟机本身管理，而是通过节点的操作系统进行分配。

与 EVM 不同，EVM 自身管理栈、内存和存储资源，Solana 的内存管理则通过节点操作系统和 eBPF 虚拟机进行协作管理。
