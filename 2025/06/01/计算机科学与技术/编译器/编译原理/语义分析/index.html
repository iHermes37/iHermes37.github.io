语义分析

语义分析做什么事情？

语法分析器所做的事情：输入程序语言的语义规则，输出Yes或者No。同时语义分析器也是代码和编译器的工作界限，一段代码如果通过了语义分析器再有任何问题就不属于代码的问题，而是编译器的问题。

程序语言的语义规则：与之前的语法分析规则一样，语义分析也是计算机科学中的一个研究点，这是一个难点。不过大多数语言都是采取自然语言描述的，是教条的，是人为规定的，例如在C语言中会告诉你加号前后应该是整型数字或浮点型数字。编程语言的使用者可以仅仅知道加号两边只用数字就行了，但是实现者必须知道 浮点数+浮点数、浮点数+整数等等是否属于一个语言中的语义。

输入 输出
抽象语法树（AST） 经过语义检查的AST（附加类型、作用域等属性）。
符号表 更新后的符号表（含类型和作用域信息）。
错误报告 检测到的语义错误（如类型不匹配、未声明变量）。

语义分析的核心任务

任务 说明 示例
类型检查 验证操作符与操作数的类型是否兼容。 int a = "hello"; → 报错（字符串不能赋给整型）。
变量声明检查 确保变量在使用前已声明，且未重复定义。 x = 10;（若x未声明）→ 报错。
作用域分析 检查变量和函数的作用域是否合法（如局部变量不能在外部访问）。 { int x; } print(x); → 报错（x超出作用域）。
函数调用检查 验证函数是否存在、参数数量和类型是否匹配。 foo(1, "a")（若foo需两个整型参数）→ 报错。
控制流合法性 检查break、continue、return是否出现在正确位置。 if (x) break;（若不在循环内）→ 报错。
常量折叠 优化常量表达式（如2 + 3替换为5）。 int x = 2 + 3; → 优化为int x = 5;。
歧义性检查 处理可能的多义性（如运算符重载时的优先级冲突）。 a + b * c 确保按a + (b * c)计算。

语义分析是编译器前端最后一道关卡，确保程序逻辑正确性。

  • 核心依赖：AST（结构） + 符号表（上下文）。
  • 关键检查：类型、作用域、控制流、函数调用。

符号表处理作用域的方法

方法一：用一张表，可以通过栈来实现控制作用域。

每个程序块中的变量都是独立的，我们通过栈来实现，进入到内部的数据块（进入作用域时）就压栈（插入元素），退出作用域的时就弹栈删除元素。这样就可以保证栈顶元素是最内部的作用域，外部的作用域在栈底。

再次回顾一下前面所说的：

  1. 词法分析（Lexical Analysis） → 生成Token流
  2. 语法分析（Syntax Analysis） → 生成AST（抽象语法树）
  3. 语义分析（Semantic Analysis） → 验证逻辑正确性
  4. 中间代码生成（Intermediate Code Generation） → 输出IR（如三地址码、LLVM IR）

部分编译器将语义分析和中间代码生成合并（如Clang在AST上直接生成LLVM IR）。

可以看到语义分析本身不直接生成二进制或中间代码，而是为后续阶段提供验证后的程序表示。为中间代码生成提供已验证的AST（如类型注解后的树）。中间代码生成是前端最后一步，将AST转换为IR，后端进一步处理为二进制。下面一节将讨论中间代码生成。
