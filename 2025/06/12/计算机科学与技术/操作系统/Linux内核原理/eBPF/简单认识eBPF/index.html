什么是 eBPF ? An Introduction and Deep Dive into the eBPF Technology

eBPF（Extended Berkeley Packet Filter） 是一种运行在 Linux 内核中的高效、安全的虚拟机技术，允许开发者在不修改内核源码或加载内核模块的情况下，动态注入自定义程序来监控、分析和优化系统的运行状态（如网络、性能、安全等）。它通过即时编译（JIT）和严格的验证机制确保安全，广泛应用于实时观测、网络加速、安全检测和故障排查等领域，是现代可观测性和高性能网络的核心技术之一。

整体运行流程

第一步，用户态使用 C 语言开发一个 eBPF 程序；用户态开发者需要 显式编写代码，指定 eBPF 程序绑定到哪个事件。

  • 用户态程序 必须明确指定 事件类型（如 kprobe、tracepoint、XDP）。
  • 事件名称（如 sys_enter_openat）需要与内核支持的符号匹配。

第二步，借助 LLVM 把 eBPF 程序编译成 BPF 字节码【用 clang 把 C 代码编译成 字节码（eBPF 的“机器语言”）】

第三步，用户态程序调用 bpf() 系统调用，将字节码送入内核；把 eBPF 程序挂载到 事件点（如系统调用、网络包到达）

第四步，内核验证并运行 BPF 字节码，事件发生时运行，用户态程序从 eBPF Map 或日志中读取数据并把相应的状态保存到 BPF 映射中；

第五步，用户程序通过 BPF 映射查询 BPF 字节码的运行状态。

内核钩子

内核预定义的钩子类型

有些是 Linux 内核原生支持的基础机制，有些则是 专为 eBPF 设计或强化的接口。

eBPF 支持多种钩子类型，每种类型的实现机制不同：

钩子类型 实现机制 示例事件
kprobe 动态替换内核函数的指令（在函数入口插入断点） kprobe/do_sys_openat2
tracepoint 内核静态定义的跟踪点（代码中显式的 trace_ 调用） tracepoint/syscalls/sys_enter_openat
XDP 网卡驱动接收数据包时的早期处理点 XDP 程序直接处理网络包
perf_event 利用性能监控单元（PMU）触发事件（如 CPU 周期、缓存命中） perf_event 周期采样
cgroup 控制组（cgroup）的特定操作（如文件打开、网络包过滤） cgroup/skb 过滤网络包

内核如何动态挂载 eBPF 程序？

以最常见的 kprobe 为例，内核的挂载流程如下：

  1. 用户态发起请求：
    调用 bpf(BPF_PROG_ATTACH) 或 ioctl，指定要绑定的 eBPF 程序和事件（如 do_sys_openat2）。
  2. 内核查找符号：
    内核解析事件名称，找到对应的函数地址（通过 kallsyms_lookup_name()）。
  3. 修改指令：
    在目标函数的入口地址插入断点指令（如 int3 on x86），触发时跳转到 eBPF 程序。
  4. 执行 eBPF 程序：
    当断点被触发时，内核调用 eBPF 虚拟机或 JIT 编译的代码。

关键数据结构：

  • struct bpf_prog：代表一个加载的 eBPF 程序。
  • struct bpf_prog_array：存储多个绑定到同一事件的 eBPF 程序（支持链式调用）。

eBPF 字节码从加载到挂载

当用户态程序通过 bpf() 系统调用将 eBPF 字节码送入内核时，内核会经历一系列复杂的步骤来解析、验证并最终将程序挂载到指定事件上。

1. 用户态发起 bpf() 系统调用

用户态程序（如通过 libbpf 或 BCC）调用 bpf() 系统调用，主要涉及两个关键操作：

  • **BPF_PROG_LOAD**：加载字节码到内核。
  • **BPF_PROG_ATTACH**：将已加载的程序绑定到事件钩子。

2. 内核的字节码处理流程

内核收到 bpf() 请求后，按以下步骤处理：

（1）验证阶段（Verifier）

指令合法性检查：确保所有指令是 eBPF 规范允许的操作（如有限的算术、内存访问）。禁止非法跳转（如跳出程序范围）或危险操作（如直接访问任意内核内存）。

控制流分析：验证程序无不可达代码或无限循环（通过模拟执行所有路径）。限制程序复杂度（指令数、栈大小等）。

辅助函数检查：确保调用的 bpf_helper 函数与程序类型匹配（如网络程序不能调用文件操作辅助函数）。

若验证失败：返回 -EACCES 错误（如 Operation not permitted）。

（2）即时编译（JIT，可选）

解释执行：默认情况下，内核使用 eBPF 解释器逐条执行字节码（性能较低）。

JIT 编译：若启用（CONFIG_BPF_JIT=y），将字节码编译为宿主机的原生机器码：

编译后的代码直接存储在内存中，后续触发事件时直接执行机器码。

（3）创建内核对象

内核分配 struct bpf_prog 结构体，存储程序元信息（类型、指令、JIT 代码等）。返回一个文件描述符（fd）给用户态，用于后续操作（如绑定事件）。

3. 绑定到事件钩子

用户态程序通过另一个 bpf() 调用或 ioctl 将程序绑定到事件：

（1）指定事件类型

程序类型决定可绑定的事件：

  • BPF_PROG_TYPE_KPROBE → 绑定到内核函数（如 do_sys_openat2）。
  • BPF_PROG_TYPE_TRACEPOINT → 绑定到静态跟踪点（如 sys_enter_openat）。
  • BPF_PROG_TYPE_XDP → 绑定到网卡驱动接收路径。
（2）内核的挂载机制

根据事件类型，内核采用不同的挂载方式：

事件类型 挂载机制
kprobe 修改目标函数的指令，在入口插入断点（int3），触发时跳转到 eBPF 程序。
tracepoint 将 eBPF 程序注册到跟踪点的回调链表（trace_event_call.prog_array）。
XDP 网卡驱动将 eBPF 程序附加到接收队列的特定阶段（如 netif_receive_skb 之前）。

以 kprobe 为例的挂载流程：

  1. 用户态调用 bpf(BPF_PROG_ATTACH)，指定程序 fd 和函数名（如 do_sys_openat2）。
  2. 内核通过 kallsyms_lookup_name() 查找函数地址。
  3. 替换函数入口的指令为断点（int3），保存原指令。
  4. 当函数被调用时，CPU 触发断点，内核执行 eBPF 程序，然后恢复原函数执行。
（3）多程序链式调用
  • 同一事件可绑定多个 eBPF 程序，内核按加载顺序依次调用。
  • 通过 struct bpf_prog_array 管理程序链。

4. 事件触发时的执行流程

当绑定的事件发生时（如调用了 do_sys_openat2）：

  1. 中断/回调触发：
    • kprobe：CPU 执行到断点指令，陷入内核的 kprobe_handler。
    • tracepoint：内核执行到 trace_sys_enter_openat()，遍历回调链表。
  2. 执行 eBPF 程序：
    • 内核将事件上下文（如寄存器值、参数）传递给 eBPF 程序。
    • 通过解释器或 JIT 代码运行 eBPF 指令。
  3. 处理返回值：
    • 某些程序类型（如 XDP）可决定后续操作（如丢弃数据包）。
    • 数据可通过 perf_event 或 map 传回用户态。

5. 关键数据结构

  • **struct bpf_prog**：存储 eBPF 程序的所有信息（指令、JIT 代码、类型等）。
  • **struct bpf_map**：管理 eBPF 程序与用户态共享的数据。
  • **struct bpf_prog_array**：存储绑定到同一事件的多个程序。

开发环境

BCC（BPF Compiler Collection）

提供 Python、C++ 等高级语言接口，简化 eBPF 开发。内置大量现成的工具（如 execsnoop、opensnoop、tcptop）。适用于快速开发动态追踪和性能分析工具。

适用场景：

  • 快速编写 eBPF 工具，特别是用于系统监控、网络分析、安全检测等。
  • 适合 Python/C++ 开发者，不需要深入掌握 eBPF 汇编或 LLVM。

bpftrace

类似于 awk 或 DTrace 的高级脚本语言，专为 eBPF 设计。语法简洁，适合单行命令或短脚本。内置许多有用的探针（如 kprobe、uprobe、tracepoint）。

适用场景：快速调试和临时分析（如查看系统调用、函数调用）。适合运维和开发者快速排查问题，不需要编译代码。

示例：

libbpf（C 原生开发）

由 Linux 内核社区维护，是 eBPF 的官方库。相比 BCC，libbpf 更轻量，不依赖 LLVM 运行时编译。支持 BPF CO-RE（Compile Once, Run Everywhere），解决内核兼容性问题。

适用场景：需要高性能、低开销的 eBPF 程序（如网络过滤、安全监控）。适合 C 开发者，希望直接控制 eBPF 行为。

相关工具：

  • bpftool：用于管理 eBPF 程序和 maps。
  • BTF（BPF Type Format）：提供内核数据结构信息，支持 CO-RE。

eBPF Go 库（cilium/ebpf）

Go 语言实现的 eBPF 库，由 Cilium 团队维护。适用于编写用户态 eBPF 加载器（如网络策略、安全工具）。

适用场景：Go 开发者希望用 eBPF 增强应用程序（如 Kubernetes CNI、服务网格）。

Rust eBPF 生态

Rust 语言编写的 eBPF 库（如 aya-rs），提供内存安全和零成本抽象。适合安全关键型应用（如防火墙、入侵检测）。

适用场景：Rust 开发者希望用 eBPF 构建高性能、安全的应用。

Kernel 原生 eBPF（直接写 C + LLVM）

直接编写 eBPF C 代码，用 clang 编译，通过 bpftool 加载。最底层的方式，适合内核开发者。

适用场景：需要极致优化或定制 eBPF 行为（如 XDP 网络加速）。

对比总结：

工具/框架 语言支持 适用场景 学习难度 典型用途
BCC Python/C++ 快速开发工具 低 监控、安全分析
bpftrace 脚本语言 临时调试 最低 单行命令分析
libbpf C 高性能程序 中 网络、安全、CO-RE
Go (cilium/ebpf) Go 云原生应用 中 Kubernetes、Cilium
Rust (aya-rs) Rust 安全关键应用 中高 防火墙、IDS
Kernel eBPF C/LLVM 内核开发 高 XDP、内核模块
