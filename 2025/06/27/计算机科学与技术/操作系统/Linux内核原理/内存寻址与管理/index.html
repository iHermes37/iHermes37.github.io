内存寻址与管理

内存寻址

地址转化

  • 逻辑地址：程序直接使用的内存地址（如指针值）。
  • 线性地址：分段转换后的连续虚拟地址空间。
  • 物理地址：实际内存硬件中的地址。

逻辑地址转化过程

内存控制单元（MMU）的地址转换流程：

  1. 分段单元（Segmentation Unit）
    将逻辑地址（Logical Address）转换为线性地址（Linear Address）。
  2. 分页单元（Paging Unit）
    将线性地址转换为物理地址（Physical Address）。

注意在多处理器系统中，所有CPU共享同一内存，RAM芯片可被多个CPU并发访问。由于RAM读写操作必须串行执行，内存仲裁器（Memory Arbiter）作为硬件电路插入总线与每个RAM芯片之间，负责协调访问权限。

  • 空闲时：立即批准CPU的访问请求。
  • 忙碌时：延迟其他CPU的访问（当前正为另一处理器服务）。

即使单处理器也需要仲裁器，因DMA控制器与CPU可能并发操作（参见第十三章“直接内存访问(DMA)”）。而在多处理器系统中仲裁器结构更复杂，需支持多个输入端口。例如：

  • 双Pentium系统：每个CPU芯片入口配置两端口仲裁器，CPU需先交换同步信息再请求使用公用总线。

硬件中的分段

段寄存器

到了 32 位和 64 位时代，Linux 主要使用的是分页机制来进行内存管理，它提供了灵活、高效的虚拟内存。那么，为什么还要保留“过时”的段机制呢？

答案是：强制性的权限控制和模式切换。

Intel 的段机制不仅关乎寻址，更关乎权限。每个段描述符都包含一个 Descriptor Privilege Level 字段，它定义了该段所在代码的权限级别。x86 架构定义了 4 个特权级，称为 Ring：

  • Ring 0: 最高权限，内核态。
  • Ring 3: 最低权限，用户态。
  • Ring 1 和 Ring 2 很少被使用。

Linux 主要使用 Ring 0 和 Ring 3。

段寄存器的作用就是明确地告诉 CPU：“当前正在执行的代码是处于内核态还是用户态”。

一个逻辑地址由两部分组成：一个段标识符和一个指定段内相对地址的偏移量。段标识符是一个16位长的字段，称为段选择符，而偏移量是一个32位长的字段。

为了快速方便地找到段选择符，处理器提供段寄存器，段寄存器的唯一日的是存放段选择符。这些段寄存器称为cs、ss、ds、es、fs和gs。尽管只有6个段寄存器，但程序可以把同一个段寄存器用于不同的目的，方法是先将其值保存在内存中，用完后再恢复。
6个寄存器中3个有专门的用途：

  • cs: 代码段寄存器，指向包含程序指令的段。
  • ss: 栈段寄存器，指向包含当前程序栈的段。
  • ds: 数据段寄存器，指向包含静态数据或者全局数据段。

其他3个段寄存器作一般用途，可以指向任意的数据段。

它们的关系远不止“容器与内容”那么简单。实际上，段寄存器是一个复合结构，它包含两个部分：

  1. 可见部分： 程序员可以直接或间接操作的部分，里面存储的就是段选择符。
  2. 不可见部分： 对软件是透明的，也称为“影子寄存器”或“描述符缓存”。里面存储的是完整的段描述符（包含基地址、界限、权限等所有8字节信息）。

段描述符

Linux LDT

LDT（Local Descriptor Table） 是 x86 架构中与 GDT（Global Descriptor Table） 并列的段描述符表，用于定义进程私有的内存段（如代码、数据段）。每个进程可以有自己的 LDT，实现段级内存隔离（现代 Linux 基本弃用，改用分页机制）。

  • 历史用途（早期 Linux/Windows）：为进程提供独立的段描述符，隔离不同任务的地址空间。支持 分段内存管理（如 DOS 时代的实模式程序）。
  • 现代替代：被 分页机制（Paging） 取代（线性地址直接映射物理地址，无需段隔离）。仅保留少数场景（如 Wine 运行 Windows 16位程序时模拟 LDT）。

Linux 中的 LDT 实现

  • 默认禁用：大多数进程不启用 LDT，直接使用 GDT 的平坦模型（段基址=0，段界限=最大）。
  • 特殊情况：
    • modify_ldt() 系统调用：允许进程自定义 LDT（需 CONFIG_X86_32 支持）。
    • Wine/虚拟机：模拟旧架构时可能动态创建 LDT。

在单处理器系统中只有一个GDT，而在多处理器系统中每个CPU对应一个GDT。

段选择符 -> GDT/LDT -> 段描述符 -> 线性地址

  1. 段描述符： 这是基础。它是一个8字节的数据结构，存储在内存中的一张表里。它真正地描述了一个内存段，包含了以下关键信息：

    • 段的基地址： 这个段在内存中的起始地址。
    • 段的界限： 这个段的大小/边界。
    • 访问权限： 这个段是否可读、可写、可执行。
    • 特权级： 描述符特权级，即这个段属于哪个Ring（0-3）。

  2. 全局描述符表： 这是一个由操作系统维护的数组，里面存放了所有的段描述符。CPU中有一个特殊的寄存器叫做GDTR，它存储了GDT在内存中的起始地址和大小。

    每个段由一个8字节的段描述符表示，它描述了段的特征。段描述符放在全局描述符表（Global Descriptor Table—GDT）或局部描述符表（LocalDescriptor Table—LDT）中。通常只定义一个GDT，而每个进程除了存放在GDT中的段之外如果还需要创建附加的，就可以有自己的LDT。

    GDT在主存中的地址和大小存放在gdtr控制寄存器中，当前正被使用的LDT地址和大小放在ldtr控制寄有器中。

段选择符

现在主角登场了。它是一个16位（2字节）的值，存储在段寄存器（CS, DS, SS, ES, FS, GS）中。

Linux分段

分段单元

分段单元是 CPU 内部的一个硬件部件，它的职责是执行“分段内存管理模型”，将逻辑地址转换为线性地址。

在现代操作系统中，分段的作用已经被大大弱化，转而主要使用分页机制来进行灵活的内存管理和隔离。但为了兼容x86架构，分段单元依然存在且必须工作。

Linux 采用了一种叫做 “扁平模型” 的策略：

  • 它定义了非常少的几个段（内核代码段、内核数据段、用户代码段、用户数据段）。
  • 将所有段的基地址都设置为 0，界限都设置为 4GB。
  • 这样一来，逻辑地址直接等于线性地址（因为 基地址0 + 偏移量 = 偏移量本身）。

它的核心作用从内存划分转变为了权限控制和模式切换。

  • 当 CS 段寄存器加载的是 __USER_CS 时，CPU处于用户态。

  • 当发生系统调用，CS 被硬件自动切换为 __KERNEL_CS 时，CPU进入了内核态。

8086微处理器中的分段鼓励程序员把他们的程序化分成逻辑上相关的实体，例如子程序或者全局与局部数据区。然而，Linux以非常有限的方式使用分段。实际上，分段和分页在某种程度上有点多余，因为它们都可以划分进程的物理地址空间：分段可以给每一个进程分配不同的线性地址空间，而分页可以把同一线性地址空间映射到不同的物理空间。与分段相比，Linux更喜欢使用分页方式，因为：当所有进程使用相同的段寄存器值时，内存管理变得更简单，也就是说它们能共享同样的一组线性地址。

Linux设计目标之一是可以把它移植到绝大多数流行的处理器平台上。然而，RISC体系结构对分段的支持很有限。2.6版的Linux只有在80x86结构下才需要使用分段。

运行在用户态的所有Linux进程都使用一对相同的段来对指令和数据寻址。这两个段就是所谓的用户代码段和用户数据段。类似地，运行在内核态的所有Linux进程都使用一对相同的段对指令和数据寻址：它们分别叫做内核代码段和内核数据段。

相应的段选择符由宏_USER CS、_USER_DS、_KERNEL_CS和 KERNEL_DS分别
定义。例如，为了对内核代码段寻址，内核只需要把KERNEL_CS宏产生的值装进cs
段寄存器即可。

注意，与段相关的线性地址从0开始，达到22-1的寻址限长。这就意味着在用户态或内核态下的所有进程可以使用相同的逻辑地址。所有段都从0×00000000开始，这可以得出另一个重要结论，那就是在Linux下逻辑地址与线性地址是一致的，即逻辑地址的偏移量字段的值与相应的线性地址的值总是一致的。

如前所述，CPU的当前特权级（CPL）反映了进程是在用户态还是内核态，并由存放在cs寄存器中的段选择符的RPL字段指定。只要当前特权级被改变，一些段寄存器必须相应地更新。例如，当CPL=3时（用户态），ds寄存器必须含有用户数据段的段选择符，而当CPL=0时，ds寄存器必须含有内核数据段的段选择符。

当对指向指令或者数据结构的指针进行保存时，内核根本不需要为其设置逻辑地址的段选择符，因为CS寄存器就含有当前的段选择符。例如，当内核调用一个函数时，它执行一条ca11汇编语言指令，该指令仅指定其逻辑地址的偏移量部分，而段选择符不用设置，它已经隐含在cs寄存器中了。因为“在内核态执行”的段只有一种，叫做代码段，由宏__KERNEL_CS定义，所以只要当CPU切换到内核态时将KERNEL_CS装载进cs就足够了。同样的道理也适用于指向内核数据结构的指针（隐含地使用ds寄存器）以及指向用户数据结构的指针（内核显式地使用es寄存器）

硬件中的分页

特性 分段机制 分页机制
控制对象 当前执行代码的权限 (CPL) 单个内存页的访问权限
控制粒度 粗粒度（整个执行模式） 细粒度（4KB 的页）
检查时机 模式切换时（如执行系统调用） 每次内存访问时
核心作用 定义“你是谁”（用户态还是内核态） 定义“你能对这个页做什么”（读、写、执行）
如何绕过 原则上不可绕过（除漏洞外） 理论上可通过修改页表绕过（但内核保护页表）
  1. 分段提供了不可绕过的执行上下文隔离：它从硬件层面确保了用户态代码无法随意进入内核态。除非通过少数几个由内核预设的、安全的“门”（如系统调用入口），否则用户程序永远无法获得内核特权。这是模式切换的基石。
  2. 分页提供了灵活细粒度的内存保护：它在已进入的执行上下文内，对每一页内存进行精细的读写执行控制。这既保护了用户进程彼此之间，也保护了内核免受自身错误代码的侵害。

**将它们结合起来，就构成了一个既坚固又灵活的内存保护和权限控制系统： **分段单元确立了“战斗的层面”（用户 vs 内核），而分页单元则管理着该层面内的“每一场遭遇战”（对具体内存的访问）。

分页单元把线性地址转换成物理地址。其中的一个关键任务是把所请求的访问类型与线性地址的访问权限相比较，如果这次内存访问是无效的，就产生一个缺页异常。

为了效率起见，线性地址被分成以固定长度为单位的组，称为页（page）。页内部连续的线性地址被映射到连续的物理地址中。这样，内核可以指定一个页的物理地址和其存取权限，而不用指定页所包含的全部线性地址的存取权限。我们遵循通常习惯，使用术语“页”既指一组线性地址，又指包含在这组地址中的数据。

分页单元把所有的RAM分成固定长度的页框（有时叫做物理页）。每一个页框包含一个页（page），也就是说一个页框的长度与一个页的长度一致。页框是主存的一部分，因此也是一个存储区域。区分一页和一个页框是很重要的，前者只是一个数据块，可以存放在任何页框或磁盘中。

把线性地址映射到物理地址的数据结构称为页表。页表存放在主存中，并在启用分单元之前必须由内核对表进行适当的初始化。从80386开始，所有的80x86处理器都支持分页，它通过设置C0寄存器的PG标志启用。当PG=0时，线性地址就被解释成物理地址。

常规分页

线性地址的转换分两步完成，每一步都基于一种转换表，

  • 第一种转换表称为页目录表。
  • 第二种转换表称为页表。

使用这种二级模式的目的在于减少每个进程页表所需RAM的数量。如果使用简单的1级页表，那将需要高达2^20个表项（也就是，在每项4个字节时，需要4MB RAM）来表示每个进程的页表（如果进程使用全部4GB线性地址空间），即使一个进程并不使用那个范围内的所有地址。二级模式通过只为进程实际使用的那些虚拟内存区请求页表来减少内存容量。

每个活动进程必须有一个分配给它的页目录。不过，没有必要马上为进程的所有页表都分配RAM。只有在进程实际需要一个页表时才给该页表分配RAM会更为有效率。

正在使用的页目录的物理地址存放在控制寄存器cr3中。线性地址内的Directory字段决定页目录中的目录项，而录项指向适当的页表。地址的Table字段依次又决定页表中的表项，而表项含有页所在页框的物理地址。Offset字段决定页框内的相对位置。由于它是12位长，故每一页含有4096字节的数据。

扩展分页

从Pentium模型开始，80x86微处理器引人了扩展分页，它允许页框大小为4MB而不是4KB。扩展分页用于把大段连续的线性地址转换成相的物理地址，在这些情况下，内核可以不用中间页表进行地址转换，从而节省内存并保留TLB项

硬件保护方案

分页单元和分段单元的保护方案不同。尽管80x86处理器允许一个段使用4种可能的特级别，但与页和页表相关的特权级只有两个，因为特权由前面 “常规分页”一节中所提到的User/Supervisor标志所控制。若这个标志为0，只有当CPL小于3（这意味着对于Linux而言，处理器处于内核态）时才能对页寻址，若该标志为，则总能对页寻址。

此外，与段的3种存取权限（读，写，执行）不同的是，页的存取权限只有两种（读，写）。如果页目录项或页表项的Read/write标志等于0，说明相应的页表或页是只读的，否则是可读写的。

硬件高速缓存

硬件高速缓存是计算机系统中一种位于CPU和主内存之间的小容量、高速度的存储部件，主要用于缓解CPU与主内存之间的速度差异矛盾。现代CPU的处理速度远高于主内存的访问速度，如果没有高速缓存，CPU将花费大量时间等待数据从内存中读取，形成性能瓶颈。硬件高速缓存采用SRAM静态随机存储器制造，相比主内存使用的DRAM动态随机存储器，虽然成本更高、容量更小，但访问速度要快得多。

其工作原理基于局部性原理，包括时间局部性（最近访问的数据很可能再次被访问）和空间局部性（访问某个数据时，其相邻数据也很可能被访问）。当CPU需要读取数据时，首先在高速缓存中查找，如果找到（缓存命中）则直接返回数据；如果未找到（缓存缺失），则从主内存读取数据，并按照特定策略将数据及其相邻数据存入缓存。

现代多级缓存体系通常包含L1、L2、L3三级缓存，L1缓存速度最快但容量最小，通常分为指令缓存和数据缓存；L2缓存容量较大但速度稍慢；L3缓存为多个CPU核心共享，容量最大但速度最慢。

硬件高速缓存通过减少CPU等待时间显著提升了系统性能，其管理完全由硬件自动完成，对软件透明。缓存的设计考虑了映射策略（直接映射、组相联映射、全相联映射）、替换算法（LRU、FIFO等）和写策略（直写、回写）等多个方面，以在命中率和实现复杂度之间取得平衡。

转换后的缓冲器(TLB)

转换后援缓冲器（TLB） 是计算机内存管理单元（MMU）中的一个专用高速缓存，用于加速虚拟地址到物理地址的转换过程。在现代计算机体系结构中，进程使用的是虚拟地址空间，而实际访问内存时需要将虚拟地址转换为物理地址，这个转换过程通常通过查询页表来完成。然而，页表存储在主内存中，每次地址转换都需要访问内存，这会带来显著的性能开销。TLB通过缓存最近使用过的虚拟页到物理页框的映射关系，避免了每次地址转换都访问主内存的需求，从而大幅提高了系统性能。

TLB的工作原理类似于CPU缓存，同样基于局部性原理。当CPU发出虚拟地址时，MMU首先在TLB中查找对应的虚拟页号。如果找到（称为TLB命中），则立即获得物理页框号，与页内偏移组合成物理地址；如果未找到（称为TLB缺失），则需要访问主内存中的页表完成地址转换，并将新的映射关系存入TLB。TLB的典型容量相对较小，通常只有几十到几百个条目，但由于程序访问的空间局部性，其命中率通常可达90%以上。

现代处理器通常采用多级TLB结构，如Intel Skylake架构具有64条目的L1指令TLB、128条目的L1数据TLB，以及1536条目的统一L2 TLB。TLB的管理涉及替换策略（如LRU、随机替换）、一致性维护（当页表更新时需要无效化或更新TLB中对应条目）等多方面考虑。在多处理器系统中，TLB一致性更为复杂，需要通过IPI（处理器间中断）来保证所有CPU的TLB状态同步。TLB作为内存访问的关键路径组件，其性能直接影响整个系统的内存访问效率。

Linux分页

Linux采用了一种同时适用于32位和64位系统的普通分页模型。正像前面“64位系统中的分页”一节所解释的那样，两级页表对32位系统来说已经足够了，但64位系统需要更多数量的分页级别。直到2.6.10版本，Linux采用三级分页的模型。从2.6.11版本开始，采用了四级分页模型。下图中展示的4种页表分别被为：

  • 页全局目录（Page GlobalDirectory）
  • 页上级目录（PageUpperDirectory）
  • 页中间目录（PageMiddleDirectory）
  • 页表（PageTable）

页全局目录包含若干页上级目录的地址，页上级目录又依次包含若干页中间目录的地址，而页中间目录又包含若干页表的地址。每一个页表项指向一个页框。线性地址因此被分成五个部分。下图没有显示位数，因为每一部分的大小与具体的计算机体系结构有关。

对于没有启用物理地址扩展的32位系统，两级页表已经足够了。Linux通过使“页上级目录”位和“页中间目录”位全为0，从根本上取消了页上级目录和页中间目录字段。不过，页上级目录和页中间目录在指针序列中的位置被保留，以便同样的代码在32位系统和64位系统下都能使用。内核为页上级目录和页中间目录保留了一个位置，这是通过把它们的页目录项数设置为1.并把这两个目录项映射到页全局目录的一个适当的目录项而实现的。

启用了物理地址扩展的32位系统使用了三级页表。Linux的页全局目录对应80x86的页目录指针表（PDPT），取消了页上级目录，页中间目录对应80x86的页目录，Linux的页表对应80x86的页表。最后，64位系统使用三级还是四级分页取决于硬件对线性地址的位的划分。Linux的进程处理很大程度上依赖于分页。事实上，线性地址到物理地址的自动转换使下面的设计目标变得可行：

  • 给每一个进程分配一块不同的物理地址空间，这确保了可以有效地防止寻址错误。
  • 区别页（即一组数据）和页框（即主存中的物理地址）之不同。这就允许存放在某个页框中的一个页，然后保存到磁盘上，以后重新装入这同一页时又可以被装在不同的页框中。这就是虚拟内存机制的基本要素。

页框管理

Linux采用4KB页框大小作为标准的内存分配单元。

页描述符

内核必须记录每个页框当前的状态。例如，内核必须能区分哪些页框包含的是属于进程的页，而哪些页框包含的是内核代码或内核数据。类似地，内核还必须能够确定动态内存中的页框是否空闲。如果动态内存中的页框不包含有用的数据，那么这个页框就是空闲的。在以下情况下页框是不空闲的：包含用户态进程的数据、某个软件高速缓存的数据、动态分配的内核数据结构、设备驱动程序缓冲的数据、内核模块的代码等等。

在 Linux 内核中，页描述符是由 struct page 结构体表示的数据结构，用于跟踪和管理物理内存中的每一个页帧（page frame）。每个物理页帧在内核中都有一个对应的 struct page 结构体实例，这些实例被组织为一个称为 mem_map 的数组。页描述符的关键作用：

  • 记录页帧的物理状态（是否空闲、是否被锁定等）
  • 跟踪页帧的使用情况（被哪个进程或内核子系统使用）
  • 管理页帧的引用计数（防止过早释放）
  • 支持内存回收机制（如页缓存和交换系统）

Linux 内核为系统中的每一个物理页帧都维护了一个对应的 struct page 描述符。这些描述符在系统启动时初始化，并存储在 mem_map 数组中，通过物理页帧号（PFN）可以直接索引到对应的 struct page。

非一致内存访问

在Linux 2.6内核中，非一致内存访问（Non-Uniform Memory Access, NUMA）是一种针对多处理器系统的内存架构设计，其中处理器的内存访问时间取决于内存相对于处理器的物理位置。在NUMA系统中，每个处理器或处理器组拥有本地内存，访问本地内存速度较快，而访问其他处理器的远程内存则速度较慢，这种差异形成了内存访问的非一致性。

Linux 2.6内核通过引入NUMA感知的内存管理机制来优化这种架构的性能，包括节点（node）的概念，将物理内存划分为多个节点，每个节点关联一个或多个处理器，并维护本地内存池。

内核的调度器、内存分配器和页缓存机制均被增强以优先使用本地内存，减少跨节点访问带来的延迟。

例如，当进程在某个节点上运行时，内核会优先从该节点的空闲列表中分配内存，从而避免远程内存访问的开销。此外，Linux 2.6还提供了NUMA相关的API和策略配置选项，允许系统管理员或应用程序开发者根据负载特性调整内存分配策略，以进一步优化性能。这种设计使得Linux能够高效地利用NUMA架构的资源，提升多处理器系统整体的吞吐量和响应速度。

内存管理区

在 Linux 内核的内存管理体系中，内存管理区（Zone）是应对硬件限制而设计的关键概念。虽然理想情况下所有内存页框（page frame）应该可以平等地用于任何目的，但实际的计算机体系结构（特别是 x86 架构）存在硬件约束，迫使 Linux 采用分区管理策略。

Linux 内核需要处理 x86 架构的两个主要硬件限制：

  1. DMA 限制：ISA 总线的 DMA 控制器只能访问物理内存的前 16MB

    • 历史硬件设计：早期ISA总线的DMA控制器（如Intel 8237）使用24位地址线，其寻址能力为 224=16MB224=16MB。

    • 兼容性保留：虽然现代设备已支持更高位宽（如PCI总线支持32/64位DMA），但为兼容老式ISA设备（如声卡、网卡），内核仍需保留此区域。

    • 硬件约束：DMA操作由设备直接访问内存，绕过CPU，因此设备自身的寻址能力决定了可访问的内存范围。

  2. 线性地址空间限制：32 位系统 CPU 不能直接访问所有物理内存（地址空间限制为 4GB，内核通常只能直接映射 1GB）

    虚拟地址空间划分：32位系统的虚拟地址空间为4GB（232232），Linux默认采用 3:1分割：

    • 用户空间：0-3GB（供进程使用）

    • 内核空间：3GB-4GB（共1GB，用于映射内核代码、数据结构和物理内存）

    • 直接映射需求：内核需要快速访问物理内存，因此将ZONE_NORMAL（16MB-896MB）线性映射到内核空间的3GB~3GB+896MB区域。

      • 物理地址 P → 虚拟地址 P + 3GB，通过简单偏移即可转换。

    高端内存问题：若物理内存 >896MB，超出部分（ZONE_HIGHMEM）无法全部线性映射到剩余的1GB内核空间（还需容纳内核代码、页表等）。访问ZONE_HIGHMEM需动态调用kmap()/kmap_atomic()临时映射到内核的小型”窗口”中。

为应对这些限制，Linux 2.6 将物理内存划分为三个管理区：

  1. ZONE_DMA32：在64位系统中，部分设备可能仍需要32位地址限制的DMA，因此扩展出支持4GB以下的ZONE_DMA32。
  2. 现代变化：
    • 64位系统中，ZONE_HIGHMEM被废弃（因虚拟地址空间足够大）。
    • NUMA架构下，每个节点（Node）会进一步划分上述区域，以优化本地内存访问。
  3. 水线管理：每个Zone维护min/low/high三个水线值，用于触发内存回收（如kswapd）。

页框池

在Linux内核中，页框池是为满足特定内存分配需求而预先保留的一组物理内存页框的集合，主要用于处理紧急内存请求或保证关键操作（如DMA、中断处理等）能够快速获得可用的物理内存，避免因内存碎片或临时短缺导致系统性能下降或功能异常。

内核通过维护不同类型的页框池（如紧急内存池、DMA内存池等）来优化内存分配效率，确保即使在内存压力较大时，关键子系统（如网络栈、存储驱动）仍能可靠地获取内存资源。

页框池的实现通常基于伙伴系统（Buddy System），通过预先从ZONE_DMA或ZONE_NORMAL等内存管理区分配并保留一定数量的连续页框，后续需要时直接从池中分配，省去了常规内存分配的搜索和碎片整理开销。

例如，当设备驱动需要DMA缓冲区时，可直接从DMA页框池中获取符合硬件限制的内存页，而无需遍历复杂的空闲列表。页框池的大小和策略可通过内核参数动态调整，以平衡内存利用率和系统可靠性。这种机制显著提高了Linux内核在高负载或低内存条件下的稳定性，尤其适用于实时系统和嵌入式环境。

分区页框分配器

分区页框分配器（Zoned Page Frame Allocator）是 Linux 内核中负责处理连续物理页框分配请求的核心子系统。该分配器通过分层设计高效管理系统物理内存，满足不同场景的内存分配需求。

当内核需要分配连续物理页框时：

  1. 管理区分配器首先确定目标内存区（考虑DMA标志、GFP掩码等）
  2. 尝试从请求CPU的本地页框高速缓存获取（针对单页请求）
  3. 若失败则调用对应内存区的伙伴系统：
    • 伙伴系统搜索满足大小的最小空闲块
    • 若找到则分割分配，剩余部分加入空闲链表
    • 分配失败可能触发内存回收后再尝试
  4. 释放内存时逆向执行上述过程，伙伴系统会尝试合并相邻空闲块

高端内存页框的内核映射

在 32 位体系结构（如 x86）中，内核虚拟地址空间有限（通常为 1GB或 3GB/1GB 拆分），无法直接线性映射所有的物理内存。高于 896MB（此值因配置而异）的物理内存被称为高端内存。为了访问这些页面，内核需要动态地为其建立映射。永久内核映射和临时内核映射就是为解决此问题而生的两种主要机制。

为了更好地理解，可以做一个简单的类比：

  • 永久内核映射（kmap） 就像在图书馆长期租用了一个固定座位。你需要登记，可能需要等待空位，但一旦拥有，就可以长时间使用，非常适合需要稳定环境的工作。但你不能在“不允许排队等待”（中断上下文）的时候做这件事。
  • 临时内核映射（kmap_atomic） 就像使用图书馆的公共速查桌。桌子数量有限，但你先到先得，无需等待，用完即走，把位置让给下一个人。它速度快、灵活，非常适合快速查阅资料，即使在时间紧迫（中断上下文）时也能使用。但你不能把书长期放在那里。

提示：在 64 位体系结构上，由于内核虚拟地址空间极其巨大（例如 x86_64 有 128TB 或 64TB 的内核空间），可以轻松地对所有物理内存进行线性映射，因此通常不再需要这两种映射机制。kmap() 和 kmap_atomic() 在 64 位系统上可能只是空操作或简单的页表转换，直接返回页面对应的线性映射地址。这两种机制主要是为了解决 32 位系统的地址空间限制问题。

伙伴系统算法

每CPU页框高速缓存

每CPU页框高速缓存 是 Linux 内核中一种用于高效分配和释放单个页框（即 4KB 物理内存页）的机制。它的核心思想是通过为每个 CPU 核心维护一个本地的小型页框池，来减少在频繁分配/释放单个页面时的全局锁竞争，从而极大地提升内存分配性能，特别是在多处理器（SMP）系统中。

设计原理与目的
  • 减少锁竞争：在单一全局页框缓存中，多个 CPU 同时分配或释放页面时需要竞争同一个锁，这会成为严重的性能瓶颈。每CPU缓存将全局资源“本地化”，使得每个 CPU 在大部分时间都可以无锁地访问自己的缓存。
  • 利用CPU本地性：CPU 访问其本地缓存数据的速度更快，因为这通常意味着更好的硬件缓存命中率。
  • 批处理操作：当本地缓存为空或已满时，内核会与全局的伙伴系统进行一次“批量”交换，而不是每次分配都去打扰伙伴系统，这降低了操作的整体开销。
两种类型的每CPU页框高速缓存

Linux 内核为其两种最基本的内存分配需求（适用于普通页和适用于DMA操作的页）分别维护了每CPU缓存：

  1. 每CPU页面分配器缓存：服务于 GFP_USER、GFP_KERNEL 等普通分配标志的请求。它缓存的是普通内存域（ZONE_NORMAL 和 ZONE_HIGHMEM） 的页框。
  2. 每CPU DMA页框缓存：服务于 GFP_DMA 分配标志的请求。它专门缓存DMA内存域（ZONE_DMA） 的页框，用于需要直接内存访问的设备。

非连续内存区管理

非连续内存区管理 是 Linux 内核提供的一种机制，允许内核分配一组在虚拟地址空间上连续，但在物理内存中不一定连续的内存页框。它通过页表映射，将多个离散的物理页框“拼凑”成一个大的、虚拟上连续的内存区域。

这类似于用户空间的 malloc，但它服务于内核的分配需求。其核心目的是为了满足内核中需要大块、连续虚拟地址空间，但又无法或难以通过伙伴系统获取连续物理内存的情况（尤其是系统运行较长时间、内存碎片化严重后）。

为什么需要它？

  • 物理内存碎片化：系统长时间运行后，尽管有大量空闲物理内存，但可能都是小块的碎片，无法满足大块的连续物理内存分配请求。伙伴系统可能会失败。
  • 大内存需求：某些内核子系统（如模块加载、I/O 缓冲区、网络栈等）可能需要几十 KB 甚至几 MB 的连续虚拟内存，分配连续的物理内存代价高昂或不必要。
  • 灵活性：它提供了一种比伙伴系统更灵活的内存分配方式，特别适合那些不关心底层物理地址是否连续，只关心虚拟地址连续的用例。

非连续内存区的结构体

内核使用 struct vm_struct 的一个链表来管理每一个已分配的非连续内存区。

  • pages 数组是这个机制的关键。它保存了所有被映射的物理页框的 struct page* 指针。这些物理页框是通过多次调用伙伴系统的 alloc_page 分别分配得来的，因此它们在物理上很可能是离散的。
  • addr 是该区域在内核虚拟地址空间中的起始地址。
  • size 包含了实际数据大小以及为了满足页对齐而引入的“缝隙”

分配与释放过程

分配过程（vmalloc）：

  1. 参数检查与大小计算：vmalloc(size) 首先检查请求大小，然后计算需要多少个页框，并加上为页表和对齐可能需要的额外开销。
  2. 分配 vm_struct 描述符：在 vmalloc 区域中寻找一个足够大的虚拟地址空间间隙。
  3. 分配物理页框：通过伙伴系统逐页分配 (alloc_page) 所需的物理页框，并将每个页框的 struct page 指针存入 pages 数组。
  4. 修改页表：内核修改内核页表，为步骤1中保留的虚拟地址范围建立页表条目（PTE），将这些条目映射到步骤3中分配的离散物理页框上。
  5. 刷新TLB：由于修改了页表，需要刷新对应地址范围的 TLB（Translation Lookaside Buffer），以确保所有CPU看到一致的映射。

释放过程（vfree）：

  1. **找到 vm_struct**：根据传入的虚拟地址，在 vm_struct 链表中找到对应的描述符。
  2. 清除页表映射：清除内核页表中对应虚拟地址范围的页表条目。
  3. 释放物理页框：遍历 pages 数组，对其中每个非空的 struct page*，调用 __free_page 将物理页框返还给伙伴系统。
  4. 释放描述符和数组：释放 vm_struct 描述符和 pages 数组本身占用的内存。

回收页框

Linux 内存管理(9)：回收页框

Linux 教程 | 内存管理篇之页框管理 | Aiden Blog

回收页框 是 Linux 内核内存管理子系统中最核心、最复杂的任务之一。它的目的是当系统空闲内存不足时，通过释放某些当前正在使用的物理页框，并将其返还给伙伴系统，以满足新的内存分配请求。这个过程由内核自动触发，对应用程序基本透明。

内核主要从两个地方回收页框：

回收类型 目标 机制与特点
页面缓存 磁盘文件数据在内存中的副本。 这是最快、成本最低的回收源。如果缓存中的数据是”干净”的（与磁盘一致），直接丢弃即可，无需任何I/O操作。如果是”脏”的（被修改过），则需要先写回磁盘才能释放。
匿名内存 进程的堆、栈、共享内存等不与文件直接关联的内存。 这是成本高昂的回收源。这些页框没有对应的磁盘文件作为备份，因此内核需要通过交换机制，将其内容写入到专用的交换区（Swap Area）后，才能释放该页框。

触发回收的时机

  1. 直接回收：当进程在分配内存时（通过 alloc_pages），如果伙伴系统无法满足请求，并且通过 kswapd 回收后仍然不足，分配进程本身会尝试直接内存回收。这可能会阻塞当前进程。
  2. 周期性回收：由内核线程 kswapd 负责。它定期检查每个内存节点的空闲内存水平。当空闲内存低于某个低水位线 时，kswapd 被唤醒并开始异步回收页框，直到空闲内存达到高水位线。这种方式不会阻塞应用程序。
  3. 内存严重不足回收：当内存压力极大，连直接回收都无法满足请求时，会触发 OOM Killer。它会选择一个”最不重要”的进程并将其杀死，从而一次性释放大量内存。这是最后的手段。
