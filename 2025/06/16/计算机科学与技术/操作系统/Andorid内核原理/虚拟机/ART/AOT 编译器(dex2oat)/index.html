AOT 编译器(dex2oat)

dex2oat运行原理

dex2oat.cc - OpenGrok cross reference for /art/dex2oat/dex2oat.cc

在 Android 系统中，dex2oat 的行为和应用运行方式会根据系统版本、设备配置和应用特性有所不同。

应用安装时的编译行为

当应用安装时，dex2oat 不会总是将全部 DEX 文件完全编译为本地机器码（OAT 文件）。具体行为取决于以下因素：

(1) 编译器过滤器

Android 通过 CompilerFilter 决定编译的优化程度，常见的模式包括：

  • **speed/speed-profile**：
    尽可能编译所有方法为机器码（OAT），适用于高性能需求或已收集 Profile 数据的应用。
  • verify/quicken：
    仅验证 DEX 文件或做基础优化，    不生成完整机器码    ，运行时仍需解释执行或 JIT 编译。
  • **extract**：
    仅解压 DEX 文件到 VDEX（验证过的 DEX），完全不编译，完全依赖运行时解释或 JIT。

(2) 系统版本与策略

  • Android 5.0~7.0（ART 初期）：
    安装时默认全量编译（speed），生成完整的 OAT 文件，但导致安装时间较长。
  • Android 7.0+（引入 Profile-Guided Optimization）：
    安装时可能仅做轻量编译（如 verify 或 quicken），后续通过 后台优化 或 运行时 JIT 收集热点 再优化。
  • **Android 10+**：
    进一步细化策略，例如对大型应用默认用 speed-profile，结合云端下发的 Profile 数据优化关键代码。

运行时使用 OAT 还是 DEX？

  • 有 OAT → 直接绑定本地机器码入口；
  • 无 OAT → 绑定解释器/JIT 桥函数，由运行时动态生成或执行。

应用运行时是否会使用 OAT 文件，取决于以下条件：

场景 使用的代码 原因
OAT 文件存在且有效 优先执行 OAT 机器码 直接运行本地代码效率最高
OAT 缺失或无效（如版本不匹配） 回退到 VDEX/DEX + 解释器/JIT 兼容性保障
未编译的方法（如冷门代码） DEX 字节码 + 解释器/JIT 安装时未编译或按需编译

关键机制

  • 解释器： 直接执行 DEX 字节码，无需编译，但速度慢。
  • JIT（Just-In-Time）： 运行时动态编译热点方法，结果缓存到内存（不持久化）。
  • AOT（Ahead-Of-Time）： dex2oat 生成的 OAT 文件是 AOT 编译结果，持久化存储，启动时加载。

优化流程的演进

Android 通过以下方式平衡安装速度与运行性能：

  1. 安装时快速验证： 确保 DEX 文件合法（生成 VDEX），但可能不编译。
  2. 后台优化： 在设备空闲时，根据 Profile 数据重新编译热点代码（如 cmd package compile -f -m speed-profile <pkg>）。

OAT 文件格式

OAT（OAT 是 “OAT“ 的递归缩写，代表 “OAT AOT”）是 dex2oat 编译生成的二进制文件，包含从 DEX 字节码转换的本地机器码。

![](https://skytoby.github.io/2019/Android dex，odex，oat，vdex，art文件结构/oat.png)

文件结构

部分 说明
OAT Header 元数据：编译器版本、目标架构、DEX 文件校验和等
DEX 文件副本 原始 DEX 的拷贝（可选，用于回退）
.text 段 编译后的机器码（核心部分）
.rodata 段 只读数据（字符串、类信息等）
.bss 段 未初始化静态数据
OAT Class 表 类和方法的结构化索引，指向对应的机器码

关键特性

  • 跨平台兼容性：不同 CPU 架构（ARM/x86）生成不同的 OAT 文件。
  • 依赖 Boot Image：OAT 文件需要与系统的 boot.art（核心类预编译镜像）匹配。
  • 回退机制：若 OAT 无效，ART 会回退到解释执行原始 DEX。

ART 文件格式

ART 文件（如 boot.art/app.art）是 Android Runtime 的 预编译镜像，包含核心类或应用类的预加载状态。

.art是一些类/filed/方法，app启动直接map到内存，从odex中拆分出来的,art文件主要为了加快应用的对“热代码”的加载与缓存

文件类型

文件类型 说明
boot.art 系统核心类预编译镜像（如 java.lang.Object）
boot.oat 关联的机器码（与 boot.art 配对）
app.art 部分应用的预编译类镜像（需满足条件）

文件结构

部分 说明
Image Header 魔法数、版本、内存布局信息
Class 表 预加载的类对象（含静态字段值）
ArtMethod 数组 方法元数据（链接到 OAT 中的机器码）
ArtField 数组 字段元数据
Interned 字符串池 去重后的字符串常量

关键特性

  • 内存映射直接加载：启动时直接 mmap 到内存，减少类加载开销。
  • 跨进程共享：boot.art 被所有应用进程共享。
  • 与 OAT 协同工作：ART 文件存储类结构，OAT 文件存储执行代码。

OAT 与 ART 的关系

对比项 OAT 文件 ART 文件
内容 本地机器码 + DEX 副本 预加载的类对象内存快照
生成时机 应用安装/更新时 系统启动（boot.art）或应用首次运行（app.art）
优化目标 加速代码执行 加速类加载和初始化
依赖关系 依赖 ART 文件中的类结构 依赖 OAT 文件中的方法实现

DEX → OAT → ART

层次 文件/结构 作用 所属阶段
DEX (.dex) Java 类与方法的字节码文件（Dalvik EXecutable） 存储类定义、方法签名、访问标志、字节码 编译前端（Java → DEX）
OAT (.oat) 由 dex2oat 编译生成的本地机器码封装文件 包含 DEX 索引、优化信息、JIT/AOT 编译结果 部署阶段（安装时）
ART (.art) 运行时堆镜像（Image Space）文件 包含类加载后的运行时对象（ArtMethod、ArtField、mirror::Class 等） 运行时（启动时加载）

其中：

  • **CompileImage()**：用于编译 boot 镜像或 app 镜像，生成 .art 文件
  • **CompileApp()**：用于编译普通应用，不生成 .art 文件

它们的主要工作流程相似：

  1. 加载和验证 DEX 文件
  2. 创建 OatWriter 对象
  3. 执行编译（将 DEX 转换为 OAT）
  4. 生成 VDEX 文件（包含验证过的 DEX）
  5. 写入输出文件

区别在于 CompileImage() 会额外处理镜像特有的内容，如：

  • 生成映像文件
  • 处理引导类路径
  • 处理特殊的映像选项

ParseArgs

解析命令行参数，决定编译模式（如是否编译为镜像）、输入输出文件等,具体来说：这个函数负责解析命令行参数，决定：

  • 输入 DEX 文件位置
  • 输出 OAT 和 VDEX 文件位置
  • 编译模式（如 speed、balanced、space 等优化策略）
  • 是否生成镜像文件
  • 目标架构
  • 线程数等

OpenFile

输出文件准备：创建并打开最终的输出文件（通常是 .oat 文件）,确保输出目录存在且可写

  • 在某些模式下会创建临时文件，最后再重命名为正式文件

输入文件验证：检查输入的 DEX/APK 文件是否存在且可读并验证文件路径的有效性

文件系统状态检查：确保有足够的磁盘空间用于编译输出。检查文件权限是否适当

Setup

环境准备函数，主要工作：

  1. 创建并初始化编译器运行时
  2. 设置类加载器
  3. 准备 dex 文件验证器
  4. 初始化各种编译选项
  5. 准备线程池
