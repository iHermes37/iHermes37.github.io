类的加载、链接和初始化

虚拟机首先要从文件里加载（Load）这个class的相关信息。所谓的加载，其实就是把这个类的信息从class文件或其他格式的文件（比如dalvik中的dex文件、art中的oat文件）里提取到虚拟机内部。

目标类加载后的下一步工作就是链接。具体来说，Link工作还可细分为校验（Verify，用于校验类的相关信息是否合法，比如类的格式是否正确，类中方法的字节码是否合法等）、准备（Prepare，是指为这个类和相关信息的存储分配一块存储空间）和解析（Resolve，如果该类的成员中如果有引用其他类的话，可能还需要把其他类也加载到虚拟机）三个小任务。

链接成功后，虚拟机需要初始化这个类，也就是初始化静态成员变量的值，执行static语句块等。

类的加载(Java ClassLoader)

Java 层的 ClassLoader（如 PathClassLoader、DexClassLoader）是面向开发者的抽象，负责 定位和读取类文件（如 .dex 或 .jar 文件），并通过 defineClass 方法将字节码转换为 JVM 可识别的 Class 对象。

工作流程：

  1. 查找类文件：根据类名在指定的路径（如 APK 内的 classes.dex）中查找。
  2. 读取字节码：将文件内容加载到内存。
  3. 调用虚拟机：通过 ClassLoader.defineClass 最终调用到虚拟机的底层加载逻辑（如 ART 的 ClassLinker）。

特点：

遵循 双亲委派模型（Parent Delegation Model）。仅负责 类的查找和初步加载，不涉及真正的类解析、链接和初始化。

接着上面的概念，这里就谈谈classLoader是如何从.dex文件中识别class信息并交给底层的ART识别的。

基础知识

Android系统中的Java虚拟机有三种不同作用的加载器，它们分别是Bootstrap加载器（Bootstrap ClassLoader）、System加载器（System ClassLoader）以及应用加载器（APP ClassLoader）。

  • ClassLoader是抽象类，包含一个parent成员变量，指向被委托的另外一个Class-Loader对象。Android系统中，ClassLoader有两个比较重要的派生类，分别是BootClassLoader和BaseDexClassLoader。

  • BootClassLoader如其名所示，和bootstrap类的加载有关。也就是我们上文提到的Boot加载器。

  • BaseDexClassLoader是Android系统中从dex文件中（文件格式可以是.dex或者包含.dex文件的jar包或apk文件）加载类的关键类。

Android系统中一般不直接使用BaseDexClassLoader对，而是构造它的派生类PathClassLoader的对象。PathClassLoader本身很简单，只有两个构造函数，此外再无其他成员变量和函数。PathClassLoader也就是上文提到的用户自定义加载器。

类的链接(Linker)

在前面我们知道：

  1. ClassLoader 触发加载：Java 代码调用 ClassLoader.loadClass()，首先由 ClassLoader 从文件系统读取 .dex 文件。
  2. 移交虚拟机：ClassLoader.defineClass 调用虚拟机的 JNI 接口（如 DexFile_defineClassNative），进入 ART 的 ClassLinker。

下面我们将进入类的链接过程：ClassLinker::LoadClass解析字节码，生成Class` 对象并返回给 Java 层。

DEX 是源数据，OAT 是编译结果，DexFile 是运行时的 DEX 封装对象，ArtMethod 是 OAT 和 DEX 的桥梁。

ClassLinker 是 ART 虚拟机（或 Dalvik）的核心组件，负责 类的链接（Linking）和初始化。LoadClass 是其内部方法，用于将字节码转换为可执行的运行时数据结构。

工作流程：

  1. 验证与解析：检查字节码合法性，解析类、字段、方法的引用。
  2. 分配内存：为类的方法、静态变量等分配内存空间。
  3. 符号解析：将符号引用转换为直接引用（如方法指针）。
  4. 初始化：执行静态代码块（<clinit>）。

特点：直接操作内存和运行时数据结构（如 ArtMethod、ArtField）。处理 真正的链接和初始化，而不仅仅是文件加载。

ArtMethod、ArtField它们并不直接存放在 .oat 文件中，而是运行时（ART 虚拟机）加载 .oat 文件后，在内存中构建出来的 C++ 对象。

ART 不是只加载 DEX，它要让类能“直接执行本地机器码”，这部分数据在 OAT 文件中。

OAT 文件是 ART 虚拟机对 DEX 字节码进行 AOT（Ahead-Of-Time）编译后生成的可执行文件

因此：

  • DEX 提供“逻辑结构”（类定义、签名、索引）；
  • OAT 提供“物理实现”（编译后的机器码位置、入口点）。

ClassLinker 在加载类时，会尝试关联 OAT 文件中的已编译方法实现。

当 Java 层调用 ClassLoader.loadClass() 或 Class.forName() 时，ART 虚拟机开始加载类：

  1. **ClassLinker::LoadClass**（核心入口）

    从 Dex 文件读取 class_def，初始化 mirror::Class（Java 类在 ART 的表示）。分配 ArtField 和 ArtMethod 数组内存。

  2. VisitFieldsAndMethods（遍历 Dex 数据）

    使用 ClassAccessor 遍历 Dex 中的字段和方法信息。对每个字段和方法，调用 LoadField 和 LoadMethod 填充 ArtField/ArtMethod。

ClassLinker::LoadClass

可以看到该函数完成了：

  1. 字段加载：分配内存空间存储静态字段（sfields）和实例字段（ifields）。处理可能的重复字段定义（兼容性设计）。
  2. 方法加载与链接：为方法分配内存（ArtMethod 数组）。加载方法字节码，并链接到执行代码（AOT 编译的 .oat 或 JIT 生成代码）。处理直接方法（如构造函数）和虚方法（如普通成员方法）。
  3. 优化支持：检查 OatClass 信息，利用 AOT 编译结果加速执行。

如果同早些版本中【《深入理解Android：Java虚拟机ART》为例】做对比，我们可以看到：

方法的存储被拆分为直接方法（direct_methods）和虚方法（virtual_methods），而非单一数组。字段数组的分配逻辑从显式 LoadFields 函数改为内联在 LoadClass 中（通过 VisitFieldsAndMethods 遍历）。

  1. 字段数组：

    统一分配内存：AllocArtMethodArray + SetMethodsPtr。通过 LoadMethod 和 LinkCode 加载和链接方法代码。

ClassLinker::DefineClass:

这个函数主要的作用是：

父类加载与验证

  • 循环继承检测:检查类是否直接继承自身（如 class A extends A），防止简单循环依赖。
  • 父类解析:通过 ResolveType 解析父类类型，可能触发父类的加载（递归过程）。若解析失败（如父类不存在），保留异常并返回 false。
  • 访问权限验证:确保当前类有权访问父类（如非 public 父类需在同一包内）。

接口加载与验证:从Dex文件的 interfaces 列表逐个解析接口类型，并验证访问权限。

状态更新:将类状态从 kIdx（仅索引分配）更新为 kLoaded（父类和接口已加载），标志类层次结构就绪。

VerifyClass

method_verifier.cc - OpenGrok cross reference for /art/runtime/verifier/method_verifier.cc

Android 8.0 (Oreo) 的 AOT/JIT 混合编译
ART 引入了更灵活的验证策略，将验证从类级别拆分为方法级别。这样可以在运行时按需验证方法，避免启动时全量验证的延迟。

Android 10 的优化重点
进一步减少验证对应用启动时间的影响，改为​​懒加载验证​​（Lazy Verification），仅在方法首次执行时触发验证。

代码表现：在 Android 10 的源码中，验证逻辑主要通过 MethodVerifier::VerifyMethod 实现，而类级别的 VerifyClass 被移除，因为类验证被拆解为对每个方法的独立验证。
