解释执行和JIT

ART虚拟机是如何执行Java指令的 ，笼统而言，ART虚拟机中Java指令的执行只有如下两种方式：

  • 解释执行。
  • 执行编译后得到的机器码。

必须说明的是，上述两种执行方式并不是泾渭分明，独立行动。普遍情况下，它们将穿插运行

Runtime::ArtMethod

Runtime ArtMethod是ART虚拟机在运行时动态创建的ArtMethod对象，它们不代表任何Java源码中声明的方法，而是由虚拟机内部创建用于特殊目的的方法结构。

与普通ArtMethod的区别

  1. 来源不同：

    • 普通ArtMethod：对应Java源码中声明的方法
    • Runtime ArtMethod：由虚拟机在运行时动态创建

  2. 生命周期：

    • 普通ArtMethod：随类加载创建，类卸载时销毁
    • Runtime ArtMethod：随虚拟机需要创建，通常在运行时持续存在

  3. 功能目的：

    • 普通ArtMethod：执行用户Java代码
    • Runtime ArtMethod：实现虚拟机内部机制

  4. ImtConflictMethod：

    • 接口方法分派冲突处理
    • 当接口方法分派发现冲突时使用

  5. ImtUnimplementedMethod：

    • 未实现的接口方法处理
    • 用于标记尚未实现的接口方法

  6. ResolutionMethod：

    • 方法解析跳板
    • 处理首次方法调用时的解析过程

  7. AbstractMethodErrorMethod：

    • 抽象方法错误处理
    • 调用抽象方法时抛出异常

  8. NoSuchMethodErrorMethod：

    • 方法不存在错误处理
    • 调用不存在的方法时使用

  9. RuntimeMethod：

    • 通用运行时方法
    • 用于各种运行时特殊处理

解释执行

.so 文件（native 库）是本地机器码，由操作系统内核直接加载和执行，不经过 ART 虚拟机。
ART 只负责执行 Java 层 / DEX / OAT 层 的字节码或编译代码；
一旦进入 JNI 或 System.loadLibrary() 调用 .so，
就跳出了 ART 的控制范围，进入 Linux 内核和动态链接器（ld.so）管理的原生执行空间。

ART 只负责找到 .so 中对应的函数指针（通过 JNI 注册或反射），
一旦跳转过去，执行权就交给本地 CPU 指令流（不再经过 ART）。

  • ExecuteGotoImpl：C++编写，基于goto逻辑实现。但它不支持clang编译。

传统 switch-case 的问题

  • 分支预测失败：switch 语句依赖跳转表，当指令类型随机出现时，CPU 分支预测器效率下降。
  • 指令缓存压力：庞大的 switch 代码块（200+ 个 case）导致缓存行利用率降低。

handler.OPCODE_NAME() 的优势

  • 直接调用：通过模板生成的独立处理函数，每个指令的处理逻辑更紧凑，提高 CPU 缓存局部性。
  • 内联优化：编译器可针对高频指令（如 ADD_INT）单独内联，减少跳转开销。

art_quick_invoke_static_stub

quick_entrypoints_x86.S - OpenGrok cross reference for /art/runtime/arch/x86/quick_entrypoints_x86.S

首先准备好栈空间，尤其是将机器码函数所需的参数拷贝到栈上。在相关参数中，EAX寄存器存储着目标方法对应的ArtMethod对象。

然后通过call指令跳转到该ArtMethod对象的机器码入口。如此这般，我们就将以机器码方式执行这个方法。

该方法的机器码执行完后将返回到art_quick_invoke_stub执行。此时，art_quick_invoke_stub将把执行结果存储到result位置。

当调用流程从art_quick_invoke_stub返回后，解释执行的处理逻辑就得到了方法C机器码执行的结果。
