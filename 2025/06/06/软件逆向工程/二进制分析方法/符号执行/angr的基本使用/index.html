angr的基本使用

在 angr 框架中，Project 类是分析二进制文件的入口点，它封装了程序的所有信息和分析接口。以下是 project.loader 和 project.factory 的详细说明，以及 Project 下的关键组件：

1. angr.Project 的核心组件

属性/方法 作用
project.loader 管理二进制文件的加载和内存布局（通过 CLE 模块实现）。
project.factory 创建符号执行的状态（state）和模拟管理器（simulation_manager）。
project.analyses 提供静态分析功能（如 CFG 生成、数据依赖分析）。
project.kb 知识库（Knowledge Base），存储分析结果（如函数、钩子、符号表信息）。
project.hook 用于劫持/替换特定地址的代码（如系统调用劫持）。
project.arch 目标架构信息（如 x86、ARM）。

2. project.loader：二进制加载器

project.loader 是 CLE（Cle Loads Everything） 模块的实例，负责加载二进制文件及其依赖库，并提供内存空间管理。

关键功能

属性/方法 说明
loader.main_object 主程序二进制对象（如 ELF 文件）。
loader.shared_objects 动态链接库字典（如 libc.so）。
loader.memory 虚拟地址空间管理，支持符号化内存访问。
loader.find_symbol(name) 通过符号名查找地址（如 "printf"）。
loader.mapped_base 主程序的加载基地址（对 PIE 程序很重要）。

project.factory 用于创建符号执行的初始状态和控制模拟过程。

关键方法

方法 作用

factory.entry_state(args) 从程序入口点创建状态，支持符号化命令行参数。
factory.call_state(addr, args) 模拟调用特定函数（addr）的状态。
factory.simulation_manager(state) 创建模拟管理器，控制符号执行过程（路径探索、合并等）。

2. sm.step() 如何遍历路径？

（1）分支分裂机制

  • 条件分支：
    当遇到 if (x > 0) 时（x 是符号变量），sm.step() 会生成两个状态：
    • 状态A：x > 0 为真，继续执行 if 块。
    • 状态B：x > 0 为假，执行 else 块。
      ​​此时 sm.active 数量从 1 变为 2​​。
  • 间接跳转：
    如 jmp [eax]（eax 是符号值），会尝试求解 eax 的可能取值，分裂出多个状态。

（2）路径独立性

每个分裂的状态独立执行，维护自己的：

  • 寄存器/内存值
  • 路径约束（如 x > 0 ∧ y == 10）
  • 程序计数器（PC）

在 angr 的符号执行中，sm.step() 和 while len(sm.active) > 0 的组合是为了 逐步追踪所有路径的执行过程，而不仅仅是“完成遍历”。以下是详细解释：

1. sm.step() 的单步性

  • sm.step() 的局限性：
    每次调用 sm.step() 仅执行 ​​当前所有活跃状态的一条指令​​，并更新状态集合（sm.active）。
    • 它不会自动完成所有路径的遍历，只是推进一步。
    • 分支分裂：遇到条件分支时，会生成新状态，但需要多次 step() 才能完全展开所有路径。
  • 类比调试器：
    • sm.step() 类似 手动按 F7，每次只走一步，需多次调用才能覆盖整个程序。
    • while 循环相当于 反复按 F7，直到所有路径终止。

6. 总结

  • sm.step()：单步推进所有活跃状态，不保证完成所有路径。

  • while 循环

    ：

    • 确保所有路径被完全展开（从开始到终止）。
    • 动态记录每个基本块的访问情况（flow.add(active.addr)）。

  • 关键区别

    ：

    • 单次 step() ≈ 按一次 F7（仅走一步）。
    • while 循环 ≈ 持续按 F7 直到程序结束。

这种设计允许精细控制符号执行过程，同时确保路径覆盖的完整性。

、在 angr 中，关于 sm.step() 的行为，两种描述都是正确的，但需要从不同层次理解其工作逻辑。以下是关键点解析：

1. sm.step() 的核心行为

特性 说明
单步执行 对当前所有活跃状态（sm.active）各执行一条指令
自动分支分裂 若某状态的指令是条件分支（如 jz/jnz），会生成对应路径的新状态
非递归推进 仅执行一步，不会自动递归展开所有路径，需多次调用 step() 才能继续探索

2. 看似矛盾的解释

(1) “探索所有可能路径”

  • 分支分裂时：当某个活跃状态遇到分支指令，step() 会生成两个新状态（分别对应条件成立/不成立），保证所有可能性被捕获。
  • 但：新状态只是加入状态集合，不会立即执行后续指令，需再次调用 step() 推进。

(2) “不会自动完成遍历”

  • 单步性：每次 step() 只执行当前活跃状态的一步，不自动递归执行新生成的状态。
  • 需要循环：完全展开所有路径需手动循环调用 step()，直到满足终止条件（如所有路径结束或超时）。

4. 关键结论

  • 正确表述：
    sm.step() 会处理当前所有活跃状态的分支分裂（保证路径完整性），但不会自动递归执行新状态（需用户显式推进）。
  • 用户责任：
    需要手动管理状态探索策略（如循环调用 step() 或使用 sm.run() 自动完成）。
