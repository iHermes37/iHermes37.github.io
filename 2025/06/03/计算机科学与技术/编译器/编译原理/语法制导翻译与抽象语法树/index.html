语法制导翻译与抽象语法树

前情回顾

前面“语法分析”中，我们了解到：

语法分析的任务是检查输入程序是否符合文法规则（如 if (x > 0) {...} 是否满足 if (表达式) 语句 的结构）。但它无法处理以下问题：

  • 变量 x 是否已声明？
  • x > 0 中的 x 是否是数值类型？
  • if 的条件表达式是否返回布尔值？

同时在“编译原理导读“中，注意到语义分析的任务是验证程序的逻辑正确性（如类型匹配、作用域规则）。其语义规则通常依赖语法结构的上下文。【上下文无关文法（CFG）无法直接表达变量作用域、类型兼容性等上下文相关约束。】

例如：在表达式 a + b 中，+ 的合法性取决于 a 和 b 的类型（需通过语法树父子/兄弟节点传递信息）。

试想一下传统方法里若完全分离语法和语义分析，可能需要先构建完整的语法树（AST）【后面讲解如何从语法分析树到抽象语法树】，再单独遍历AST进行语义检查，效率较低。

为了解决这些问题从而引入的语法制导翻译：

通过将语义动作（如属性计算）直接绑定到文法规则上，实现模块化设计。语法分析器只需按文法推导，自动触发相关语义动作，二者解耦。

  • 通过声明式的语义规则（如 E.val = E₁.val + T.val），编译器工具（如Yacc）可自动生成翻译代码，减少人工干预。在语法分析的同时，通过附加的语义动作（如查符号表、类型检查）
  • 通过属性传递（如继承属性）实现上下文信息传播。【通过属性文法（继承属性/综合属性）在语法树中传递上下文信息。】
  • 适应不同语法分析方法，解决了自顶向下（LL）和自底向上（LR）分析需要不同的语义处理方式。综合属性（自底向上）：LR分析中直接计算父节点属性（如表达式求值）。继承属性（自顶向下）：LL分析中传递上下文信息（如变量类型）
  • 在语法分析过程中即时触发语义动作，边解析边检查，减少额外遍历。

这里提一个不成熟的问题：语法制导翻译使得语法分析也有了语义检查的能力，为啥还要语义分析？【语法制导翻译（SDT）在语法分析阶段已经能够处理许多语义任务（如属性计算、简单类型检查）】

职责分工不同

SDT的核心任务： 在语法分析过程中即时执行局部语义动作（如计算表达式值、填充符号表）。 局限性：它仅能处理与当前产生式直接相关的、局部的语义规则（例如 E → E + T 中的类型一致性检查）。

语义分析的核心任务：

  • 全局性检查：如函数的所有路径是否有返回值、循环中的变量是否被合理使用。
  • 跨上下文依赖：如检查函数调用时的实参与形参匹配，这需要遍历整个符号表，超出单个语法规则的范围。

类比： SDT 像“实时语法检查器”，在写代码时提示拼写错误。 语义分析 像“全局代码审查”，检查整个项目的逻辑一致性。

处理问题的复杂度不同

SDT能处理的： 简单属性传递（如 E.val = E₁.val + T.val）。依赖父子/兄弟节点的上下文（如继承属性）。

需要语义分析的：

  • 跨作用域规则：如变量是否在嵌套作用域中被重复定义？

  • 复杂类型系统：如多态函数调用时的类型推导、泛型实例化。

  • 控制流检查：如 break 语句是否位于循环内、return 是否覆盖所有分支。

阶段分离的必要性

SDT的局限性：

  • 单遍限制：SDT通常在语法分析的单遍处理中完成，而某些语义检查需要完整语法树（如Java的“向前引用”变量）。
  • 无法回溯：SDT无法在解析到某处时回头修改之前的结果（如类型推断需多次迭代）。

语义分析的全局视角： 需要完整遍历AST或符号表，进行跨函数/跨文件的综合分析。

典型场景：

  • C++的模板实例化、Python的闭包变量捕获等，均需在语义分析阶段解决。

SDT vs 语义分析的对比表

特性 语法制导翻译（SDT） 语义分析
触发时机 语法分析过程中即时触发 语法分析完成后，基于完整AST或符号表
范围 局部（当前产生式周边） 全局（跨作用域、跨函数）
典型任务 填充符号表、简单类型检查、代码生成片段 类型推导、控制流检查、生命周期验证
工具支持 可自动生成（如Bison） 通常需手动实现
属性传递 依赖综合/继承属性 可能需多遍遍历或复杂推理
  • SDT 是语法分析和语义分析之间的“粘合剂”，负责局部的、上下文相关的语义处理。
  • 语义分析 是独立的深层验证阶段，解决全局的、跨上下文的逻辑问题。
  • 二者协作：SDT快速过滤低级错误，语义分析确保高级逻辑正确，共同保证编译的严谨性和效率。

这种分工类似于“语法分析是认字，SDT是组词造句，语义分析是检查文章逻辑”——缺了哪一步都无法产出正确的程序。

关键任务

  • SDT：在语法分析过程中（归约或推导时）触发语义动作，完成：
    1. 构建AST节点（如将 E → E + T 转换为 BinaryOpNode(+)）。
    2. 填充符号表（记录变量类型、作用域）。
    3. 生成中间代码（如 t1 = b * 2; t2 = a + t1）。
  • AST：通过SDT的语义动作构造，省略无关语法细节。

下面我们将先了解该过程输出的【抽象语法树】的结构，在进一步了解语法制导翻译是如何构建这样一颗树的

抽象语法树（AST）

AST是源代码的结构化中间表示，它省略了语法细节（如括号、分号），仅保留程序逻辑的关键结构，便于后续语义分析和代码生成。

分析树包含了一些不必要的信息，分析树的 左右括号、E等信息的叶子位置我们可以拿来直接存储具体的数值，去掉这些东西也没有什么影响。我们也不需要用括号来区分优先级，通过树的深度就可以知道推导的优先级。——越靠下面的树枝优先级越高。

  • 具体语法是语法分析器所采用的的语法，关注语法的约束和改写、关注消除左递归等等。
  • 抽象语法是一种接口，是语法分析器的最终生成产物，一旦生成抽象语法树就会抛弃源代码，下一步就交给语义分析了。

抽象语法树 vs 具体语法树

抽象语法树的结构

  • 每个 结点 代表一个语法结构，对应于一个 运算符
  • 结点的每个 子结点 代表其子结构，对应于 运算分量
  • 表示这些子结构按照特定方式组成更大的结构
  • 可以忽略掉一些标点符号等非本质的东西

语法树的表示方法

每个结点用一个对象表示，对象有多个域

  • 叶子结点中只存放词法值
  • 内部结点中存放 opop（操作符）值和参数（通常指向其它结点）

语法制导定义(SDD)

语法制导定义是编译原理中一种形式化方法，用于将语义规则（如属性计算、代码生成）与文法产生式关联，从而在语法分析过程中自动执行语义动作。它是语法制导翻译（SDT）的理论基础，通过属性文法（Attribute Grammar）描述程序语言的语义。

语法制导定义 (Syntax-Directed Definition, SDD) 是一个上下文无关文法 (Context-Free Grammar, CFG) 和属性及规则的结合，属性和文法符号相关联，而规则和产生式相关联。如果 X 是一个符号而 a 是 X 的一个属性，那么我们用 X.a 表示 a 在某个标号为 X 的分析树结点上的值。

基本组成

  • 上下文无关文法（CFG）：定义语言的结构（如 E → E + T）。
  • 语义规则：为每个产生式附加的属性计算规则。
  • 属性：
    • 综合属性（Synthesized Attributes）：自底向上传递（子节点→父节点）。
    • 继承属性（Inherited Attributes）：自顶向下传递（父节点→子节点）。

形式化表示:对每个产生式 A → α，关联一组形如 b = f(c₁, c₂, ..., cₖ) 的语义规则，其中：

  • b 是 A 或 α 中符号的属性。
  • c₁, ..., cₖ 是其他属性的值。

属性分类与计算

综合属性

在分析树节点Ｎ上的非终结符号Ａ的综合属性是由Ｎ上的产生式所关联的语义规则来定义的。注意：这个产生式的头一定是Ａ。节点Ｎ上的综合属性只能通过Ｎ的子节点或Ｎ本身的属性值来定义。【终结符可以有综合属性，为词法分析器提供的词法值】

  • 方向：子节点→父节点（自底向上）。

  • 用途：计算表达式值、构建AST节点等。

继承属性

在分析树节点N上的非终结符号B的继承属性是由N的父节点上的产生式所关联的语义规则来定义的。注意：这个产生式的体中必然包括符号B。节点N上的继承属性只能通过Ｎ的父节点、N本身和N的兄弟节点上的属性值来定义。【终结符没有继承属性；终结符本身的属性已经被归为了综合属性】

  • 方向：父节点→子节点（自顶向下）。
  • 用途：传递上下文信息（如变量类型、作用域）。

节点N上的继承属性不可以通过Ｎ的子节点上的属性值来定义，但节点Ｎ上的综合属性可以通过Ｎ本身的继承属性来定义。终结符号可以具有综合属性，但是不能具有继承属性。终结符号的属性值是由词法分析器提供的词法值。在SDD中没有计算终结符号的属性值的语义规则。

SDD 求值顺序

SDD为CFG中的文法符号设置了语义属性。语义规则建立了属性之间的依赖关系，在对语法分析树节点的一个属性求值之前，必须首先计算出这个属性值所依赖的所有属性值。

和终结符号的综合属性直接由词法分析器给出不同，非终结符号的综合属性和继承属性都或多或少地依赖于其他属性，因此确定一棵注释语法分析树中属性的求值顺序是十分重要的，依赖图可以帮助我们完成这个工作。

SDD的两种类型

之前我们讨论过，SDD的求值顺序很重要，产生循环的SDD不可能找到一个求值顺序。从计算的角度看，给定一个SDD，很难确定是否存在某棵语法分析树，使得SDD的属性之间存在循环依赖关系，但是，确实存在SDD的有用子类，能够保证对每棵语法分析树都存在求值顺序，因为它们不允许产生带有环的依赖图。而且，这两类SDD可以和自顶向下以及自底向上的语法分析过程一起高效地实现。这节讨论这两类重要的SDD：“S属性SDD”和“L属性SDD”。“L属性翻译”（L代表从左到右）的语法制导翻译方案，这一类方案代表了所有可以在语法分析过程中完成的翻译方案。以及“S属性翻译方案”（S代表综合）这类方案可以很容易地和自底向上语法分析过程联系起来。

语法制导翻译（SDT）

语法制导翻译是一种在语法分析过程中（自顶向下或自底向上）执行翻译动作的技术，通过将语义规则（如代码生成、类型检查）与文法产生式关联，实现从源代码到目标代码或中间表示的转换。

定义：语法制导的翻译方案（syntax-directed translation scheme，SDT）是对语法制导定义的补充，也称作语法制导的翻译模式。

  • 把 SDD 的 语义规则改写为计算属性值的程序片段，用花括号 {}{} 括起来，插入到产生式右部的任何合适的位置上
  • 这种方法表示语法分析和语义动作交错，可以在按 深度优先 遍历分析树的过程中随时执行语义动作

说人话：

  • SDT 是在语法分析过程中附带语义动作（程序计算片段）
  • 语义动作可以放在产生式的任意位置，通常用大括号 {}{} 包围

SDT是在产生式体中嵌入了称为语义动作的程序片段的CFG，对文法符号的属性值进行计算。

一般情况下，语义动作可以出现在产生式体中的任何位置。SDD是关于语言翻译的高层次规格说明，隐藏了许多实现细节；SDT可以看做是对SDD的一个补充，它显式地指明了语义规则的计算顺序，以便说明某些实现细节。

SDD翻译成SDT

如上所述，语法制导的翻译方案（syntax-directed translation scheme,SDT）是在产生式中嵌入了程序片段的上下文无关文法。这些程序片段称为语义动作，可以出现在产生式体中的任何地方。

这里主要关注如何使用SDT来实现两类重要的SDD，因为在这两种情况下，SDT可以在语法分析过程中实现

  1. 基本文法可以使用LR分析技术，且SDD是S属性的
  2. 基本文法可以使用LL分析技术，且SDD是L属性的

S属性的SDD翻译成SDT比较简单，所有产生的动作位于产生式的右端，也称后缀SDT。因为S属性只包括综合属性，依赖于子结点，因此需要在所有的子结点全部分析处理完毕之后处理。

L 属性定义的 SDT

L属性的SDD翻译成SDT的规则如下：

  • 将计算某个非终结符号A的继承属性的动作插入到产生式右部中紧靠在A的本次出现之前的位置上；
  • 将计算一个产生式左部符号的综合属性的动作放在这个产生式右部的最右端

语法制导翻译的应用

语法制导的翻译技术通常用于类型检查与中间代码生成。本节主要应用与抽象语法树的构造。为了完成到中间代码的翻译，编译器接下来可能使用一组规则来编译这棵语法树 (实际建立在 SSD 上)。

抽象语法树的构建

一棵语法树中的每个结点代表一个程序构造，这个结点的子结点代表这个构造有意义的组成部分。比如表达式 E1+E2E_{1}+E_{2}E1+E2 的语法树结点的标号为 +++，两个子结点分别代表子表达式 E1E_{1}E1 和 E2E_{2}E2。

我们将使用具有适当数量的字段的对象来实现一棵语法分析树的各个结点。每个对象将有一个 op 字段，即这个结点的标号。这些对象将具有如下所述的其他字段：

  • 如果结点是叶结点，那么对象将有一个附加字段来存放这些叶节点的词法值。构造函数 Leaf(op, val) 创建一个叶对象。也可以把结点看作记录，那么 Leaf 可以返回指向叶结点对应的新记录的指针。
  • 如果结点是内部结点，那么它的附加字段的个数与该结点在语法分析树中的子结点个数相同。构造函数 Node 带有两个或多个参数 Node(op, c1, c2, ..., ck)。

比如说输入 a−4+ca-4+ca−4+c 构造一棵抽象语法树，这棵抽象语法树的结点被显示为记录，这些记录的第一个字段是 op。现在抽象语法树的边用实线表示，基础的语法分析树的边用点状虚线表示 (并没有真的生成它)，最后一种线状虚线表示 E.node 和 T.node，每条线都指向适当的抽象语法树的结点。
