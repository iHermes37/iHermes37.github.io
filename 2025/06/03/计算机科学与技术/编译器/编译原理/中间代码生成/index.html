中间代码生成

前情回顾

在“语义分析”一节中，我们知道语义分析为中间代码生成提供已验证的AST（如类型注解后的树）。中间代码生成是前端中的最后一步

最简单的一种情况，抽象语法树直接翻译成目标代码（汇编代码）。

但是现代编译器会生成很多的中间表示，经过不断的翻译或者优化，最终才会生成汇编代码。

在将给定源语言的一个程序翻译成特定机器代码的过程中，一个编译器可能构造出一系列中间表示。高层的中间表示接近源语言，而底层的表示接近目标语言。语法树是高层表示，它刻画了源程序的自然层次性结构，且适用于静态类型检查。低层表示适用于机器相关处理，如寄存器分配、指令选择等。

编译器引入中间代码(Intermediate Representation, IR)这一层是编译过程中的关键设计，主要原因包括：

类别 说明
主要作用
抽象与解耦 分离前端（语法分析）和后端（代码生成），使两者独立演化。
多目标支持 同一IR可生成不同架构（x86、ARM等）的机器码，减少重复工作。
优化便利性 在IR层进行跨平台的通用优化（如常量传播、循环优化）。
多语言支持 不同语言（C/C++、Rust等）可编译到同一IR，实现语言互操作。
核心优势
降低开发复杂度 前端和后端只需关注IR接口，无需直接耦合。
提高可移植性 新增目标平台只需实现新后端，无需修改前端。
统一优化流程 优化算法只需针对IR实现一次，即可应用于所有语言和平台。

地址和指令

三地址码基于两个基本概念：地址和指令。地址通常具有以下形式之一：

  • 名字，为方便起见，允许源程序名字作为三地址码的地址。实现中通常是指向符号表的指针
  • 常量
  • 编译器生成的临时变量

还有其他一些三地址指令的形式：

四元式表示

前面一节中介绍了三地址码的具体组成，但在数据结构上的表示方法，常用四元式、三元式和间接三元式这三种描述方式。

四元式 (quadruple) 有四个字段，分别是 op、arg1、arg2 和 result，其中 op 是运算符的内部编码。比如三地址码 x=y+z 可以用四元式 (=, y, z, x) 表示。但有一些特例：

  1. 形如 x = -y 这样的单目运算符和赋值运算符 x = y，不使用 arg2
  2. 形如 param 这样的运算既不使用 arg2 也不使用 result
  3. 条件或非条件转移指令将目标标号存入 result 字段

比如说语句 a = b * -c + b * -c，可以得到三地址码

三元式表示

三元式 (triple) 只有 op、arg1、arg2 三个字段，四元式中 result 字段主要保留的是临时变量，因此三元式中用运算的位置表示其结果。但是像 x[i] = y 这样的三元式，需要两个条目，我们可以把 x 和 i 置于同一个三元式，y 置于另一个三元式。

作为语句 a = b * -c + b * -c，可以得到三元式

对于优化器来说，指令的位置会经常发生变化，此时四元式用临时变量引用结果，不会有任何变化，但三元式中使用位置引用结果的，这将导致需要频繁修改指令。

间接三元式 (indirect triple) 包含了一个指向三元式的指针列表，而不是指向序列本身。在优化重排三元式时，可以直接对间接的指针列表进行重排，不会影响三元式本身。

当然间接引用列表和三元式表不在一起，只是为了方便展示，才放在一起。

静态单赋值形式

静态单赋值形式 (static single assignment, SSA) 是另一种中间表示，它有利于实现某种类型的代码优化。SSA 中所有的赋值都是针对具有不同名字的变量。

SSA（Static Single Assignment，静态单赋值） 是编译器中间表示（IR）的一种形式，其核心规则是：

  • 每个变量只能被赋值一次（静态单次定义）。
  • 若变量需要多次赋值，必须引入新版本变量（通常通过下标区分，如 x1, x2）。

其特点在于：

特性 说明 示例
单次赋值 变量一旦定义，不能被重新赋值。 x1 = 1（合法） x1 = 2（非法，需改为 x2 = 2）
版本化变量 通过数字后缀区分不同版本（如 x_1, x_2）。 原始代码：x = x + 1 → SSA形式：x_2 = x_1 + 1
Phi（φ）函数 在控制流合并点（如 if-else 后）选择正确的变量版本。 x_3 = φ(x_1, x_2)（若来自分支A用 x_1，分支B用 x_2）

基于SSA可以进一步对程序进行优化

优势 应用场景
简化优化分析 明确的变量依赖关系，便于数据流分析（如常量传播、死代码删除）。
消除冗余 避免重复计算同一变量（如循环不变式外提）。
并行化支持 清晰的变量版本依赖，有助于指令调度和寄存器分配。

三地址码（Three-Address Code, TAC）是一种线性中间表示，每条指令形如 x = y op z。虽然简洁，但缺乏显式的控制流信息，导致：

  • 无法直接回答跳转目标问题
    TAC 中的跳转（如 goto L2）是文本标签，需遍历整个代码才能统计哪些指令跳转到 L2，效率低下。
  • 难以分析循环和分支
    TAC 中的 if-else 或 while 被拆分为多个 goto，需手动重建控制流结构才能分析。
  • 优化受限
    例如，死代码删除需要知道某段代码是否被引用，而 TAC 无法快速获取引用关系（需额外构建 CFG）。

基本概念

控制流图（CFG）是编译器用于分析和优化程序控制流的核心数据结构，而基本块是构成CFG的基本单元。以下是它们的核心概念、特性和应用。

基本块

一个基本块是满足以下条件的线性指令序列：

  1. 单入口：只有第一条指令可以被跳转到（其他指令不能作为跳转目标）。
  2. 单出口：只有最后一条指令是终止指令（如br、ret、call等），其他指令不能包含跳转或返回。
  3. 无内部跳转：块内没有分支或循环（即顺序执行）。

示例（LLVM IR）:

  • 合法基本块：以br终止，无内部跳转。
  • 非法基本块：若在中间插入br或ret，则违反单出口规则。

控制流图

控制流图以有向图的形式表示程序的控制结构，其中：

  • 结点（Node）：表示基本块（Basic Block），即顺序执行的指令序列。
  • 边（Edge）：表示跳转关系（如条件分支、循环、函数调用等）。
概念 说明
前驱（Predecessor） 指向当前块的所有基本块（即“谁可能跳转到这里”）。
后继（Successor） 当前块可能跳转到的所有基本块（即“下一步去哪里”）。
支配（Dominance） 如果从入口到块B的所有路径必须经过块A，则A支配B（用于优化和循环分析）。
回边（Back Edge） 指向祖先块的边（用于识别循环，如while的跳转到循环头）。

CFG的构建步骤

  1. 划分基本块：
    • 扫描指令序列，以终止指令（如br、ret）或跳转目标（如label %loop）为边界分割块。
  2. 建立边关系：
    • 对每个基本块，分析其终止指令的跳转目标，添加CFG边。
  3. 处理特殊结构：
    • 循环：通过回边识别（如br label %loop指向已访问的块）。
    • 不可达代码：无前驱的块（可删除）。

CFG的应用

编译器优化

优化技术 依赖CFG的分析
死代码删除 删除未被任何路径执行的代码（无入边的块）。
常量传播 沿CFG传播常量值（如%x = 1在所有路径中均为1）。
循环优化 通过回边识别循环，进行外提、展开等优化。

静态分析

  • 检测未初始化变量：检查变量在所有路径中是否被定义。
  • 路径敏感分析：如符号执行需遍历CFG的所有可能路径。

反编译与调试

  • 从机器码重建CFG，帮助理解程序逻辑。
  • 可视化CFG辅助调试（如LLVM的opt -view-cfg）。
