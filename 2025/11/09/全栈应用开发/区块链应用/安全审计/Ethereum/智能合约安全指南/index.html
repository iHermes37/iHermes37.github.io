智能合约安全指南

ABI Hash Collisions

在以太坊智能合约中，ABI（应用二进制接口）是一个非常重要的标准，它用于智能合约与外部世界（比如用户或其他合约）之间的交互。每次与合约交互时，数据（如函数参数）都需要进行编码，确保可以被正确地传输和解码。

有两种常见的编码方法：

  1. **abi.encode**：更为安全，它会对输入的数据进行严格的类型和长度校验，生成的数据比较大，因此会消耗更多的 gas。
  2. **abi.encodePacked**：这种方式将所有的输入紧密打包在一起，节省了存储空间，生成的数据较小，因此更便宜，但缺乏类型的严格校验，容易发生哈希碰撞。

哈希碰撞是指不同的数据输入可能生成相同的哈希值。这个问题尤其在 abi.encodePacked 中非常常见，因为它没有对每个参数的边界做严格分隔，从而容易引发两个不同的数据结构生成相同的哈希。

如果一个攻击者能够制造出两个不同的输入，它们生成相同的哈希值，就可能成功绕过 require(allowedPayouts[payoutKey], "Unauthorized claim"); 这一权限检查，从而使得他们能够非法领取本不属于他们的奖励。

Ambiguous Evaluation Order

The evaluation order of expressions is not specified (more formally, the order in which the children of one node in the expression tree are evaluated is not specified, but they are of course evaluated before the node itself). It is only guaranteed that statements are executed in order and short-circuiting for boolean expressions is done.

  • High-level language: a + b might evaluate a then b, or b then a.
  • EVM: everything is step-by-step stack instructions.

Solidity ensures that each statement is executed in order, but within an expression, the order is unspecified unless you explicitly separate statements.

To mitigate this vulnerability, a prevalent strategy involves storing intermediate results in temporary variables. This method ensures that the evaluation order remains unambiguous, regardless of compiler variations or complex functional interactions.

Approval Vulnerabilities

Exposed Data

在区块链上存储任何形式的可识别个人身份的信息都是危险的，既可能被攻击者利用，也可能触犯数据保护法律。即使看似“匿名”的链上操作，也有被去伪匿名化的风险。

公共可访问性（Public Accessibility）

  • 区块链上的数据是公开且不可篡改的。
  • 任何人都可以通过区块链浏览器查看和分析这些数据。
  • 这对于保证透明性和信任是优点，但对敏感数据则是问题。

风险：

  • 如果你把 PII 或可追踪到 PII 的信息（如用户的操作记录、身份标识符）放上链，攻击者可能将这些链上数据和现实世界的信息关联起来，从而揭示用户真实身份。
  • 区块链的“匿名”实际上只是伪匿名（pseudonymous），并不能真正保护个人身份。
