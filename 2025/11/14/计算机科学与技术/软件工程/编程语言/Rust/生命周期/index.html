生命周期

为什么需要生命周期？

因为 Rust 的引用（&T）不拥有数据，只是借用。编译器必须确保：

  • 引用不能比它指向的数据活得更久（避免悬空）
  • 可变引用唯一（避免数据竞争）

Rust 的核心目标是内存安全，要防止“悬垂引用”（Dangling Reference）：

这里，r 变量被赋予了生命周期 'a，x 被赋予了生命周期 'b，从图示上可以明显看出生命周期 'b 比 'a 小很多。

在编译期，Rust 会比较两个变量的生命周期，结果发现 r 明明拥有生命周期 'a，但是却引用了一个小得多的生命周期 'b，在这种情况下，编译器会认为我们的程序存在风险，因此拒绝运行。

如果想要编译通过，也很简单，只要 'b 比 'a 大就好。总之，x 变量只要比 r 活得久，那么 r 就能随意引用 x 且不会存在危险：

生命周期 = 编译器用来推断“引用活多久”的一种静态标注系统，用来保证引用永远指向有效内存。它不影响运行时，只约束编译期。可以把它理解成：

Rust 在编译期做“逃逸分析 + 引用有效性验证”。
C/C++ 是不做检查，可能悬空；Java/Go 等由 GC 做；Rust 由类型系统静态保证。

生命周期约束 = 让借用检查器能证明“引用安全”

  • **借用 = 不变性约束（不可同时存在冲突）
  • 生命周期 = 活跃区间约束（引用不能悬空）**

多线程下的 Send/Sync 与生命周期共同保证：

  • 无数据竞争
  • 无悬空引用
  • 无 use-after-free

Rust 靠类型系统在编译期完成所有验证。更深层理解：生命周期 ≈ 区域（Region）分析

从系统安全的视角可以这样理解：Rust 编译器会把代码划分成不同生存区域（region），'a、'b、'static 等都是这些区域的“名字”，编译器根据使用关系构建一个“区域约束图”，最终求出最小满足约束的解
⇒ 保证引用安全

即：Rust 生命周期 = 编译期间的形式化推理系统。

为什么要 'a？因为返回值引用可能来自 x 或 y，编译器必须知道：返回的引用至少与两个参数中较短的生命周期一致。所以 'a 不是“真实的时间”，而是“统一的关系标签”。

不加生命周期会怎样？

编译器报：无法自动推断返回值引用与哪个参数生命周期一致。也就是：“我不知道你会返回谁，所以不能保证它有效”。

三种常见情况

Rust 有 3 条内置规则，大部分函数都能自动推断：

  1. 每个输入引用都有独立生命周期。
  2. 如果只有一个输入引用，则输出引用与它绑定。
  3. 如果 &self 或 &mut self 出现，输出默认跟 self。
