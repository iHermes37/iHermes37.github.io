SOLID 原则

SOLID 原则是面向对象编程和设计中的五个基本原则，它们旨在使软件设计更易于理解、维护和扩展。下面以表格形式为您清晰呈现。

SOLID 五大设计原则

原则名称 核心思想 描述 遵循后带来的好处 违反的常见症状
S
单一职责原则
(Single Responsibility Principle)		 一个类应该有且仅有一个引起它变化的原因。 一个类应该只负责一项职责。这意味着将不同的变化原因分离到不同的类中。 - 高内聚：类的功能集中，易于理解。
- 低耦合：减少类之间的依赖。
- 易于维护：修改一个功能不会影响其他功能。
- 提高复用性。		 - 类非常庞大，做了很多事。
- 修改一个功能，会影响到其他不相关的功能。
- 类的方法被其他模块以“组合”的形式调用（即只用了某个方法，却要引入整个类）。
O
开闭原则
(Open/Closed Principle)		 软件实体（类、模块、函数等）应该对扩展开放，对修改关闭。 当需要改变程序的功能或增加新功能时，应该通过添加新的代码来实现，而不是修改已有的代码。 - 稳定性：核心模块无需修改，系统更稳定。
- 可扩展性：易于添加新功能。
- 便于进行单元测试。		 - 每次添加新功能，都需要修改现有的核心类。
- 添加一个小功能，却需要在多个地方进行 if/else 或 switch 判断。
- 测试原有功能时，因代码被修改而经常需要回归测试。
L
里氏替换原则
(Liskov Substitution Principle)		 所有引用基类（父类）的地方必须能透明地使用其子类的对象。 子类必须能够完全替代它们的父类，而不产生任何错误或异常。子类可以扩展父类的功能，但不能改变父类的原有功能（行为）。 - 保证继承体系的健壮性。
- 是开闭原则的重要实现方式之一。
- 加强程序的可复用性和可维护性。		 - 子类覆盖或改变了父类的方法，导致功能不一致。
- 在需要使用父类对象的地方，传入子类对象后，程序出现错误或行为异常。
- 需要检查对象类型来决定执行哪个子类的方法（如 if (obj is ClassA)）。
I
接口隔离原则
(Interface Segregation Principle)		 客户端不应该被强迫依赖于它不使用的接口。 将一个庞大的、臃肿的接口拆分成更小、更具体的接口，让客户端只需要知道它们感兴趣的方法。 - 高内聚：接口职责单一。
- 低耦合：类之间的依赖关系建立在最小的接口上。
- 避免接口污染，减少不必要的依赖。		 - 接口中声明了大量方法，但实现类中很多方法都是空实现。
- 客户端被迫实现它根本不需要的方法。
- 一个接口的修改会影响到所有它的实现类，即使有些类并不需要这个新方法。
D
依赖倒置原则
(Dependency Inversion Principle)		 1. 高层模块不应该依赖于低层模块，二者都应该依赖于抽象。
2. 抽象不应该依赖于细节，细节应该依赖于抽象。		 程序应该依赖于抽象接口（Abstract Interfaces），而不是具体实现（Concrete Implementations）。这通过依赖注入等方式实现。 - 降低耦合：模块间通过抽象交互，减少直接依赖。
- 提高灵活性：易于替换不同的实现（例如，更换数据库、更换支付方式）。
- 便于单元测试：可以通过 Mock 抽象接口来测试高层模块。		 - 高层模块直接 new 了一个低层模块的对象。
- 想替换一个底层组件（如从 MySQL 换到 PostgreSQL）需要大量修改代码。
- 难以对某个类进行单元测试，因为它强依赖于一个难以模拟的外部资源。

核心思想总结

  • S (SRP) 和 I (ISP) 主要关注职责的分离和接口的粒度，目标是降低复杂性。
  • O (OCP) 和 D (DIP) 是构建灵活、可扩展架构的核心，目标是应对变化。
  • L (LSP) 是继承关系的基石，确保了多态性的正确使用，是 OCP 和 DIP 能够有效实施的重要保障。

这五个原则相辅相成，共同构成了面向对象设计的坚实框架。遵循这些原则能显著提升代码质量。

想象一下，如果一个函数把自己的整个“栈帧”（包括局部变量等）复制了一份到“堆”上，那么即使这个函数执行完毕了，这份“副本”依然会保留在内存中。

“这样函数定义的本地变量就可以在函数返回之后继续存在”

这是第一步的直接结果。原本函数返回后就消失的局部变量，因为其所在的环境（栈帧）被移到了堆上，所以它们得以幸存。这些“幸存下来的局部变量”，就变成了 “对象的成员变量”。

“这个函数就成为了一个类的构造函数”

现在，我们有一个特殊的函数：

  • 它的职责不再是执行计算然后返回，而是在堆上构建一个“数据实体”（即对象）并返回其引用（或指针）。
  • 这个函数就是 “构造函数”。

“构造函数定义的嵌套函数就成为了成员方法”

在函数内部定义的函数，叫做“嵌套函数”或“闭包”。当这个“构造函数”创建了一个对象后，这些嵌套函数需要能够继续访问和操作那些“幸存下来”的成员变量。

这些能够操作特定对象内部数据的函数，就变成了 “成员方法”。在成员方法内部，this或self指针本质上就是指向那个保存在堆上的、“被固化了的栈帧”（即对象本身）的引用。
