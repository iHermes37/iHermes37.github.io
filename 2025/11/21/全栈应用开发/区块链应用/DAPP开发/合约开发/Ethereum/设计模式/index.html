设计模式

模式类别 模式名称 核心目的 关键机制/思想 典型应用场景
可升级性与代理 代理模式 使已部署的合约逻辑可以升级，同时保留状态。 使用 delegatecall 将执行逻辑与数据存储分离。代理合约持有状态和逻辑合约地址。 需要修复Bug或迭代功能的DApp、复杂DeFi协议。
透明代理 解决代理与逻辑合约间的函数选择器冲突。 基于调用者身份：管理员调用在代理执行，用户调用被转发。 OpenZeppelin升级库的标准模式，适用于大多数可升级合约。
UUPS代理 更Gas高效的代理模式，将升级逻辑放在逻辑合约中。 升级函数是逻辑合约的一部分，代理合约更轻量。 对Gas成本敏感的项目，逻辑合约本身需要处理升级。
访问控制 权限控制 限制特定函数只能被授权地址调用。 使用修饰器（如 onlyOwner, hasRole）进行函数前置检查。 所有者功能（如提款、暂停）、管理员操作、角色化访问（铸造、升级）。
紧急停止 在发现漏洞或攻击时，快速暂停合约关键功能。 设置一个 paused 布尔标志，在函数开始时检查。 任何需要风险控制的合约，尤其是在项目初期。
生命周期与状态 状态机 明确管理合约业务逻辑的阶段性状态流转。 定义一组状态（如 Pending, Active, Completed），并通过函数在不同状态间转换。 ICO（募资中/已完成）、投票、游戏关卡、跨链交易状态。
资金管理 提款模式 让用户主动提取属于他们的资金，避免主动发送的高Gas成本和风险。 将用户的余额记录在映射中，用户调用 withdraw 函数来领取。 空投、收益奖励分配、彩票奖金、退款。
速率限制 控制在一定时间内资金流出的总量，降低漏洞影响。 记录时间段内的流出总量，超过限额则拒绝交易。 国库管理、Vault合约，作为安全防护层。
Gas优化 批量操作 降低用户执行重复操作的总体Gas成本。 允许用户在一次交易中执行多个操作（如批量转账、批量授权）。 NFT Mint、空投、DeFi组合操作。
索引模式 使前端应用能高效地过滤和查询合约事件。 在事件（Event）中包含索引参数（indexed），最多3个。 需要按特定地址、Token ID等快速查询历史记录的DApp。
依赖与集成 依赖注入 降低合约间的耦合度，便于测试和升级依赖组件。 将关键依赖合约的地址保存在一个状态变量中，允许更改。 价格预言机、随机数生成器、治理合约等可更换组件的集成。
合约注册表 集中管理系统中所有合约的地址，便于查找和升级。 一个主合约充当注册表，通过名称映射到最新合约地址。 复杂的多合约DApp系统，其中合约间需要相互寻址。
创建与初始化 工厂模式 批量创建和部署相同逻辑的新合约。 一个工厂合约使用 new 关键字或CREATE2操作码来部署多个子合约。 部署多个NFT集合、创建新的交易对、创建个人钱包代理。
初始化函数 在代理模式或工厂模式中，替代构造函数来初始化合约状态。 一个可被外部调用的函数（如 initialize），通常带有初始化保护。 可升级合约的初始设置，工厂创建合约后的参数配置。

代理模式应运而生，它允许我们将合约逻辑与数据存储分离，实现合约的可升级性。

为什么需要这个手动步骤？

问题所在：

  • 逻辑合约的构造函数在逻辑合约部署时运行
  • 但代理模式中，存储属于代理合约
  • 所以逻辑合约构造函数设置的初始值都存储在逻辑合约自己的存储中，而不是代理合约的存储中

解决方案：

  • 用 initialize 函数替代构造函数
  • 在代理部署后手动调用一次
  • 这样初始化操作就在代理合约的存储中执行

Calldata 的结构：

汇编指令的作用：

  • calldatacopy(0, 0, calldatasize())

    • 将调用数据从calldata复制到内存
    • 0, 0：目标内存位置和源calldata位置
    • calldatasize()：要复制的数据大小

  • delegatecall(gas(), impl, 0, calldatasize(), 0, 0)

    • gas()：转发所有可用gas
    • impl：逻辑合约地址
    • 0：输入数据在内存中的位置
    • calldatasize()：输入数据大小
    • 0, 0：输出数据位置和大小（暂时为0）

总结

透明代理完美地处理了参数传递：

  1. 自动传递：所有调用参数通过 calldatacopy 和 delegatecall 自动转发
  2. 完整保留：参数类型、顺序、值都完全保持不变
  3. 支持复杂类型：包括数组、结构体、嵌套数据等
  4. ETH传递：配合 payable 可以同时传递以太币
  5. 返回值处理：逻辑合约的返回值通过 returndatacopy 返回给原始调用者

用户完全不需要关心代理机制，就像直接调用逻辑合约一样使用所有功能，这正是”透明”代理名称的由来。

虽然通过delegatecall的方式将外部对代理合约的调用全部转接到远程合约上，省去了部署一次合约的开销，但是它存在以下问题：

  • 代理合约只拷贝了远程合约的runtime code，由于涉及初始化部分的代码在init code中，故代理合约无法拷贝远程合约的构造函数内的内容，需要一个额外的initialize 函数来初始化代理合约的状态值。
  • delegatecall只能调用public 或者 external的方法，对于其internal 和 private 方法无法调用。所以代理合约相当于只拷贝了远程合约的公开的方法。

代理让合约保持一个公共地址，同时在幕后交换逻辑。透明代理 (EIP-1967) 通过 delegatecall 将用户调用路由到一个实现，将所有状态保存在代理中，并公开一个单独的管理平面用于升级。你展示了三个标准插槽（实现/管理/信标），为什么构造函数不起作用（使用 initialize），以及一个完整的 Foundry 演练：部署 V1，使用 init calldata 部署代理，通过代理读取/写入，检查插槽，然后升级到 V2 并确认状态连续性。

在智能合约架构中，”上下文构建”是一个重要的设计概念，它指的是在合约执行过程中，有意识地构建和管理执行上下文环境，确保状态的一致性、安全性和可预测性。

核心概念

上下文构建关注的是：在哪个环境中、以什么身份、在什么状态下执行合约逻辑。

具体表现形式

1. 执行上下文明确化

2. 权限上下文构建

3. 交易上下文构建

实际应用场景

场景1：DeFi协议中的上下文构建

场景2：治理合约中的投票上下文

上下文构建的优势

1. 提高可测试性

2. 增强安全性

3. 改善代码可维护性

总结

智能合约中的上下文构建本质上是：

  1. 显式化执行环境 - 将隐式的全局变量转化为显式的上下文对象
  2. 统一验证入口 - 在明确的上下文中进行集中验证
  3. 隔离业务逻辑 - 业务逻辑在受控的上下文中执行
  4. 增强可测试性 - 可以轻松模拟各种执行场景
  5. 提高安全性 - 在统一的上下文中进行安全检查

这种模式使得智能合约更加结构化、可预测和可维护，特别适合复杂的DeFi协议和治理系统。

核心类比：智能合约 ≈ 云端API服务
