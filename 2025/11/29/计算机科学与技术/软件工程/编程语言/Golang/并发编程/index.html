消息传递(Channel)

channel的内部实质就是一个缓冲池+两个协程队列（send recv）如果 Channel 是有缓冲的，缓冲区（buf）的内存也会在堆上分配。

原理

分配内存

Channel 是一个引用类型。在 Go 中，Channel 是一个引用类型，它的底层数据结构（hchan）是在堆上分配的。当创建一个 Channel 时，Go 运行时会调用 mallocgc 函数在堆上分配内存。

  • Channel 是 Goroutine 之间通信的核心机制，需要在多个 Goroutine 之间共享。
  • 堆上的内存可以被所有 Goroutine 访问，而栈上的内存是 Goroutine 私有的。

注意这里的设计思想：

  • 在 Channel 的发送和接收操作中，会先尝试快速路径，即在不加锁的情况下检查是否可以立即完成操作。
  • Channel 的发送队列（sendq）和接收队列（recvq）是基于无锁队列实现的。这些队列使用原子操作（如 CAS）来管理 Goroutine 的入队和出队，避免了锁的开销。
  • Channel 使用条件变量（如 gopark 和 goready）来实现 Goroutine 的挂起和唤醒。这些机制避免了显式的锁竞争，提高了并发性能。
阻塞与非阻塞

nil Channel：向 nil Channel 发送或接收数据会永久阻塞（除非是非阻塞模式）。【nil Channel 是一个未初始化的 Channel 变量，其值为 nil。比如：var ch chan int // ch 是一个 nil Channel】

在 Go 语言中，阻塞模式和非阻塞模式是指 Goroutine 在执行 Channel 操作时的行为方式。这两种模式的区别在于，当 Channel 操作不能立即完成时，Goroutine 是否会挂起等待。

在阻塞模式下，如果 Channel 操作不能立即完成（例如向无缓冲 Channel 发送数据时没有接收方，或向有缓冲 Channel 发送数据时缓冲区已满），当前 Goroutine 会被挂起，直到操作可以继续。Goroutine 会等待，直到 Channel 操作完成。这是 Go 中 Channel 的默认行为。

  • 如果 ch 是无缓冲 Channel，ch <- 1 会阻塞，直到有其他 Goroutine 执行 <-ch。

  • 如果 ch 是有缓冲 Channel，ch <- 1 只有在缓冲区满时才会阻塞。

在非阻塞模式下，如果 Channel 操作不能立即完成，Goroutine 不会挂起，而是直接返回失败（例如返回 false）。

  • Goroutine 不会等待，而是立即返回。
  • 非阻塞模式通常通过 select 语句的 default 分支实现。

如果 ch 的缓冲区已满，ch <- 1 会立即执行 default 分支。如果 ch 的缓冲区为空，<-ch 会立即执行 default 分支。

接下来，是selectgo()函数的主处理逻辑，它会分三个阶段查找或等待某个channel准备就绪：

  1. 查找是否有可以立即收发的 channel。
  2. 如果没有，将当前 goroutine 加入所有 case 的 channel 的等待队列并挂起。
  3. 被唤醒后，找到被选中的 case 并执行相应的操作。

需要说明的是，runtime.selectgo 函数会根据不同情况通过 goto 语句跳转到函数内部的不同标签执行相应的逻辑。其中包括：bufrecv：可以从channel缓冲区读取数据；bufsend：可以向channel缓冲区写入数据；recv：可以从休眠的发送方获取数据；send：可以向休眠的接收方发送数据；rclose：可以从关闭的 channel 读取 EOF；sclose：向关闭的 channel 发送数据；retc：结束调用并返回；

第一阶段

按照 pollorder 的顺序遍历所有 case，检查是否有 channel 可以立即执行发送或接收操作。

  • 如果某个 channel 的 sendq 或 recvq 上有等待的 goroutine，或者缓冲区有数据（接收）或空闲空间（发送），则立即执行相应的操作。
  • 如果找到可以立即执行的 case，跳转到对应的标签（如 recv、send、bufrecv、bufsend）进行处理。
第二阶段

如果没有 channel 可以立即执行操作，将当前 goroutine 加入所有 case 对应的 channel 的发送队列（sendq）或接收队列（recvq），并挂起当前 goroutine，等待被唤醒。

  • 为每个 case 创建一个 sudog，并将其加入对应 channel 的等待队列。
  • 调用 gopark 挂起当前 goroutine，等待其他 goroutine 的唤醒。

第三阶段

当前 goroutine 被唤醒后，找到被选中的 case，并执行相应的操作。遍历所有 case，找到被选中的 case（即唤醒当前 goroutine 的 channel）。根据 case 的类型（发送或接收），执行相应的操作。清理未选中的 case，释放资源。

selectgo函数的执行分为四个步骤：首先，随机生成一个遍历case的轮询顺序 pollorder 并根据 channel 地址生成加锁顺序 lockorder，随机顺序能够避免channel饥饿，保证公平性，加锁顺序能够避免死锁和重复加锁；然后，根据 pollorder 的顺序查找 scases 是否有可以立即收发的channel，如果有则获取case索引进行处理；再次，如果pollorder顺序上没有可以直接处理的case，则将当前 goroutine 加入各 case 的 channel 对应的收发队列上并等待其他 goroutine 的唤醒；最后，当调度器唤醒当前 goroutine 时，会再次按照 lockorder 遍历所有的case，从中查找需要被处理的case索引进行读写处理，同时从所有case的发送接收队列中移除掉当前goroutine。

互斥锁的结构

Sync.Mutex由两个字段构成，state用来表示当前互斥锁处于的状态，sema用于控制锁状态的信号量。相信各位道友读完这两个字段的描述后，好像懂了，又好像没懂。下面我们详细理解下这两个字段到底都作了哪些事。

互斥锁state主要记录了如下四种状态：

  • waiter_num： 记录了当前等待抢这个锁的goroutine数量
  • starving： 当前锁是否处于饥饿状态 (后文会详解锁的饥饿状态) 0: 正常状态 1: 饥饿状态
  • woken： 当前锁是否有goroutine已被唤醒。 0：没有goroutine被唤醒； 1: 有goroutine正在加锁过程
  • locked： 当前锁是否被goroutine持有。 0: 未被持有 1: 已被持有

sema信号量的作用

当持有锁的gorouine释放锁后，会释放sema信号量，这个信号量会唤醒之前抢锁阻塞的gorouine来获取锁。

锁的两种模式

Go 语言中的 互斥锁（Mutex） 是用于保护共享资源的重要工具。为了在高并发场景下提高锁的公平性和性能，Go 的互斥锁设计了两种模式：正常模式 和 饥饿模式。这两种模式的主要目的是在保证性能的同时，尽可能避免某些 Goroutine 长时间无法获取锁（即“饥饿”问题）。

正常模式

在正常模式下，所有阻塞在等待队列中的 Goroutine 会按顺序进行锁获取。当唤醒一个等待队列中的 Goroutine 时，此 Goroutine 并不会直接获取到锁，而是会和新请求锁的 Goroutine 竞争。通常，新请求锁的 Goroutine 更容易获取锁，因为它正在占用 CPU 时间片执行，大概率可以直接执行到获取锁的逻辑。

优点：性能较高，因为新请求锁的 Goroutine 可以直接参与竞争，减少了上下文切换的开销。适合低竞争的场景。

缺点：可能导致某些 Goroutine 长时间无法获取锁（即“饥饿”问题）。

饥饿模式

在饥饿模式下，新请求锁的 Goroutine 不会直接参与锁的竞争，而是会被加入到等待队列的尾部，阻塞等待获取锁。只有等待队列中的 Goroutine 才能获取锁，确保了锁的公平性。

触发条件：当一个 Goroutine 等待锁的时间超过 1ms 时，互斥锁会切换到饥饿模式。

取消条件：当获取到锁的 Goroutine 是等待队列中的最后一个 Goroutine 时，互斥锁会切换到正常模式。当获取到锁的 Goroutine 的等待时间在 1ms 之内时，互斥锁会切换到正常模式。

优点：避免了某些 Goroutine 长时间无法获取锁的问题，提高了锁的公平性。适合高竞争的场景。

缺点：性能较低，因为新请求锁的 Goroutine 必须加入等待队列，增加了上下文切换的开销。

两种模式的切换

  • 正常模式 → 饥饿模式：当一个 Goroutine 等待锁的时间超过 1ms 时，互斥锁会切换到饥饿模式。
  • 饥饿模式 → 正常模式：当获取到锁的 Goroutine 是等待队列中的最后一个 Goroutine 时，互斥锁会切换到正常模式。当获取到锁的 Goroutine 的等待时间在 1ms 之内时，互斥锁会切换到正常模式。

使用 Mutext 锁虽然很简单，但仍然有几点需要注意：

  • 同一协程里，不要在尚未解锁时再次使加锁
  • 同一协程里，不要对已解锁的锁再次解锁
  • 加了锁后，别忘了解锁，必要时使用 defer 语句

RWMutex

读写锁的结构：

读操作如何防止并发读写问题的？

  • RLock(): 申请读锁，每次执行此函数后，会对readerCount++，此时当有写操作执行Lock()时会判断readerCount>0,就会阻塞。
  • RUnLock(): 解除读锁，执行readerCount–，释放信号量唤醒等待写操作的goroutine。

写操作如何防止并发读写、写写问题？

  • Lock(): 申请写锁，获取互斥锁，此时会阻塞其他的写操作。并将readerCount 置为 -1，当有读操作进来，发现readerCount = -1， 即知道有写操作在进行，阻塞。
  • Unlock(): 解除写锁，会先通知所有阻塞的读操作goroutine，然后才会释放持有的互斥锁。

为什么写操作不会被饿死？

写操作要等待读操作结束后才可以获得锁，而写操作在等待期间可能还有新的读操作持续到来，如果写操作等待所有读操作结束，很可能会一直阻塞，这种现象称之为写操作被饿死。

通过RWMutex结构体中的readerWait属性可完美解决这个问题。当写操作到来时，会把RWMutex.readerCount值拷贝到RWMutex.readerWait中，用于标记排在写操作前面的读者个数。前面的读操作结束后，除了会递减RWMutex.readerCount，还会递减RWMutex.readerWait值，当RWMutex.readerWait值变为0时唤醒写操作。

  • 读操作：通过 RLock() 和 RUnlock() 实现并发读操作。读操作不会阻塞其他读操作，但会阻塞写操作。

  • 写操作：通过 Lock() 和 Unlock() 实现独占写操作。写操作会阻塞其他读写操作。

  • 写操作不被饿死：通过 readerWait 属性，确保写操作在等待期间不会被新的读操作无限阻塞。
