为什么需要?

非结构化数据处理流程

Milvus 为解决非结构化数据的检索问题而生：海量的非结构化数据一般会存储在分布式文件系统或对象存储上，之后通过深度学习网络完成推理，将这些非结构数据转化成 embedding 向量，并在向量空间内完成近似性检索，从而发现数据背后的一些特征。

比如，有很多原始的食物图片，通过卷积神经网络做训练和推理，为每一幅照片得出一组向量，再把这些向量按照空间中的近似维度做排序，最后得到这样的结果：最上面一排是一些长得像薯条的东西，中间都是一些长得像拉面的东西，底下都是长得像寿司的东西。也就是说，图片这种非结构化数据，经过深度学习处理之后，转化成了embedding 向量，并通过在向量空间的近似度比对来表征其相似性，这在很大程度上能跟人类理解的近似度是高度一致的。

向量与标量

传统的标量数据和 Milvus 面向的向量数据之间，到底有哪些不同呢？

从基本操作上来讲，对于标量数据，针对数值类数据一般会做加减乘除的操作；对字符串类型的数据一般会做一些 term 的匹配， 或者一些类似 like 的近似匹配，抑或一些前缀匹配。

而针对向量数据而言，很少进行这种 100% 的完全匹配，更多是看近似度，也就是高维空间下的距离。较常见的距离表示有余弦距离、欧式距离等。空间中向量之间的距离，很大程度上能表示非结构化数据之间的相似度。

除了对数据的操作会有很大不同以外，数据的组织方式也会有很大不同。如，传统数据很容易比较大小，无论是数值类，还是字符串，都可以通过二叉树或者 skip list 的方式排列组合，然后做二分查找。对于向量数据来讲，则更加复杂，因为它维度较高，很难像传统的数值类数据一样通过排序的方式做加速，往往需要一些特殊的索引结构和存储方式。

Milvus 的核心就是——高性能存储 + 高维相似搜索。可以把它理解成：

“一个能对向量空间做 SQL 操作的数据库”，
能在亿级语义嵌入中高效查找最相似内容，是构建 RAG、推荐系统、语义检索、图像搜索 的底层引擎。

运行流程

向量数据库的运行基于以下关键步骤：

  1. 嵌入生成：使用预训练模型（如 BERT、CLIP）将非结构化数据转换为高维向量（例如 768 维或 1536 维）。这些向量捕捉了数据的语义信息。
  2. 向量索引：为了加速相似性搜索，向量数据库使用专门的索引结构（如 HNSW、IVF、DiskANN）组织向量数据，减少搜索时的计算量。
  3. 相似性搜索：通过距离度量（如欧几里得距离、余弦相似度）计算查询向量与数据库中向量的相似度，返回 Top-K 结果。
  4. 元数据过滤：支持结合元数据（如时间、类别）进行过滤，提升搜索精准度。
  5. 实时更新：支持动态插入、更新和删除向量，保持数据新鲜。

结构

基础

在 Milvus 上，可以创建多个 Collections 来管理数据，并将数据作为实体插入到 Collections 中。Collections 和实体类似于关系数据库中的表和记录。

Schema 定义了 Collections 的数据结构。在创建一个 Collection 之前，需要设计出它的 Schema。其是 Collection 的蓝图，它严格定义了 Collection 的结构，包括：

  • **Fields (字段)**： 定义每个实体包含哪些数据。
  • **Primary Key (主键)**： 指定唯一标识一个实体的字段。
  • **Vector Field (向量字段)**： 指定存储向量的字段及其维度。

一个 Entity 就是一条完整的数据记录，是 Fields 的具体值的集合。它完全对应于关系型数据库中的一行 Row 或一条 Record。

一个 Collection 可以包含多个 Partitions，但一个 Partition 必须属于且仅属于一个 Collection。

  • Collection 就像一个完整的图书馆。
  • Partition 就像是图书馆里的特定分区书架，比如“计算机科学区”、“文学区”、“历史区”。
特性 Collection Partition
概念层级 顶级数据组织单位 次级数据组织单位，隶属于 Collection
类比 数据库中的 Table 数据库表中的 Partition

必需性 必须的。插入数据前必须先创建 Collection。 可选的。每个 Collection 都有一个名为 _default 的默认分区。

索引

如果没有索引（即使用 FLAT），要进行最近邻搜索，就必须计算查询向量与集合中每一个向量的距离，然后排序。这在数据量巨大时（例如上亿条），计算成本是无法接受的。

因此，所有向量索引的基本思想都是：通过预先计算和组织数据，将搜索范围从“整个数据库”缩小到“最有可能的几个候选区域”，用微小的精度损失换取巨大的速度提升。这就是近似最近邻搜索。

索引是建立在数据之上的附加结构。其内部结构取决于所使用的近似近邻搜索算法。索引可以加快搜索速度，但在搜索过程中会产生额外的预处理时间、空间和 RAM。此外，使用索引通常会降低召回率（虽然影响可以忽略不计，但仍然很重要）。

为特定字段创建索引可提高搜索效率。建议为服务所依赖的所有字段创建索引，其中必须为向量字段创建索引。

搜索

ANN 搜索

这是 Milvus 最核心的功能。给定一个查询向量，在庞大的向量库中快速找出与它最相似的 K 个向量。使用向量索引（如 HNSW, IVF）进行近似最近邻搜索，在精度和速度之间取得平衡。关键参数：limit (即 K值，返回前K个结果)

  • 示例场景：
    • 以图搜图：上传一张猫的图片，找出图库中所有猫的图片。
    • 语义文本搜索：输入“快乐的节日”，找出所有表达喜庆、欢乐情感的文本段落。

过滤搜索

在执行 ANN 搜索之前或之后，应用基于标量元数据的过滤条件。先根据标量条件（如 author == "John"）筛选出一个子集，然后在这个子集上进行向量搜索，或者先做向量搜索再过滤结果。这极大地提高了搜索的精确性和实用性。关键语法：使用布尔表达式（如 >, <, ==, and, or）。

  • 示例场景：
    • 电商：“在我的价格范围 100-200 元内，找出与这个手提包最相似的商品。”
    • 内容管理：“在2023年之后发布的科技类文章中，找出与‘人工智能伦理’语义相近的文章。”

范围搜索

又称半径搜索。查找与查询向量距离在指定半径范围内的所有向量。不限制返回数量（K），而是限制相似度的“距离”。距离小于半径 radius 的向量都会被返回。关键参数：radius (距离半径), range_filter (可选，用于对结果进行二次过滤)

  • 示例场景：
    • 人脸门禁：“在员工库中，找出所有与当前摄像头人脸相似度**高于 95%**（即距离非常小）的人。” 确保只有极高相似度的匹配才被允许通过。
    • 抄袭检测：“找出所有与待检测文档相似度超过一定阈值的潜在源文档。”

混合搜索

基于多个向量字段进行联合搜索，并将结果合并。一个实体（记录）可能包含多个向量字段（如“图片向量”和“文本描述向量”）。混合搜索允许您同时对这两个字段进行查询，并通过加权平均等方式合并两个搜索结果的分数。

  • 示例场景：
    • 多模态搜索：输入一段文本“一只在沙滩上奔跑的金毛犬”，系统可以同时搜索“图片向量”和“图片的文本描述向量”，返回最相关的结果。
    • 产品搜索：同时匹配产品的“图像特征向量”和“用户评论情感分析向量”。

全文搜索

基于经典的 BM25 算法对文本字段进行关键词搜索。BM25 是一种用于评估文档与搜索查询相关性的排名函数。它处理的是标量文本字段，而不是向量。

  • 示例场景：
    • 文档检索：在文档库中，直接搜索包含关键词“机器学习”、“框架”的文档。
    • 与向量搜索结合：先通过全文搜索快速筛选出包含关键字的文档，再对这些文档进行更精确的语义向量搜索。

Rerankers

对 ANN 搜索返回的初步结果进行重排序，以优化最终排名。ANN 索引为了速度牺牲了一些精度。Reranker 作为一个更精细、可能也更耗时的模型，对 Top K 的结果进行二次计算，得到更准确的相似度分数并重新排序。

  • 示例场景：
    • 交叉编码器：在向量搜索初步找出 100 个相关文档后，使用一个更强大的、但运行慢的交叉编码器模型对这 100 个文档进行精排，提升前 10 个结果的准确性。
    • 多因素排序：在电商搜索中，将“向量相似度”、“销量”、“用户评分”、“上新时间”等因素综合起来，对初始的相似商品列表进行重新排序。

获取

根据已知的主键值，直接检索出对应的完整实体数据。类似于键值数据库的 get 操作或 SQL 中的 SELECT * FROM table WHERE id = ?。它不涉及任何向量计算或相似性比较。

  • 示例场景：
    • 详情页查询：在向量搜索返回了一组商品的 ID 后，前端需要获取这些商品的完整信息（如名称、描述、价格）来展示，这时就使用“获取”操作。
    • 缓存穿透：直接通过 ID 拉取数据。

查询

使用表达式基于标量字段来检索数据，不涉及向量相似性。类似于 SQL 中的 WHERE 子句查询。它用于按元数据过滤和查找数据。

  • 示例场景：
    • “找出所有作者是‘Alice’且状态为‘已发布’的文章。”
    • “统计分类为‘电子产品’且创建时间在 2024 年 1 月之后的所有记录数量。”
  • 与“过滤搜索”的区别：
    • 查询：目标是获取符合条件的数据本身，不进行向量相似度排序。
    • 过滤搜索：目标是进行向量搜索，过滤只是其前置或后置步骤，最终结果按向量相似度排序。
