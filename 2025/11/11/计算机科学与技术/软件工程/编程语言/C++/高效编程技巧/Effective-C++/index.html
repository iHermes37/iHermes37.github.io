C++编译器自动生成的函数

关键要点：

  1. 了解默认生成规则避免意外行为
  2. 资源管理类通常需要自定义这些函数
  3. 使用= default和= delete让意图更明确
  4. 移动操作在C++11中很重要，但生成条件更严格

别让异常逃离析构函数

在析构函数中，如果有可能抛出异常，必须要在析构函数内部捕获并处理它，绝不能让它传播到析构函数外部。

方案 做法 适用场景 优点 缺点
吞掉异常 try-catch在析构函数内部 非关键操作 简单，保证程序继续 可能隐藏重要错误
显式释放方法 提供close()等方法 需要错误处理的资源 用户可处理异常，更灵活 依赖用户正确调用
终止程序 调用std::abort() 关键资源失败意味着程序状态不可信 明确失败，避免数据损坏 用户体验差
自定义删除器 与智能指针结合 现代C++资源管理 类型安全，自动管理 稍复杂

核心原则：

  1. 析构函数绝不能让异常传播到外部
  2. 在栈展开期间（处理异常时），析构函数抛异常会导致std::terminate()
  3. 优先提供显式释放方法，让用户有机会处理异常
  4. 析构函数中必须捕获所有可能的异常
  5. 使用noexcept标记析构函数，明确承诺不抛异常

记住：析构函数应该专注于资源释放，而不是错误处理。把错误处理的机会留给调用者。

复制对象时勿忘记其每一个成分

“复制对象时勿忘记其每一个成分” 指的是：当你自定义拷贝构造函数或拷贝赋值运算符时，必须确保复制所有的数据成员，包括基类部分和新增的成员变量。

当我们自定义拷贝操作时，编译器不再自动生成默认的拷贝行为，需要我们手动管理所有成员的复制。如果遗漏了某些成员，就会导致部分拷贝的bug。

当自定义拷贝操作时，问自己：

场景 错误做法 正确做法
新增成员变量 忘记在新成员复制 更新所有拷贝操作，复制新成员
继承体系 只复制派生类成员 调用基类拷贝操作Base(other)或Base::operator=(other)
动态资源 浅拷贝指针 深拷贝或使用智能指针
维护不变量 只复制部分相关成员 复制所有维持不变性需要的成员

检查清单：

  1. 是否复制了所有非静态数据成员？
  2. 在继承体系中，是否正确处理了基类部分？
  3. 是否检查了自赋值情况？（拷贝赋值中）
  4. 异常安全是否得到保证？
  5. 是否真的需要自定义拷贝操作？能否用= default？

要么完全不自定义拷贝操作（让编译器生成），要么确保完全正确地复制所有成分。 任何部分拷贝都会导致对象状态不一致，这是很难调试的bug来源。记住：编译器生成的拷贝操作总是正确的，因为它知道类的完整结构。当你接管这个责任时，你必须同样正确。

继承和面对对象设计

区分接口继承和实现继承

  • 接口继承：定义”做什么“（What）的契约，关注行为规范，不关心具体实现，在C++中通过纯虚函数实现
  • 实现继承：提供”怎么做“（How）的具体代码，关注代码复用和具体实现，在C++中通过非虚函数和普通虚函数实现

is-a 和 has-a

is-a 关系（继承）表示”是一个“的关系，用于分类和特化，在C++中通过公有继承实现，is-a 关系描述的是类型之间的层次关系，派生类是基类的一种特殊形式。

has-a 关系（组合）表示”有一个“的关系，用于包含和组合，在C++中通过成员变量实现，has-a 关系描述的是整体与部分的关系，一个对象包含另一个对象作为其组成部分。
