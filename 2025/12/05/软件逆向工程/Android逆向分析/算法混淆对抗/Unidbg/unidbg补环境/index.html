unidbg补环境

为什么需要补环境

Unidbg 是一个基于 Unicorn 的 Android Native（SO）模拟执行框架，它通过 Hook 和补环境 的方式让 SO 库在模拟环境中正常运行。由于 Unicorn 只是一个纯 CPU 指令模拟器（支持 ARM/ARM64/x86/x64 等架构），它本身 不提供任何系统调用、外部库函数或 JNI 环境，因此 Unidbg 需要手动实现这些外部依赖。

Unidbg的作用是模拟执行so中的函数,处于Native层。Java层的函数可以通过JNI调用Native层函数,那么同理Native层也可以通过JNI去调用Java层的函数。在Native层调用Java层函数的时候,因为Unidbg中没有这些函数的实现,模拟执行必然失败.故我们需要在Unidbg中补充这些Java层函数,让Native层的函数去调用.

所以unidbg需要在unicorn模拟运行时补充上述函数

虽然unidbg 底层使用unicorn进行二进制汇编模拟执行，但android native so库编写处理native自定义函数 还可以引用的底层库函数（系统调用），比如：

  • 标准 C/C++ 库函数：如 libc、libm 中的函数（malloc、printf、sin 等）。C++ STL 或第三方 C/C++ 库（需确保 NDK 支持或静态链接）。

  • NDK 提供的原生 API

    Android NDK APIs：

    • liblog（__android_log_print 打印日志）。
    • libandroid（访问 Android 特定功能，如 AssetManager）。
    • libEGL/libOpenGLES（图形渲染）。
    • libmediandk（多媒体处理）。
    • libvulkan（Vulkan 图形 API）。

  • 其他 SO 库中的函数：动态链接其他 SO 库（通过 dlopen/dlsym 或 NDK 的 SharedLibrary 加载）。静态链接的第三方库（如 OpenCV、FFmpeg 等）。

  • JNI 环境提供的函数：JNIEnv 接口中的方法（如 FindClass、GetMethodID、CallObjectMethod 等），用于与 Java 交互。

  • 系统调用（Syscall）：直接调用 Linux 系统调用（如 open、read、ioctl 等），需注意权限和兼容性。

  • 内联汇编或编译器内置函数：针对特定 CPU 架构的优化（如 ARM NEON 指令集）。

  • 回调函数（通过函数指针）：Native 层自定义的回调函数，或由 Java 通过 JNI 传递的函数指针。

Unicorn 不能直接调用外部函数（如 malloc、__android_log_print 或 JNI 方法），因为：Unicorn 仅模拟 CPU 指令执行，不模拟操作系统或运行时环境。外部函数通常依赖 系统库（如 libc.so、libandroid.so） 或 操作系统功能（如系统调用）。

Unidbg 通过以下方式实现对外部函数的调用：

  1. 拦截函数地址：当 SO 库调用外部函数（如 malloc）时，Unicorn 会触发 执行中断（因为函数地址不存在）。Unidbg 拦截这些调用，并映射到自定义的 Java/Native 实现。

  2. 补环境（实现缺失函数）：对于标准 C 函数（如 printf）、Android NDK API（如 __android_log_print）或 JNI 方法，Unidbg 需要 手动实现它们的逻辑。实现方式包括：

    • Java 层模拟：用 Java 代码模拟函数行为（如实现 malloc 的内存分配）。
    • Native 桥接：通过 JNI 调用宿主系统的真实函数（如直接调用宿主机器的 sin 函数）。
    • 虚拟返回值：对于不重要的函数，直接返回固定值（如 0）。

  3. 系统调用（Syscall）模拟：Linux 系统调用（如 open、read）需要通过 Unicorn 的 异常处理机制 拦截，并由 Unidbg 模拟实现。例如：当 SO 调用 open("/data/file.txt", O_RDONLY) 时，Unidbg 可以虚拟一个文件内容返回。

同时为了正确运行 SO 代码，必须同时模拟 JNI（Java Native Interface）层，因为 SO 库通常依赖 Java 环境。

  1. Unidbg 主要用于模拟 SO 层算法，但需部分模拟 JNI 以支持 SO 与 Java 的交互。
  2. JNI 模拟是必要的：因为 SO 代码通常依赖 Java 环境（如获取设备信息、调用工具类）。
  3. 不模拟完整 Java 层：按需实现 JNI 函数，兼顾效率和功能。

Unidbg 是 “SO 代码的沙箱”，通过模拟 JNI 让 SO 以为自己运行在真实 Android 中，从而解密其逻辑。

  1. Unicorn 负责基础指令执行，Unidbg 在其上实现 SO 加载和符号解析。
  2. Hook 框架（Dobby/xHook 等）动态修改函数跳转，使 Unidbg 能拦截和替换函数逻辑。

Unicorn 执行到 Hook 点 → 跳转到 Hook 框架 → 框架处理后再返回 Unicorn 继续执行。

Hook机制

Unicorn 只能执行指令，但 无法动态修改函数跳转逻辑。Hook 框架的作用是：动态修改函数行为

Inline Hook（如 Dobby/HookZz）：直接修改memcpy的机器码，插入跳转指令（如B ），劫持控制流。

PLT/GOT Hook（如 Whale/xHook）：修改 SO 的 ​​PLT/GOT 表​​（动态链接表），将 memcpy 的地址替换为 Hook 函数。

Hook 框架在 Unidbg 中运行，但修改的是 Unidbg 维护的 虚拟内存（非真实内存）。Unicorn 执行到被 Hook 的函数时，会 自动跳转到 Hook 框架的代码，而非原函数。

注意Unicorn 是一个 CPU 指令级模拟器，它本身不直接理解 ELF 动态链接库的加载和符号解析。Unidbg 在 Unicorn 之上实现了以下关键逻辑：

SO 库加载与符号解析

Unidbg 模拟 Android 的 dlopen/dlsym，加载目标 SO 文件并解析导出符号（如 memcpy）。符号地址会被记录在 Unidbg 的 虚拟内存映射表 中（如 memcpy → 0x40001000）。

函数调用跳转

当模拟执行到 BLX memcpy（ARM）或 BL memcpy（ARM64）指令时：

  1. Unicorn 遇到跳转指令，检查目标地址（如 0x40001000）。
  2. Unidbg 拦截该地址，检查是否已 Hook：
    • 如果未 Hook：直接调用 Unidbg 实现的 memcpy 模拟函数（或原生函数代理）。
    • 如果已 Hook：跳转到 Hook 框架（如 Dobby）的替换函数。

Unicorn 何时触发 Hook？

  1. CPU 访问未映射内存：SO 库尝试读取/写入/执行未映射的内存地址（如调用外部函数 malloc 的地址）。Unicorn 检测到异常，暂停执行。
  2. Hook 事件回调：Unicorn 调用注册的 EventMemHook.hook() 方法。参数包括：
    • address：访问的目标地址。
    • size：操作大小（如 4 字节读）。
    • value（仅写操作有效）：要写入的值。
  3. 事件处理：Unidbg 通过 callback.hook() 决定如何处理：
    • 补内存映射：动态映射一块内存到 address。
    • 模拟函数调用：如果 address 是外部函数地址，执行替代逻辑。
    • **返回 false**：继续抛出异常（崩溃）。

Hook源码调用逻辑

  • 作用：根据 type 参数注册不同类型的内存异常 Hook。

  • 参数说明：

    • callback：事件触发时的回调函数。
    • type：监听的事件类型（按位掩码，如 UC_HOOK_MEM_READ_UNMAPPED | UC_HOOK_MEM_WRITE_UNMAPPED）。
    • user_data：传递给回调函数的用户数据。
  • 作用：向 Unicorn 注册具体的内存异常 Hook，并封装回调逻辑。
  • 关键步骤：
    1. 注册 Unicorn 原生 Hook：
      通过 unicorn.hook_add_new 绑定事件监听器，当发生内存异常时，Unicorn 会调用内部的 hook 方法。
    2. 回调转发：
      将事件转发给外部的 callback.hook() 处理（由 Unidbg 的上层逻辑实现，如补内存映射或模拟函数调用）。
    3. 绑定 UnHook 对象：
      将 Unicorn 返回的 UnHook 对象封装为 Unidbg 的 UnHook，便于后续管理（如动态卸载 Hook）。

如何补环境

如何补上这些环境？是按照源函数的实现自己从0到实现完全一样的功能吗？

在 Unidbg 或类似模拟环境中“补环境”时，并不需要 完全从 0 实现所有函数的功能，而是根据目标 SO 库的实际需求，选择性地模拟关键行为。以下是具体的策略和实现方法：

核心原则

让被模拟的 SO 库“认为”它运行在真实环境中，而非实现所有函数的完整逻辑。
关键思路：

  • 按需模拟：只实现目标 SO 库实际调用的函数和参数。
  • 行为等价：返回的值或副作用需满足 SO 库的预期，但不需完全复现源码。
  • 分层处理：优先补基础函数（如 malloc、printf），再补高级功能（如 JNI、系统调用）。

是否需要完全实现源码？

不需要：只模拟目标 SO 库实际调用的 参数、返回值和副作用。例如：若 SO 库仅检查 fopen 是否返回 NULL，则直接返回非 NULL 即可，无需实现真实文件操作。例外情况：若函数逻辑直接影响程序流程（如加密算法 MD5），需完全复现或调用等效实现。

分类

Unidbg 中的补环境，大体上可以分成两类，运行环境缺失和上下文缺失，

  • 上下文缺失则是由于样本在运行目标函数前对SO或目标函数做了一些初始化工作，Unidbg中对目标函数单独运行，自然就导致了上下文缺失。上下文缺失相较于运行环境缺失，是一种更加隐蔽的环境缺失。
  • 运行环境缺失。最简单的例子就是目标函数中通过 JNI 调用到了某个自己的 JAVA 方法，Unidbg 会及时报错，给出堆栈以及这个 JAVA 方法的签名，需要我们补上对应的 JAVA 方法，

补 JAVA 环境是补运行环境中主要的一部分，但是，补 JAVA 环境并不是工作的全部。还有哪些环境需要补呢？

  • 文件读写——对linux虚拟文件的读写，对ASSETS资源文件的读写、对app目录下文件的读取，对 Sharedpreference 的读取等等
  • 系统调用具体实现——比如popen函数所涉及的系统调用等
  • 系统库SO，Unidbg并没有，实际上也不可能模拟完整的 Android 系统 SO 环境，有的 SO 所依赖的 SO 比较多，很难调起来，Unidbg 只加载了依赖较少、使用频率高的库函数， libandroid.so、libart.so 和 libjnigraphics.so 等无法完全加载的库被实现为虚拟So模块(VirtualModule），有时候我们需要和它打交道。)。虚拟So模块提供了对部分函数的模拟实现，从而解决了类似 libandroid.so 这类系统库的加载问题。
  • 补不了，打 patch
函数类型 Unidbg 实现方式
Android NDK API Hook + 虚拟实现（如 __android_log_print）
JNI 函数 模拟 JNIEnv + 反射调用（如 FindClass）
系统调用（Syscall） 拦截 SWI/SVC + 模拟（如 open、read）

Unidbg 在大多数情况下是好的方案，在补环境的过程中，对算法细节有了一定的理解，而且用 Unidbg 跑出结果后，分析和还原算法会更快，因为 Unidbg 的 code trace / hook / console debugger 相比较 IDA trace，Frida hook 更快、更稳定、更方便复现。而且环境完全是我们的，有绝对的掌控。

Unidbg 并不是过往某个工具的替代品，而是 IDA、Frida 以及其工具套件的互补品。它和过去的工具结合在一起 1+1>2
