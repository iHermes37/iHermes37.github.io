数据结构

这是一个数据库建模层面的基础总表，我按 MySQL 官方语义 + 工程常用度，给你整理一份“可直接用于设计表结构”的完整表格，并在关键地方点出能举一反三的设计思路。

一、MySQL 支持的数据类型总览（工程视角）

⚠️ 注意：MySQL 不是“数据结构型数据库”
它只支持标量类型 + 少量复合类型（JSON / SET）
没有数组、列表、对象嵌套（那是 ORM / 应用层的事）

1️⃣ 数值类型（Numeric Types）

（1）整数类型

类型 字节 有符号范围 无符号范围 常见用途
TINYINT 1 -128 ~ 127 0 ~ 255 状态位、布尔
SMALLINT 2 -32,768 ~ 32,767 0 ~ 65,535 小计数
MEDIUMINT 3 -8M ~ 8M 0 ~ 16M 很少用
INT 4 -21亿 ~ 21亿 0 ~ 42亿 主键、计数
BIGINT 8 极大 极大 区块高度、ID

👉 工程经验：

  • ID / 区块高度 → BIGINT
  • 状态位 → TINYINT

（2）浮点 / 定点数

类型 字节 特点 使用建议
FLOAT 4 单精度 ❌ 不适合金额
DOUBLE 8 双精度 ❌ 不适合金额
DECIMAL(M,D) 变长 精确 ✅ 金额、余额

👉 Web3 / 量化系统：

余额、价格，一律 DECIMAL，禁止 FLOAT/DOUBLE

2️⃣ 字符串类型（String Types）

（1）定长 / 变长字符串

类型 最大长度 特点 常见用途
CHAR(n) 255 定长 hash、固定编码
VARCHAR(n) 65,535 变长 地址、名称

👉 ETH 地址：

（2）文本类型

类型 最大长度 适合存
TINYTEXT 255 短文本
TEXT 64KB 备注、描述
MEDIUMTEXT 16MB JSON、大文本
LONGTEXT 4GB 极少使用

（3）二进制类型

类型 用途
BINARY / VARBINARY 二进制 hash
BLOB / MEDIUMBLOB 二进制大对象

👉 区块链 hash：

  • **VARCHAR(66)**（十六进制）
  • 或 **BINARY(32)**（更省空间）

3️⃣ 日期与时间类型（Temporal Types）

类型 精度 说明
DATE 天 日期
TIME 秒 时间
DATETIME 秒 无时区
TIMESTAMP 秒 有时区
YEAR 年 很少用

👉 工程建议：

  • 业务时间 → DATETIME
  • 日志 / 时间线 → TIMESTAMP

4️⃣ 枚举 & 集合类型（特殊）

（1）ENUM（强烈不推荐）

❌ 问题：

  • 改值要改表
  • 不利于版本升级

👉 用 TINYINT + 业务枚举映射

（2）SET（极少使用）

❌ 查询困难
❌ 设计反范式
👉 基本不用

5️⃣ JSON 类型（MySQL 5.7+）

类型 说明
JSON 原生 JSON 存储

✔️ 可存结构
❌ 不是真正的“数组字段”
❌ 不适合高频查询

👉 适合：配置 / 快照 / 非核心数据

6️⃣ 空间数据类型（GIS，基本不用）

类型
POINT
LINESTRING
POLYGON
GEOMETRY

👉 除非做地图，否则可以忽略

二、重要：MySQL 不支持 的“数据结构”

❌ 以下 全部不支持：

类型 说明
ARRAY 无数组
LIST 无列表
MAP 无键值对
STRUCT 无结构体
NESTED 无嵌套

👉 这些只能通过：

  • 拆表
  • JSON
  • 应用层结构
