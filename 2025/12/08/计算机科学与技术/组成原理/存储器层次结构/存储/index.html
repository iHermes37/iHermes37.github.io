存储

高速缓存

什么是CPU缓存技术

CPU 和内存访问性能的差距非常大。如今，一次内存的访问，大约需要 120 个 CPU Cycle。这也意味着，在今天，CPU 和内存的访问速度已经有了 120 倍的差距。

  • CPU：按照摩尔定律，CPU 的访问速度每 18 个月便会翻一番，相当于每年增长 60%。比如我的笔记本是 Intel Core-i5-8250U 1.6GHz，也就是每秒可以访问 16 亿（= 1.6G）次。
  • 内存：每年只增长 7% 左右。内存响应时间大概是 100us，也就是极限情况下，大概每秒可以访问 1000 万（= 1s / 100ns）次。
  • HDD：磁盘寻道时间约 10ms，大概每秒可以访问 100 次。

为了弥补两者之间的性能差异，充分利用 CPU，现代 CPU 中引入了高速缓存(CPU Cache)。高速缓存分为 L1/L2/L3 Cache，不是一个单纯的、概念上的缓存（比如使用内存作为硬盘的缓存），而是指特定的由 SRAM 组成的物理芯片。

技术发展历史

第一阶段：L2位于芯片外

L1 缓存是一级缓存，速度最快、容量最小、造价最高的缓存。，物理上集成在每个CPU核心内部，与核心同频或接近同频工作，访问延迟通常在 1-3 个时钟周期。是CPU获取数据的“第一站”。容量上通常很小，每核心的L1缓存大小在 几十KB（例如32KB或64KB）。通常分为L1指令缓存和L1数据缓存，分别用于缓存即将执行的指令和需要处理的数据。这种分离可以避免取指令和存取数据的冲突，提升效率。

L2缓存：早期由于芯片制造工艺和成本限制，大容量的SRAM无法与CPU核心集成在一起。因此，L2缓存是一颗独立的SRAM芯片，被放置在CPU封装外部的主板上（或者后期在CPU封装内，但仍是独立的Die）。

缺点是数据从L2到CPU需要经过主板上的总线，速度慢、延迟高（通常是芯片内缓存的数倍），严重限制了性能。

第二阶段：L2“搬进”芯片内

随着半导体工艺进步（晶体管变小，集成度提高），从奔腾Pro、奔腾II后期（片内封装）、到奔腾III“铜矿”核心、以及AMD K6-III等处理器开始，L2缓存被集成到了与CPU核心同一个芯片上。这使L2的访问延迟从上百个周期骤降到十几个周期，性能得到质的飞跃。从此，“芯片外L2缓存”在消费级市场基本被淘汰。容量上比L1大得多，通常每核心 几百KB到1MB 左右（例如256KB, 512KB, 1MB）。作为L1缓存和更慢的L3缓存（或内存）之间的缓冲区。当L1缓存未命中时，CPU会首先在L2中查找数据。

第三阶段：L3缓存出现并集成

当多核CPU成为主流后，为了协调多个核心之间的数据共享，避免它们频繁访问慢速的主内存，L3缓存应运而生。现代的L3缓存从一开始就是芯片内缓存，被设计为所有CPU核心共享的大容量数据池。它离所有核心都“较远”，速度比L2慢，访问延迟可能在 30-50个时钟周期或更多。容量最大，从 几MB到几十MB，甚至上百MB（例如8MB, 16MB, 32MB, 64MB，服务器CPU可达数百MB）。核心作用：

  • 共享数据池：当多个核心需要访问同一份数据时，可以从L3直接获取，避免了去访问更慢的主内存，也减少了核心间通过内存交换数据的开销。
  • 内存访问的最后屏障：只有在L3中也找不到所需数据时，CPU才会被迫去访问速度慢上百倍的主内存

高速缓存读写操作

读操作

CPU 通过 Cache 的映射机制 来确定内存数据在 Cache 中的存储位置。

直接映射 Cache

内存中的每个数据块只能放到 Cache 中唯一的一个位置。 这个位置由内存地址的一部分（索引字段 Index）决定。

查找过程

  1. CPU 给出内存地址。
  2. 用 Index 位 直接找到 Cache 中的唯一一行。
  3. 将该行中保存的 Tag 与地址中的 Tag 比较。
  4. 若匹配且有效位为 1，则命中，再根据 Block Offset 取出数据。
  5. 若不匹配或无效，则未命中，需从内存/下一级 Cache 载入数据，并覆盖该行。
  • 优点：硬件简单，查找速度快（一次索引比较即可）。
  • 缺点：容易冲突，如果两个不同 Tag 但 Index 相同的内存块频繁交替访问，会导致 Cache 频繁替换。
全相连 Cache

查找过程

  1. CPU 给出内存地址，提取 Tag。
  2. 并行比较 Cache 中所有行的 Tag（使用硬件电路，如相联存储器）。
  3. 如果有某一行的 Tag 匹配且有效，则命中。
  4. 若都不匹配，则未命中。
  • 优点：Cache 空间利用最灵活，冲突最少。
  • 缺点：硬件成本高（需要很多比较器），容量不能太大（否则比较电路复杂、功耗高、速度慢）。
组相连 Cache

前两者的折中方案。Cache 分为若干组（Set），每组内有若干行（Way）。内存块先映射到唯一的一个组（类似直接映射），但在这个组内可以放到任意一行（类似全相连）。

查找过程

  1. 用 Set Index 找到对应的组。
  2. 在该组内，并行比较所有行的 Tag（路数少，比较器数量可控）。
  3. 若匹配某一行，则命中；否则不命中，需要替换该组中的某一行（按 LRU、随机等策略）。
  • 优点：平衡了速度与冲突率，是现代 CPU 最常用的方式（比如 4 路、8 路、16 路组相连）。

写操作

在搞清楚从内存加载数据到 Cache，以及从 Cache 里读取到想要的数据之后，我们又要面临一个新的挑战。CPU 不仅要读数据，还需要写数据，我们不能只把数据写入到 Cache 里面就结束了。CPU 要写入数据的时候，怎么既不牺牲性能，又能保证数据的一致性。

CPU 数据写入时会有以下两个挑战：

  • 什么时候写入主存？缓存什么时候失效？写直达 vs 写回策略。写入 Cache 的性能也比写入主内存要快，那我们写入的数据，到底应该写到 Cache 里还是主内存呢？如果我们直接写入到主内存里，Cache 里的数据是否会失效呢？
  • 多核 CPU 缓存一致性的问题。无论是写直达还是写回策略都不能解决多核 CPU 缓存一致性的问题。现在计算机采用缓存一致性协议 MESI 解决一致性问题。
特性 写直达（Write-Through） 写回（Write-Back）
写入时机 同时写缓存和内存 只写缓存，淘汰时才写内存
性能 差（每次写都访问内存） 优（多数写只访问缓存）
内存带宽 占用高 占用低
一致性 强（缓存内存始终同步） 弱（需要协议保证）
复杂度 简单 复杂（脏位、替换策略）
现代CPU使用 较少（通常只在L1指令缓存或特殊场景） 主流（几乎所有现代CPU数据缓存）
写直达

每次CPU执行写操作时，数据会同时写入：缓存和主内存，如果缓存中没有该数据（写未命中），有两种常见处理：

  • 写分配（Write-Allocate）：先将数据块加载到缓存，然后更新缓存和内存。
  • 非写分配（No-Write-Allocate）：直接写入主内存，不加载到缓存。
写回

CPU写操作时，只更新缓存中的副本，不立即写入主内存。缓存中的数据块被标记为 “脏”（Dirty）。只有当这个“脏”数据块被替换出缓存时，才被写回主内存。

如果写未命中，通常采用 写分配（Write-Allocate）：将目标数据所在的整个缓存块从内存加载到缓存。然后更新缓存中的相应数据，并标记为“脏”。

缓存一致性 MESI 协议

缓存一致性问题

在一个多核CPU系统中：

  1. 每个核心都有自己私有的L1（和通常私有的L2）缓存。
  2. 它们共享同一块主内存。
  3. 当一个核心修改了自己缓存中的数据（这块数据在内存中也有），其他核心的缓存里可能还存着这份数据的旧副本。这就产生了不一致。

具体会出现以下几种典型的不一致场景：

问题场景 描述 类比
1. 读过期数据 核心A修改了缓存中的变量X，但还没写回内存。核心B去读内存中的X，读到的就是过期的旧值。 更新了本地文档，但没上传。同事从云端下载，拿到的是旧版。
2. 写丢失 核心A和核心B几乎同时从内存读取了变量X到各自缓存。然后A修改了X，B也修改了X。后写回内存的会覆盖先写回的，导致一次修改丢失。 和同事同时下载文档，各自修改同一段后保存。后上传者的版本会覆盖先上传者的。
3. 数据不一致 核心A修改了X，并写回了内存。但核心B的缓存中还有旧的X。后续B用自己的旧X进行计算，结果就是错的。 上传了新版本，但同事的电脑没自动更新，还在用旧版本工作。

为了缓存不一致的问题，就需要有一种机制，来同步两个不同 CPU 的缓存数据。那这样的机制需要满足什么条件呢？

  • **写传播(Write Propagation)**：在一个 CPU 核心里，Cache 数据更新，必须能够传播到其他的对应节点的 Cache Line 里。
  • **事务的串行化(Transaction Serialization)**：在一个 CPU 核心里面的读取和写入，在其他的节点看起来，顺序是一样的。
解决方案
  • **总线嗅探(Bus Snooping)**：最常见的一种解决多核 CPU 数据广播问题的方案。本质上就是把所有的读写请求都通过总线(Bus) 广播给所有的 CPU 核心，然后让各个核心去“嗅探”这些请求，再根据本地的情况进行响应。
  • MESI 协议：基于总线嗅探机制的缓存一致性协议，MESI 协议也是在 Pentium 时代被引入到 Intel CPU。
总线嗅探机制

最常见的一种解决多核 CPU 数据广播问题的方案。本质上就是把所有的读写请求都通过总线(Bus) 广播给所有的 CPU 核心，然后让各个核心去“嗅探”这些请求，再根据本地的情况进行响应。每个CPU不断监听总线，发现与自己缓存相关的操作时做出响应。

MESI 协议

基于总线嗅探机制的缓存一致性协议，MESI 协议也是在 Pentium 时代被引入到 Intel CPU。

写传播 vs 写失效

  • 写失效(Write Invalidate) 协议：在写失效协议里，在这个 CPU 写入 Cache 后，它会去广播一个“失效”请求给其他所有 CPU。其他 CPU 如果也存在这个缓存行，直接标记成失效的就好了。MESI 协议使用的就是写失效协议。
  • 写传播(Write Invalidate) 协议：会将数据广播给其他 CPU，相对于写失效请求，占用更多的带宽。
特性 写失效（Write Invalidate） 写更新（Write Update/Write Broadcast）
MESI使用 是MESI采用的方式 MESI不使用
操作方式 写入时广播”失效”信号 写入时广播新数据本身
带宽占用 低（只传信号） 高（传完整数据）
后续读取 其他核心需重新获取数据 其他核心已有最新数据
网络流量 一次失效信号 可能多次广播数据
典型协议 MESI, MOESI Dragon协议

MESI 协议的由来，来自于对 Cache Line 的四个不同的标记，分别是：

  • M：代表已修改(Modified)，表示数据修改后未同步到主存的脏数据。
  • E：代表独占(Exclusive)，独占模式下，可以对应数据任意修改，不需要广播给其它 CPU。
  • S：代表共享(Shared)，当有两个 CPU 读取数据时，缓存行的状态从独占变成共享。此时修改数据时，需要广播给其他 CPU。
  • I：代表已失效(Invalidated)，表示数据已经失效，需要重新从主存中读取数据。

内存

程序运行时，指令和数据都需要先加载到内存里面，才会被 CPU 拿去执行。那程序中的虚拟地址最终是如何映射到内存中的物理地址呢？

虚拟地址和物理地址

程序在编译时不可能知道装载后的物理内存地址，实际上，程序编译生成的地址都是虚拟地址。在日常使用的 Linux 或者 Windows 操作系统下，程序并不能直接访问物理内存。为了解决这个问题，当程序装载后，会通过虚拟地址映射到真实的物理地址。

内存被分成固定大小的页(Page)，然后再通过虚拟内存地址(Virtual Address) 到物理内存地址(Physical Address) 的**地址转换(Address Translation)**，才能访问实际存放数据的物理内存位置。而我们的程序看到的内存地址，都是虚拟内存地址。

页表

**页表(Page Table)**：想要把虚拟内存地址，映射到物理内存地址，最直观的办法，就是来建一张映射表。这个映射表，能够实现虚拟内存里面的页，到物理内存里面的页的映射。这个映射表，在计算机里面，就叫作页表。

页表地址转换，把一个内存地址分成页号(Directory) 和偏移量(Offset) 两个部分。以一个 32 位的内存地址，页的大小 4KB 为例，内存地址的 20 位的高位表示页号，12 位（212 = 4KB）的低位表示偏移量。

对于一个内存地址转换，其实就是这样三个步骤：

  1. 把虚拟内存地址，切分成页号和偏移量的组合；
  2. 从页表里面，查询出虚拟页号，对应的物理页号；
  3. 直接拿物理页号，加上前面的偏移量，就得到了物理内存地址。

多级页表

很明显，大部分进程所占用的内存是有限的，只需要保存那些用到的页之间的映射关系就好了。如果你对数据结构比较熟悉，你可能要说了，那我们是不是应该用哈希表（HashMap）这样的数据结构呢？

很可惜你猜错了。在实践中，我们其实采用的是一种叫作多级页表(Multi-Level Page Table) 的解决方案。为什么我们不用哈希表而用多级页表呢？

进程的内存地址分配 - “两头实、中间空”

要知道为什么使用多级页表而不是哈希表，首先就要知道，一个进程的内存地址空间是怎么分配的。在整个进程的内存地址空间，通常是 “两头实、中间空”。在程序运行的时候，内存地址从顶部往下，不断分配占用的栈的空间。而堆的空间，内存地址则是从底部往上，是不断分配占用的。

所以，在一个实际的程序进程里面，虚拟内存占用的地址空间，通常是两段连续的空间。而不是完全散落的随机的内存地址。而多级页表，就特别适合这样的内存地址分布。

多级页表结构

事实上，多级页表就像一个多叉树的数据结构，所以我们常常称它为**页表树(Page Table Tree)**。这种数据结构其实和 B+ 树类似，允许一个结点存储多条记录，并且非叶子结点只存储索引，只有叶子结点存储数据。使用多级页表后，同样的虚拟内存地址，偏移量的部分和上面简单页表一样不变，而原先的页号部分需要拆分成多段。

假如一个进程会有一个 4 级页表，通过虚拟地址查找物理地址时：

  1. 先通过 4 级页表索引，找到 4 级页表里面对应的条目（Entry）。这个条目里存放的是一张 3 级页表所在的地址。4 级页面里面的每一个条目，都对应着一张 3 级页表，所以我们可能有多张 3 级页表。
  2. 再根据 3 级页表索引，在 3 级页表中查找对应的 2 级页表地址。
  3. 依次类推，直到 1 级页表。在最后一级页表中，保存虚拟地址对应的物理地址页号。

加速地址转换(TLB)

多级页表以时间换空间的策略，大节省了内存开销。但使用 4 级页表后，就需要访问 4 次内存才能找到物理页号。而虚拟地址和物理内存地址之间的地址转换，是一个非常高频的动作，对它的性能要求非常高，你有什么好的解决方案呢？我们最先想到的可能就是加缓存，事实上，CPU 也是这么做的。

现代计算机使用虚拟内存——每个程序都认为自己独占整个内存空间（如4GB）。但物理内存有限，所以需要地址转换：虚拟地址 → 物理地址， 这种转换通过页表（内存中的数据结构）完成。致命问题出现了，要访问一个数据：用虚拟地址查页表（需要访问内存）→ 得到物理地址， 再用物理地址访问实际数据（再次访问内存）。每次内存访问实际上需要两次内存访问， 速度直接减半。

TLB是CPU内存管理单元（MMU）内部的专用高速缓存，专门缓存最近使用过的虚拟页到物理页的映射关系。

  1. 位置：集成在CPU内部，MMU的一部分
  2. 速度：访问速度与L1缓存相当（1-3个时钟周期）
  3. 内容：只存储页表条目（VPN→PPN映射），不存储实际数据（这块缓存存放了之前已经进行过地址转换的查询结果）
  4. 大小：很小，通常64-1024个条目（因为页表条目很小，4-8字节）

程序所需要使用的指令，都顺序存放在虚拟内存里面。执行的指令，也是一条条顺序执行下去的。也就是说，对于指令地址和需要访问的数据，都存在空间局部性和时间局部性。

连续执行了 5 条指令，因为内存地址都是连续的，所以这 5 条指令通常都在同一个“虚拟页”里。可以把之前的内存转换地址缓存下来，使得不需要反复去访问内存来进行内存地址转换。

有了 TLB ，当同样的虚拟地址需要进行地址转换的时候，我们可以直接在 TLB 查询结果，而不需要多次访问内存来完成一次转换。

TLB 和前面讲的 CPU 的高速缓存类似，可以分成指令的 TLB 和数据的 TLB，也就是 ITLB 和 DTLB。同样的，我们也可以根据大小对它进行分级，变成 L1、L2 这样多层的 TLB。另外，和高速缓存一样，同样需要用脏标记这样的标记位，来实现 “写回” 这样缓存管理策略。
