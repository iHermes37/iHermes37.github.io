区分多线程和异步

什么是异步

异步，原版英文文资料写的是 Async IO 或者 Async IRQ，正常翻译为异步IO。

异步的 Input/Outout 或者是 异步的Interrupt Request

核心在IO上，也就是输入一个参数，然后计算机输出结果。异步IO的意思其实是当输入参数后调用就结束了，等待计算机有输出结果后重新“回调”返回。（类似你打个电话通知某人做某事，然后挂断。等这个人把事情做完后在电话回复你。自然这中间你还可以做其他事情）

和异步IO对应的项其实是同步IO，而不是多线程（同步IO，还是可以用打电话解释，依旧是你打电话通知某人做某事，但是同步IO的情况是，你不挂电话，你就等着对方把事情做完，然后告诉你结果）

异步编程的核心：靠 IO 设备通知，而不是靠线程等待，即CPU 不等待 IO，而是让 IO 设备自己在“准备就绪”时通知 CPU。

需要澄清的是异步IO实现手段很多，多线程仅仅是其中一种手段。可以是个线程，也可以是个事件，还可以是操作系统中断信号，还可以是异步IRQ，也可以是协程，子进程。

不同语言和平台都有不同的实现。

  • Java的nio包里有Selector, SocketChannel, SelectionKey。Netty就是基于这个东西构建的。
  • Linux底层系统调用有select(), poll(), 和epoll()。
  • Windows底层有IOCP。
  • Node.js的事件循环底层其实也是靠libuv这个库封装了epoll, kqueue, IOCP。

异步的底层实现“思想”：核心只有两种（其他都是变体）

1. 中断驱动（Interrupt-driven）

硬件完成 IO → 发中断 → OS 调度 → 回调应用
现代网络几乎都这样做。

2. 事件轮询（Event Loop）

事件循环不断检查 ready IO：

  • epoll
  • select
  • poll
  • kqueue
  • IOCP 模拟出来的回调

多线程和异步的不同

从计算机体系结构角度说：

概念 属于 定义
异步（Async） 编程模型 解决“等待 IO 不要阻塞线程”的抽象方式
多线程（Multithreading） 执行模型 解决“让多个 CPU 核心并行跑任务”的方式

异步关心“怎么写代码”，多线程关心“代码怎么跑”。

就理论来说，这二者不是一个维度的概念。就具体实现来说，多线程是模拟实现异步的方法之一。实际软件系统中，异步的真正实现，离不开底层（操作系统），但完全可以不需要多线程。

像什么面向过程、面向对象、函数式、事件驱动（回调）、异步/等待、Actor模型这些都是常见的编程模型。这些是让程序员用来描述想让计算机做什么怎么做的一套规则和抽象概念。

像什么进程、线程、协程、CPU指令执行、缓存机制、中断处理这些是常见的执行模型概念。这个层面的东西主要是操作系统内核和CPU硬件来负责实现的。

某个操作特别耗时，磁盘I/O、网络请求，我们不想等。我们想发起操作后马上就返回，任务在后台执行，等完成的时候通过回调或者通知告诉我们就行了。这种非阻塞的执行方式让多个任务逻辑上可以同时进行。

总结来说：多线程，通常只是在CPU 上跑任务时的概念。但是异步，却可以是CPU 和 IO 的之间的独立，也就是说，异步是更宽泛的概念，而如果只是讲CPU ，那么通常异步就约等于在说多线程或者多进程。

但是，CPU 通常不仅仅在执行CPU 密集型任务。有的任务，是等待IO 响应。此时，如果仅仅是等待，实际上并不需要占用CPU，在IO 完成之后，通知CPU， 然后CPU 再来恢复线程，继续处理下一步工作即可。

异步完全可以在单线程里实现，你看JS，只有一个线程，照样能异步发起多个请求并在响应到来的时候分别处理。

异步编程不会让单个任务跑得快起来。不管是同步还是异步，耗时的那部分操作客观所需要的时间不会变。异步的最大的好处就在于它可以重叠等待时间，让本来串行的等待变成并行，从而提高整体吞吐。

在IO密集型场景下，CPU他不会傻等IO，而是在IO还没好的时候切换到其他任务去处理。这样线程就不会长时间阻塞了，从而减少了线程资源浪费。但是如果在CPU密集型场景下，单靠异步并不能提高速度，因为CPU始终要一个一个的执行计算任务。这个时候就需要利用多线程或多进程让多核CPU并行工作。

比方说，你在访问一个很远很远的网站网络也不好，卡了半天才收到信息。那么，应用在访问网络时，可以使用非阻塞式IO来访问应用发出请求后，立即返回，并在事件队列里，添加一条事件。意思是，等网卡完成了，去事件队列里找到我的事件，并通知我即可。此时，网页可以是一种显示等待的状态，但CPU 可以做别的事去了等网卡数据到了之后，发出中断信号通知CPU， 再通过操作系统内核兜兜转换访问到这个事件队列再来恢复现场继续执行后边的代码。这个等待过程，CPU 没有同步等待 网卡如果没有采用非阻塞式IO那么CPU 还是需要开一个线程去等待这个过程中，CPU 的时间占用还是存在的。

采用非阻塞式IO的异步，不是CPU 多个线程（进程）之间的异步，而是 CPU 与IO 设备之间的异步。所以说，异步的概念比多进程多线程更加宽泛，或者说，多进程多线程只是异步的一种实现方式。
