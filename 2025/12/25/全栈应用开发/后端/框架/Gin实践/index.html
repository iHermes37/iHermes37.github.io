当路由参数复杂时，则需要结构化绑定：

与普通路由参数的区别：

  • 结构化绑定，自动映射到结构体字段

其特定是针对于：URL ? 后面的部分，用于过滤、分页、排序等可选参数，顺序无关，可以任意排列，可以缺失（有默认值处理）

使用场景对比：

场景 推荐使用 示例 原因

请求体绑定

将 request body 绑定到不同的结构体中，这是最灵活的绑定方式，根据 Content-Type 自动选择绑定器。

支持的 Content-Type：

  • application/json
  • application/xml
  • application/x-www-form-urlencoded (表单)
  • multipart/form-data (文件上传)

外部重定向

重定向到外部域名或完整 URL。

应用场景：

  • 域名迁移
  • 外部资源引用
  • 第三方登录跳转

内部重定向

重定向到同一应用内的其他路由。

不同 HTTP 方法的重定向：（POST 请求重定向的特殊性）

HTTP 状态码选择：

  • 301 Moved Permanently - 永久重定向（GET）
  • 302 Found - 临时重定向（方法可能改变）
  • 303 See Other - 推荐用于 POST 重定向
  • 307 Temporary Redirect - 保持原请求方法
  • 308 Permanent Redirect - 永久重定向并保持方法

Sessions

gorilla/sessions为自定义session后端提供cookie和文件系统session以及基础结构。

主要功能是：

  • 简单的API：将其用作设置签名（以及可选的加密）cookie的简便方法。
  • 内置的后端可将session存储在cookie或文件系统中。
  • Flash消息：一直持续读取的session值。
  • 切换session持久性（又称“记住我”）和设置其他属性的便捷方法。
  • 旋转身份验证和加密密钥的机制。
  • 每个请求有多个session，即使使用不同的后端也是如此。
  • 自定义session后端的接口和基础结构：可以使用通用API检索并批量保存来自不同商店的session。

路由

gin支持Restful风格的API，即Representational State Transfer的缩写。直接翻译的意思是”表现层状态转化”，是一种互联网应用程序的API设计理念：URL定位资源，用HTTP描述操作

  1. 获取文章 /blog/getXxx Get blog/Xxx
  2. 添加 /blog/addXxx POST blog/Xxx
  3. 修改 /blog/updateXxx PUT blog/Xxx
  4. 删除 /blog/delXxxx DELETE blog/Xxx
特性 API参数（路径参数） URL参数（查询参数） 表单参数（请求体参数）
位置 URL路径中 URL ? 后面 HTTP 请求体中
示例 /users/:id ?name=zs&page=1 POST 请求体中的参数
获取方法 c.Param("id") c.Query("name") c.DefaultQuery("page", "1") c.PostForm("field") c.DefaultPostForm("field", "default")
绑定到结构体 c.ShouldBindUri(&obj) c.ShouldBindQuery(&obj) c.ShouldBind(&obj) c.ShouldBindJSON(&obj)
Content-Type 不适用 不适用 application/json application/x-www-form-urlencoded multipart/form-data application/xml
主要HTTP方法 GET, POST, PUT, DELETE等 GET（主要） POST, PUT, PATCH（主要）
可见性 地址栏可见 地址栏可见 地址栏不可见
数据大小 较小 有限（URL长度限制） 较大（MB级别）
安全性 较低（日志记录） 较低（日志/书签） 较高
缓存 可缓存 可缓存 通常不缓存
中文处理 URL编码 URL编码 根据Content-Type编码
数组参数 不支持 ?tags=go&tags=java → c.QueryArray("tags") JSON数组 multipart数组
文件上传 不支持 不支持 multipart/form-data + c.FormFile()
