内核是如何接受网络包的？

⽤户执⾏完recvirom调⽤后，⽤户进程就通过系统调⽤进⾏到内核态⼯作了。如果接收队列没有数据，进程就进⼊睡眠状态被操作系统挂起。这块相对⽐较简单，剩下⼤部分的 “戏份”都是由Linux内核其他模块来“表演”了。

Linux⽹络收包总览

内核和⽹络设备驱动是通过中断的⽅式来处理的。

特性 硬中断 软中断
触发源 硬件设备 内核软件
优先级 最高，立即执行 较高，稍后执行
执行上下文 中断上下文 软中断上下文
主要任务 快速响应硬件，记录状态，触发软中断 执行耗时、繁重的后续处理
可否睡眠 绝对禁止 禁止

当设备上有数据到达时，会给CPU的相关引脚触发⼀个电压变化，以通知CPU来处理数据。对于⽹络模块来说，由于处理过程⽐较复杂和耗时，如果在中断西数中完成所有的处理，将会导致中断处理函数（优先级过⾼）过度占⽤CPU，使得CPU⽆法响应其他设备，例如⿏标和键盘的消息。

因此Linux中断处理函数是分上半部和下半部的。

上半部只进⾏最简单的⼯作，快速处理然后释放CPU，接着CPU就可以允许其他中断进来。将剩下的绝⼤部分的⼯作都放到下半部，可以慢慢、从容处理。

2.4以后的Linux内核版本采⽤的下半部实现⽅式是软中断，由ksoftirad内核线程全权处理。

硬中断是通过给CPU物理引1脚施加电压变化实现的，⽽软内核是如何接收⽹络包的中断是通过给内存中的⼀个变量赋予⼆进制值以标记有软中断发⽣。

完整的数据流可以清晰地展示两者的关系：

  1. 数据到达： 数据包从网络线到达网卡。
  2. 临时暂存： 数据包首先被存入网卡自身的硬件 FIFO 缓冲区。这个缓冲区很小，主要用于应对瞬时突发流量和速度匹配。
  3. DMA 传输准备： 网卡从位于主内存的描述符环中，取得下一个可用的描述符，读取其中的数据缓冲区物理地址。
  4. DMA 传输： 网卡通过 DMA 引擎，直接将数据包从自己的 FIFO 缓冲区写入到主内存中描述符所指向的数据包缓冲区。此过程完全不占用 CPU。
  5. 更新状态： 传输完成后，网卡再次通过 DMA 或总线写入，更新主内存中那个描述符的状态（例如，标记为“数据已就绪”）。
  6. 触发中断： 网卡向 CPU 发送一个硬中断，通知它已有数据包接收完成。

⽹络⼦系统初始化

在⽹络⼦系统的初始化过程中，会为每个CPU初始化softnet _ data，也会为RXSOFTIRQ和TX SOFTIRQ注册处理函数。

Linux内核通过调⽤subsys initcall来初始化各个⼦系统，在源代码⽬录⾥可以⽤grep命令搜索出许多对这个西数的调⽤。这⾥要说的是⽹络⼦系统的初始化，会执⾏net dev init两数。

现代 CPU 都是多核的。为了充分发挥多核性能、避免 CPU 之间争抢同一把锁，Linux 内核为许多关键数据结构创建了 “每CPU变量”。这意味着每个 CPU 核心都有自己独立的一份数据副本。softnet_data 就是这样一个每CPU数据结构。它是网络子系统软中断处理的核心上下文，每个 CPU 核心都有自己的 softnet_data 实例。

  • poll_list（通常是一个链表或队列），用于等待网络设备驱动程序将其 poll 函数注册进来。
  • open_softirq 为每一种软中断注册处理函数，内核中定义了多种软中断类型（如 NET_TX_SOFTIRQ, NET_RX_SOFTIRQ, TIMER_SOFTIRQ 等），但光有类型不行，必须告诉内核“当这种软中断被触发时，应该执行哪个函数来处理”。
  • 注册的信息记录在 softirq_vec 变量里。这是一个全局的、每软中断类型的数组。
    • 可以把它想象成一个“电话黄页”或“服务分派表”：
      • 数组的索引是软中断的类型号（如 NET_RX_SOFTIRQ 对应索引 3）。
      • 数组的每个元素是一个结构体，里面包含了该类型软中断对应的处理函数指针（action）。

协议栈注册

内核实现了⽹络层的iP协议，也实现了传输层的TCP协议和UDP协议。这些协议对应的实现西数分别是ip_rcv、tcp_v4_rcv()和udp_rev()。 和平时写代码的⽅式不⼀样的是，内核是通过注册的⽅式来实现的。Linux内核中的fs_initcal和subsys_initcall 类似，也是初始化模块的⼊口。fs_initcall调⽤inet_init 后开始⽹络协议栈注册，通过inet_init， 将这些函数注册到inet_protos和ptype_base数据结构中.

这⾥需要记住 inet_protos记录着UDP、TCP的处理函数地址，ptype_base存储着ip_ rev()函数的处理地址。后⾯将讲到软中断中会通过ptype_base找到ip_rev西数地址，进⽽将iP包正确地送到ip_rcv()中执⾏。在ip_rcv()中将会通过inet_prolos找到TCP或者UDP的处理函数，再把包转发给udp_ rev()或tep_ v4_rcv()西数。

同步阻塞⽅式接收⽹络包的整个过程分为两部分：

  • ⼀部分是我们⾃⼰的代码所在的进程，我们调⽤的socket()函数会进⼊内核态创建必要内核对象。 recv()⻄数在进⼊内核态以后负责查看接收队列，以及在没有数据可处理的时候把当前进程阻塞掉，让出CPU
  • 第⼆部分是硬中断、软中断上下⽂（系统线程ksottrad）。在这些组件中，将包处理完后会放到socket的接收队列中。然后根据socket内核对象找到其等待队列中正在因为等待⽽被阻塞掉的进程，把它唤醒。

每次⼀个进程专门为了等⼀个socket 上的数据就被从CPU上拿下来，然后换上另⼀个进程，这个过程会产生两次进程上下文切换：

  1. 阻塞时：用户进程 → 内核 → 其他进程
  2. 唤醒时：当前进程 → 内核 → 被唤醒进程

根据业界的测试，每⼀次切换⼤约花费3-5微秒，在不同的服务器上会有⼀点⼉出⼊，但上下浮动不会太⼤。要知道从开发者⾓度来看，进程上下⽂切换其实没有做有意义的⼯作。如果是⽹络IO密集型的应⽤， CPU就会被迫不停地做进程切换这种⽆⽤功。

IO多路复用之epoll

epoll 的三组件设计哲学

  • 避免忙等待：没有事件时主动让出CPU，提高系统整体效率
  • 事件驱动：只有真正有数据时才唤醒进程处理

重要概念澄清：

  • epoll本身是阻塞的：epoll_wait() 在没有事件时会阻塞进程
  • socket设置为非阻塞：避免在单个socket的读写操作中阻塞
  • 主语很重要：必须明确讨论的是哪个组件或操作的阻塞特性

为什么要克隆skb：

  • 重传需求：TCP支持丢失重传，在收到ACK前原始skb必须保留
  • 内存管理：网卡发送完成后会释放skb，但原始数据需要保留用于可能的重传
  • 性能优化：克隆比完整拷贝更高效，共享数据缓冲区

网络层发送处理

Linux内核⽹络层的发送的实现位于 net/ipv4/ip_output.c 这个⽂件。传输层调⽤到的ipqueue_xit 也在这⾥。在⽹络层主要处理路由项查找、P头设置、netilter过滤、skb切分(⼤于MTU的话），等⼏项⼯作，处理完这些⼯作后会交给更下⼀层的邻居⼦系统来处理。

邻居⼦系统

邻居⼦系统是位于⽹络层和数据链路层中间的⼀个系统，其作⽤是为⽹络层提供个下层的封装，让⽹络层不必关⼼下层的地址信息，让下层来决定发送到哪个 MAC地址。⽽且这个邻居⼦系统并不位于协议栈net/pv4/⽬录内，⽽是位于net/core/neighbour. C。因为⽆论是对⼿ PV4 还是IPV6，都需要使⽤该模块

在邻居⼦系统⾥主要查找或者创建邻居项，在创建邻居项的时候，有可能会发出实际的erp请求。然后封装MAC 头，将发送过程再传递到更下层的⽹络设备⼦系统。

网络设备⼦系统

邻居⼦系统通过dev_queue_xmi进入⽹络设备⼦系统。

软中断调度

如果发送⽹络包的时候系统态CPU⽤尽了，会调⽤_ netif_schedule触发⼀个软中断。该函数会进入__netif_reschedule，由它来实际发出NET_TX_SOFTIRQ类型软中断。软中断是由内核进程来运⾏的，该进程会进⼊net_tx_action西数，在该函数中能获取发送队列，并也最终调⽤到驱动程序⾥的入库函数dev_hard_start_xmit。

IGB⽹卡驱动发送

通过前⾯的介绍可知，⽆论对于⽤户进程的内核态，还是对于软中断上下⽂，都会调⽤⽹络设备⼦系统中的dev hard start xmit函数。 在这个函数中，会调⽤到驱动⾥的发送函数gb_xmit_frame，在驱动函数⾥，会将skb挂到RingBuffer上，驱动调⽤完毕，数据包将真正从⽹卡发送出去。

RingBuffer内存回收

当数据发送完以后，其实⼯作并没有结束。因为内存还没有清理。当发送完成的时候，⽹卡设备会触发⼀个硬中断来释放内存

总结

发送⽹络数据的时候都涉及哪些内存拷⻉操作？

第一次拷贝：用户空间 → 内核skb

特点：

  • 完全拷贝：将用户buffer中的数据完整复制到内核skb数据区
  • 开销较大：数据量越大，拷贝开销越显著
  • 不可避免：由于内核与用户空间隔离，必须进行拷贝

第二次拷贝：skb克隆（传输层）

特点：

  • 浅拷贝：只复制skb描述符结构，共享数据缓冲区
  • 目的：支持TCP重传机制，保留原始skb用于可能的重新发送
  • 开销较小：主要复制控制结构，数据区是共享的

第三次拷贝：IP分片（网络层）

特点：

  • 条件性拷贝：只在skb > MTU时发生
  • 数据拆分：将大skb拆分为多个符合MTU的小skb
  • 完全拷贝：需要复制数据到新的skb中

零拷贝到底是怎么回事？

传统方式：read + send 的拷贝过程

性能瓶颈
  • 两次CPU拷贝：数据在用户态和内核态之间来回搬运
  • 上下文切换：read和send两次系统调用
  • 缓存污染：数据污染用户空间缓存，影响其他应用性能

零拷贝方式：sendfile 的优化

如果要发送的数据量⽐较⼤，那需要花费不少的时间在⼤量的数据拷贝上。前⾯提到的sendfile就是内核提供的⼀个可⽤来减少发送⽂件时拷贝开销的⼀个技术⽅案。在sendtile系统调⽤⾥，数据不需要拷贝到⽤户空间，在内校态就能完成发送处理

katka⾼性能的原因有很多，其中的重要原因之⼀就是采⽤了sendtile系统调⽤来发送⽹络数据包
