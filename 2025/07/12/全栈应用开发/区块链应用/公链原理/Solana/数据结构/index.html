数据结构

区块结构

Slot 和 Epoch

  • Slot 是 Solana 网络中的一个时间单位。每个 slot 大约持续 400 毫秒。可以把 Slot 想象成一个时间段，预期在这个时间段内会生成一个区块。
  • Epoch 是一组连续的 slot 的集合。在 Solana 中，通常 1 Epoch 包含 432,000 个 slots，持续时间大约为 两天（具体时间取决于网络的交易量和验证器的速度）。

领导者计划表

在每个 Epoch 的开始时，Solana 会根据网络中各个 验证器（Validator） 的 权益大小（Stake） 来确定谁将在特定的 slot 中充当 领导者（Leader）。这意味着验证器拥有更多的 SOL 代币（通过质押Staking）会有更高的概率被选为领导者。

  • 领导者的角色： 当一个验证器被选为某个 slot 的领导者时，它负责在这个时间段内生成一个区块。即，领导者 负责为该 slot 创建并验证一个新区块，并将这个新区块广播到网络中。

领导者的任务

  • 收集交易： 当某个 slot 到达时，所有的交易会被转发给指定的领导者。领导者将收集在这个时间段内收到的交易，执行这些交易并将其打包成区块。
  • 交易选择与排序： 领导者有权决定哪些交易包含在区块中以及如何排序这些交易。换句话说，领导者不仅决定是否包含某个交易，还能决定区块内交易的顺序。

传播区块：Entries 和 Shreds

传统区块链的区块生产模式

在传统的区块链系统中，区块通常是一个单一的实体，生成一个区块后，必须等待下一个区块生产者接收整个区块并进行处理。处理过程包括：

  • 接收完整区块；
  • 处理区块中的所有交易；
  • 然后再基于该区块继续生成下一个区块。

这种模式会导致 显著的延迟和性能瓶颈。由于每个节点需要等待接收到完整的区块数据才能开始处理交易，因此存在以下问题：

  • 网络延迟：传输完整区块需要时间。
  • 处理延迟：节点必须处理完整的区块后，才能继续构建下一个区块。
  • 分叉问题：如果区块生成得太快，节点没有足够时间处理之前的区块，会导致网络分歧或共识失败。

这种机制可能导致 区块生成周期变长，进而影响区块链的吞吐量和响应速度。

Solana 的区块生产机制

Solana 采用了一种 创新的流式传输 方法，通过 分块 和 提前传播，最大限度减少区块传播过程中的延迟。具体流程如下：

区块分解为 Entries 和 Shreds：

  • Entries：Solana 在区块生产过程中，将区块分解为更小的部分，称为 entries。每个 entry 都包含一组交易数据。这些交易在领导者生成区块时，立即开始流式传输到网络的其他节点，而不是等到完整的区块生成后再发送。
  • Shreds：entries 进一步被划分为更小的单位，称为 shreds。这些 shreds 是数据传输的基本单元，在网络中传播以减少数据传输时的延迟。

流式传输：

在 Solana 中，区块生成的过程是流式的，也就是说，在区块完全生成之前，领导者就会开始将区块中的 entries 流式传输到网络中的其他节点。这样，其他节点在区块还没有完全生成时，就能提前开始处理这些数据。这种机制使得节点在接收到部分数据时，就可以开始执行交易处理，而不需要等到整个区块完成后再开始处理。

区块的定义：

在 Solana 中，区块 被定义为领导者在某个 slot 内生成的所有 entries 的集合。这些 entries 被传输到其他节点，而节点无需等待完整区块数据完成后再开始处理，从而大幅降低了网络延迟。

唯一的延迟：唯一的延迟来自于 最终数据包的网络传输延迟，即在所有 shreds 到达其他节点之前存在的物理网络限制。然而，一旦数据包到达，节点几乎可以立即开始处理区块中的交易。

账户模型

在 Solana 上，所有数据都存储在称为账户的结构中。每个帐户都可以通过其唯一地址进行识别，地址以Ed25519算法生成32个字节的公钥来表示。您可以将地址视为帐户的唯一标识符。

账户与其地址之间的这种关系可以看作是键值对，其中地址用于定位账户相应链上数据。

其中：

每个账户的账户信息包括以下字段：

  • data：存储账户状态的字节数组。如果账户是一个程序（智能合约），则存储可执行程序代码。该字段通常称为“账户数据”。
  • executable：一个布尔标志，指示该账户是否是一个程序。
  • lamports：账户余额的数字表示，单位为 lamports，这是 SOL 的最小单位（1 SOL = 10 亿 lamports）。
  • owner：指定拥有账户的程序的公钥（程序 ID）。

作为 Solana 账户模型的一个关键部分，每个 Solana 上的账户都有一个指定的“所有者”，即一个程序。只有被指定为账户所有者的程序才能修改账户上存储的数据或扣除 lamport 余额。需要注意的是，虽然只有所有者可以扣除余额，但任何人都可以增加余额。

程序

原生程序

在 Solana 上，程序（Program）是执行特定功能的核心组成部分。

程序类型 描述 功能
系统程序（System Program） 默认情况下，所有新账户由 System Program 所拥有。 负责账户创建、空间分配和账户所有权的转移。系统程序是 Solana 网络的核心，管理基本账户操作和交易费用支付。
BPF 加载程序（BPF Loader） 负责管理和执行所有自定义程序。 部署、升级和执行用户自定义程序。它是所有非原生程序（如用户自定义程序）的“拥有者”。
Sysvar 账户 特殊账户提供对集群状态数据的访问。 存储和更新网络集群的状态信息，帮助用户读取网络的实时数据（如租金、区块时间等）。

用户自定义程序

程序账户（Program Account） 代表链上程序的账户，存储可执行字节码的地址以及更新权限。 是程序的主账户，通常用于存储指向可执行字节码账户的地址和程序的更新权限。可以通过此账户调用程序。
可执行数据账户（Executable Data Account） 存储程序的可执行字节码的账户。 包含程序的实际字节码，一旦程序部署或升级，字节码存储在此账户中。
缓冲账户（Buffer Account） 临时账户，用于存储程序部署过程中的字节码。 在程序部署或升级期间使用，字节码会暂时存储在此账户，直到部署完成后，字节码会被转移到可执行数据账户，缓冲账户则关闭。
数据账户（Data Account） 存储程序运行时的数据，Solana程序是无状态的，因此需要单独的账户存储数据。 用于存储和修改数据，数据账户的所有权由程序控制，可以存储由程序定义的任意数据。每个程序可以定义自己的数据格式和内容。

关键注意点：

  1. 程序账户与数据账户分开存储： 在 Solana 中，程序代码和程序的数据是分开存储的，这使得程序可以独立升级，而不影响存储的数据。这种设计有助于提升合约升级的灵活性。
  2. 账户所有权转移： 系统程序可以创建新账户，但它可以将账户所有权转移给任何自定义程序，允许用户创建和管理自己的程序数据。
