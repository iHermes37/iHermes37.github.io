单向校验抓包攻防

背景

抓包原理

目前主流两种抓包方式：代理、VPN

  • 代理：在应用层（http），较为容易防范
  • VPN：在网络层（socket）

如果抓包无法看到响应，通常是：

  1. App 使用了 SSL Pinning。
  2. App 使用了非 HTTP 协议。
  3. App 使用了独立网络栈，不依赖系统代理。
特性 VPN 抓包 ADB / 系统代理抓包
拦截层级 系统路由层（TUN/TAP） 应用/系统 HTTP 代理层
覆盖范围 所有流量，包括自定义 Socket、QUIC 仅走系统代理的 HTTP/HTTPS
是否可绕过 较难，除非 App 做 VPN 检测 易绕过，Proxy.NO_PROXY、自定义 Socket
CA 证书依赖 可选，取决于是否解密 HTTPS 必须安装 CA 证书解密 HTTPS
实现复杂度 高，需要 VPNService + 用户空间转发 低，修改系统代理即可
HTTPS 解密 需要抓包工具自己 MITM 或透传 安装 CA 证书，HTTP/HTTPS MITM

代理攻防

系统代理（System Proxy）指的是操作系统层面配置的 HTTP/HTTPS 代理，例如 Wi-Fi 或移动数据网络中设置的代理。ADB / 系统代理抓包原理：

修改 系统 HTTP 代理设置（Wi-Fi 或全局代理），或通过 adb reverse / adb forward 把端口转发到抓包工具。App 发出的 HTTP/HTTPS 请求（遵循系统代理）会经过代理，然后代理解密（HTTPS 需要安装 CA 证书）。

特点：

  • 拦截范围有限：仅限使用系统代理栈的流量（大部分基于 HttpURLConnection / OkHttp / WebView 的请求）。
  • 容易被绕过：如果 App 用 Proxy.NO_PROXY 或自定义 socket，流量不走系统代理，就抓不到。
  • 实现方式：简单配置即可抓 HTTP/HTTPS，通常需要安装抓包工具的 CA 证书。

普通 HTTP/HTTPS 请求（如通过 NSURLSession、OkHttp、Retrofit）默认会走系统代理。也就是说，如果在 Charles/Fiddler/mitmproxy 里配置了全局代理，能拦截大部分标准网络请求。

绕过系统代理

在建立socket连接之前，OkHttp会获取系统的代理信息，如果设置代理，那么通过DNS解析其IP然后使用代理IP来建立socket连接。如果没有设置代理，那么会使用请求中的url的IP地址，来建立连接。

如果要防抓包，那只需要绕过代理就可以了。给app设置自己的代理地址，可能这个代理地址就是服务器的地址

OkHttp使用ProxySelector来获取代理信息，在构造OkHttpClient时是可以设置的，其默认值是ProxySelector.getDefault()，该默认值会反应出系统的代理信息。 那么我们就可以提供自己的ProxySelector实现来达到绕过系统代理的能力。

以上可以阻止基于系统/全局 HTTP 代理（在系统设置或 Wi‑Fi 中配置的代理）对该应用请求的拦截。对常见的 Charles/Fiddler（配置为系统代理）有效。

加密/自定义协议

一些 App 不通过系统 HTTP 栈，而是使用自定义 TCP/UDP 通信，或者基于 gRPC、QUIC、WebSocket、UDP Hole Punching 之类的协议。这种情况下，系统代理通常 不起作用，你抓包工具可能看不到解密后的明文流量。

加密数据需要后端配合较为麻烦，而且因为一直以来的不规范操作，要想大面积的加密隐私数据几乎不可能。但使用非http协议是可行的，比如bilibili搜索接口使用了grpc，常规代理抓包charles无法解析明文，提高了门槛

VPN攻防

通常情况下上述的办法有用，但是无法防住使用 VPN 导流进行的抓包（ Drony + Charles），使用VPN抓包的原理是，先将手机请求导到VPN，再对VPN的网络进行Charles的代理，绕过了对App的代理。

应用层代理（Proxy）

OkHttp、HttpUrlConnection 等客户端通常会读取 Java/系统代理设置（或你在代码里指定 proxy(...) / Proxy.NO_PROXY）。这影响的是应用发起连接时经过的代理/代理选择逻辑。这属于用户态、应用层的控制，容易被应用内禁用（你可以显式不走代理）。

传输/内核层：VPN 与导流

VPN（或者像 SocksDroid这类把设备流量导到本地代理的工具）会在内核层创建一个虚拟网络接口（如 tun0），并把所有 IP 包路由到该接口，然后用户空间程序/远端服务器负责转发。因为路由发生在更低层（在应用层的代理决策之前），应用即便不走系统代理，流量也已经被 VPN 转走并可被中间人代理（如 Charles）进一步处理/解密（如果用户安装了中间人证书的话）。

换句话说：Proxy.NO_PROXY ≠ “流量不可能被拦截”；它只是告诉客户端不通过 HTTP 代理，但不能阻止底层路由被改写。

即使流量被导到 Charles，是否能解密 HTTPS 取决于设备上是否信任 Charles 的根证书（系统证书/用户证书）。现代 Android（较新版本）对用户添加的 CA 有限制：如果 network_security_config 明确不信任用户 CA，或者应用做了证书绑定（pinning），MITM 就会失败。

告诉系统该 application 使用位于 res/xml/network_security_config.xml 的网络安全策略配置（Network Security Config，NSC）。

NSC 是从 Android 7（API 24）引入的，用来声明允许的 CA、是否允许明文（cleartext）、针对域名的 pinning 等。

不信任 user CA，关闭明文，针对关键域启用 pinning。

这里即使请求走了系统代理，App 开启 证书绑定（SSL Pinning），直接验证服务端证书。这会导致中间人代理失败（如 Charles/Fiddler），响应抓不到或报错 SSL handshake failed。

系统内置证书校验

代理抓包工具抓应用层的包，不会抓其他层的包，由于没有安装相对应的ssl证书，抓到的是乱码的数据无法识别出是http的请求，会默认他为其他层的数据包，就不去抓这个包，导致无法抓包

Okhttp配合X509TrustManager对服务器证书进行校验，请求框架使用的OKHttp，通过sslSocketFactory方法设置证书认证对象X509TrustManager

这里是利用设备中预先安装了权威CA机构颁发的根证书，受到了系统的信任

可以依赖OKHttp官方的支持库，但在用户设备被root，系统信任的根证书目录被篡改的情况下，这个方法就行不通了。设备被root后，系统已经丧失了安全性，攻击者往系统信任的根证书目录下插入自签名的证书，这样一来，这个方案就不可行了。

证书锁定原理

证书锁定（SSL/TLS Pinning）提供了两种锁定方式：

  • Certificate Pinning，证书锁定
  • Public Key Pinning，公钥锁定

证书锁定

  • 具体操作：将APP代码内置仅接受指定域名的证书，而不接受操作系统或者浏览器内置的CA根证书对应的任何证书。通过这种授权方式，保障了APP与服务端通信的唯一性和安全性，因此移动端APP与服务端（例如API网关）之间的通信可以保证绝对的安全。
  • 缺点：CA签发证书存在有效期问题，在证书续期后需要将证书重新内置到APP内。

公钥锁定

  • 具体做法：公钥锁定是提前证书中的公钥并内置到移动端APP内，通过与服务器对比公钥值来验证连接的合法性。
  • 优点：在制作证书密钥时，公钥在证书续期前后可以保持不变（即密钥对不变），所以可以避免证书有效期问题。

利用https握手时下发的公钥证书，与本地或者下发的证书公钥做比对，不一致则抛异常

目前的方案是只做了静态配置公钥校验。后续需要迭代证书公钥动态下发能力

证书动态下发

由后端提供一个下发接口，这个接口保障其他接口的证书校验，在证书过期时也能访问，所以不能校验证书。

不校验证书的情况下，有可能被劫持，下发攻击者自签名证书的公钥，从而让App信任攻击者的证书，所以接口必须加密，具体如下：

  • 非对称加密，后端生成秘钥和公钥，App持有公钥

  • 对称加密算法加密SSL证书的公钥字符串

  • 非对称加密算法的秘钥加密对称加密算法的秘钥

  • 接口出参两个字符串：加密后的SSL证书公钥、加密后的对称加密算法的秘钥

  • 客户端用非对称加密算法的公钥解密对称加密算法的秘钥，拿到对称加密算法的秘钥后解密SSL证书公钥，将证书公钥缓存下来

证书校验能力

后续考虑将证书校验拓展成原生能力，开放信任的证书池协助h5完成SSL证书校验工作。

攻击绕过SSL

绕过方法：
1、把App里面的证书揪出来，app和抓包软件的通信用那个证书。
2、删除掉App里面，单向认证的代码，让他能使用系统证书，然后他和抓包软件就能通
