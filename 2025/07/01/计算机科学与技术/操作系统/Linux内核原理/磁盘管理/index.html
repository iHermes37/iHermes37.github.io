虚拟文件系统

Linux使用与其他Unix变体相同的方式设法支持多种文件系统类型。

虚拟文件系统所隐含的思想是把表示很多不同种类文件系统的共同信息放入内核；其中一个字段或函数来支持Linux所支持的所有实际文件系统所提供的任何操作。对所调用的每个读、写或其他函数，内核都能把它们替换成支持本地Linux文件系统、NTFS文件系统，或者文件所在的任何其他文件系统的实际函数。

父路径名查找

在 Linux 内核的路径名查找机制中，父路径名查找（Parent Pathname Lookup） 是一种特殊场景，其核心目标是 获取路径名中最后一个分量之前的目录，而非最终的文件或目录本身。这种模式通过 LOOKUP_PARENT 标志触发，常见于以下操作：

  • 文件创建（如 open(O_CREAT)）：需要确定新文件所在的父目录。
  • 目录删除（如 rmdir）：需访问待删除目录的父目录。
  • 重命名（如 rename）：需分别解析源路径和目标路径的父目录。
父路径查找的核心逻辑

当设置 LOOKUP_PARENT 时，link_path_walk() 会：

  1. 正常解析路径，直到倒数第二个分量（父目录）。
  2. 不尝试访问最后一个分量（可能不存在或无需访问）。
  3. 在 nameidata 结构中记录：
    • last：最后一个分量的名称（如 "file.txt"）。
    • last_type：最后一个分量的类型（如普通文件、.、.. 等）。

last_type 的可能取值

值 说明
LAST_NORM 最后一个分量是普通文件名（如 /home/user/file.txt 中的 file.txt）。
LAST_ROOT 路径仅为 /（根目录）。
LAST_DOT 最后一个分量是 .（当前目录）。
LAST_DOTDOT 最后一个分量是 ..（父目录）。
LAST_BIND 最后一个分量是绑定挂载（Bind Mount）的符号链接。

符号链接查找

在 Linux 内核中，符号链接（Symbolic Link）的查找 是路径名解析中最复杂的场景之一。它需要递归处理链接指向的路径，同时防止恶意或错误的无限递归。

符号链接查找的核心流程

假设路径为 /foo/bar/file，其中 /foo/bar 是指向 ./dir 的符号链接：

  1. 第一次查找：解析 /foo/bar，发现 bar 是符号链接。
  2. 递归查找：
    • 读取 bar 的内容（./dir）。
    • 从 /foo 开始解析新路径 ./dir，得到目录 /foo/dir。
  3. 恢复原路径：继续从 /foo/dir 解析剩余分量 file。
递归控制与安全性
防止无限递归

**link_count**（进程描述符中的计数器）：每进入一层符号链接递归，link_count 加 1。若 link_count ≥ 6（默认最大嵌套深度 5），返回 -ELOOP。

**total_link_count**（进程描述符中的总计数器）：统计单次查找中所有符号链接的解析次数（包括非嵌套的）。若 total_link_count ≥ 40，返回 -ELOOP（防止恶意路径耗尽资源）。

递归终止条件:成功解析符号链接指向的最终目标。达到最大递归深度或总链接数。遇到无效路径（如权限不足、文件不存在）。

文件加锁

Linux 内核的 文件加锁机制 用于协调多个进程或线程对同一文件的并发访问，防止数据竞争和保证数据一致性。它支持多种锁类型和范围，适用于不同场景。以下是其核心机制和实现细节：

锁的类型

Linux 文件锁主要分为两类：

(1) 劝告锁

依赖进程主动检查锁状态，不强制阻止访问。只有遵守锁规则的进程才会受约束，恶意进程可忽略锁。适用场景：合作进程间的协调（如多进程读写同一配置文件）。

(2) 强制锁

内核强制阻止违反锁规则的访问（如读已加写锁的文件）。需同时满足：

  1. 文件系统挂载时启用 mand 选项（如 mount -o mand /dev/sda1 /mnt）。
  2. 文件设置 setgid 权限且关闭组执行位（如 chmod g+s,g-x file）。

适用场景：严格保证数据完整性（但较少使用，因性能开销大）。

锁的范围

(1) 文件级锁（flock）

锁定整个文件，通过 flock() 系统调用实现。但仅支持劝告锁。并且锁与文件描述符绑定，进程退出或文件描述符关闭时自动释放。支持共享锁（读锁）和独占锁（写锁）。示例：

(2) 记录锁（fcntl）

锁定文件的某一部分（字节范围），通过 fcntl() 系统调用实现。支持劝告锁和强制锁（依赖文件系统配置）。锁与进程绑定，fork 的子进程继承锁，但 exec 后不继承。支持共享锁、独占锁和阻塞/非阻塞模式。示例：

通过 fcntl(F_SETLEASE) 设置，允许进程监控文件是否被其他进程访问。当其他进程尝试打开或修改文件时，持有租借锁的进程会收到信号（SIGIO）。常用于缓存一致性（如 NFS 客户端）。示例：

(2) 锁的管理
  • 全局锁列表：内核维护所有文件锁的链表，存储在 file_lock_manager 中。
  • 冲突检测：加锁时，内核检查请求的字节范围是否与现有锁冲突。冲突规则：

场景 锁类型 说明

单进程写文件 flock(LOCK_EX) 独占写入，阻止其他进程读写。

注意事项：

  1. 死锁风险：若进程 A 持有锁 1 并请求锁 2，而进程 B 持有锁 2 并请求锁 1，会导致死锁。需设计超时或按顺序加锁。
  2. NFS 限制：网络文件系统（NFS）的锁行为可能因实现而异，需额外配置（如 rpc.statd）。
  3. 性能开销：频繁加锁可能降低并发性，应尽量缩小锁范围（如使用记录锁而非文件锁）。

页高速缓存

磁盘高速缓存（Disk Cache） 是提升 I/O 性能的核心机制，通过将磁盘数据缓存在 RAM 中，减少对慢速存储设备的直接访问。其核心组件包括 页高速缓存（Page Cache）、目录项缓存（dentry Cache） 和 索引节点缓存（inode Cache）。

其核心作用在于：

  • 缓存磁盘数据：以内存页（通常 4KB）为单位缓存文件内容，加速重复访问。
  • 减少磁盘 I/O：对同一文件的多次读写操作直接命中缓存，避免物理磁盘访问。
  • 延迟写回（Write-Back）：脏页（被修改的页）异步回写磁盘，合并多次写操作。

在绝大多数情况下，内核在读写磁盘时都引用页高速缓存。新页被加到页高速缓存以满足用户态进程的读请求。如果页不在高速缓存中，新页就被加到高速缓存中，然后用从磁盘读出的数据填充它。如果内存有足够的空闲空间，就让该页在高速缓存中长期保留，使其他进程再使用该页时不再访问磁盘。

同样，在把一页数据写到块设备之前，内核首先检查对应的页是否已经在高速缓存中，如果不在，就要先在其中增加一个新项，并要写到磁盘中的数据填充该项。I/O数的传送并不是马上开始，而是要延退几秒之后才对磁盘进行更新，从而使进程有机会又要写人磁盘的数据做进一步的修改（换句话说，就是内核执行延退的写操作）。

内核的代码和内核数据结构不必从磁盘读，也不必写人磁盘，因此，页高速缓存中的页可能是下面的类型：

  • 含有普通文件数据的页。
  • 含有目录的页。
  • 含有直接从块设备文件（跳过文件系统层）读出的数据的页。内核处理这种页与处理含有普通文件的页使用相同的函数集合。
  • 含有用户态进程数据的页，但页中的数据已经被交换到磁盘，内核可能会强行在页高速缓存中保留一些页面，而这些页面中的数据已经写到交换区（可能是普通文件或磁盘分区）
  • 属于特殊文件系统文件的页，如共享内存的进程间通信所使用的特殊文件系统shm

缓冲区高速缓存的演变

(1) 旧版本（Linux 2.4 之前）

独立的缓冲区高速缓存：缓存文件系统的块（如 inode、目录项），与页高速缓存分离。

问题：同一数据可能同时存在于两个缓存中，导致一致性问题和高内存开销。

(2) 现代内核（2.4.10 及以后）

读文件（read()）的流程

(1) 用户发起请求

(2) 内核处理路径

  1. VFS 层：调用文件对象的 read 方法（通常为 generic_file_read()）。检查页缓存（struct address_space）是否已缓存目标数据页。
  2. 页缓存未命中（Cache Miss）：分配新页框（struct page），加入页缓存。文件系统（如 ext4）定位数据在磁盘上的 物理块号（通过 inode->i_mapping）。
  3. 块设备层：将物理块号转换为 扇区号（如块号 n → 扇区 8n 到 8n+7，假设 512B 扇区）。创建 bio 请求，提交到块设备队列。
  4. 驱动执行：磁盘控制器通过 DMA 将数据从磁盘读取到内存页。中断通知完成，页缓存更新为有效。
  5. 数据拷贝到用户空间：将页缓存中的数据复制到用户缓冲区 buf。

写文件（write()）的流程

(1) 用户发起请求

(2) 内核处理路径

  1. VFS 层：调用文件对象的 write 方法（如 generic_file_write()）。检查页缓存是否存在目标页，若不存在则分配新页。
  2. 数据准备：将用户缓冲区 buf 的数据拷贝到页缓存（写时可能延迟，见下文）。
  3. 文件系统处理：
    • 普通文件系统（如 ext4）：若文件大小扩展，分配新磁盘块（更新 inode->i_blocks）。标记页为 脏页（Dirty），由后台线程（如 pdflush）异步写入磁盘。
    • 日志文件系统（如 ext4 with journal）：先写日志（Journal），再提交实际数据块，确保崩溃一致性。
  4. 块设备层：脏页回写时，生成 bio 请求写入磁盘。

(3) 写模式差异

模式 行为
缓冲写（默认） 数据先写入页缓存，异步回盘。高性能，但崩溃可能丢数据。
同步写（O_SYNC） 数据直达磁盘，阻塞直到写入完成。安全，但性能低。
直接写（O_DIRECT） 绕过页缓存，直接 DMA 到磁盘。需对齐缓冲区，适合数据库。

内存映射

内存映射（Memory Mapping） 是一种将文件或设备直接映射到进程地址空间的技术，允许进程通过访问内存来操作文件数据，无需显式调用 read()/write()。

(1) 共享映射（MAP_SHARED）

对映射内存的修改会同步到磁盘文件，对其他映射同一文件的进程可见。适用于进程间通信（IPC）或持久化共享数据。内核标志：线性区（vm_area_struct）设置 VM_SHARED。

(2) 私有映射（MAP_PRIVATE）

写操作触发 写时复制（Copy-on-Write, COW），修改仅对当前进程可见，不影响磁盘文件。初始数据从文件加载，但后续变更独立。内核标志：线性区清除 VM_SHARED。

内存映射的核心流程

(1) 创建映射（mmap()）

  1. **用户调用 mmap()**：
    指定文件描述符 fd、映射长度、保护权限（如 PROT_READ）和映射类型（MAP_SHARED/MAP_PRIVATE）。
  2. 内核处理：在进程地址空间创建线性区（vm_area_struct），关联文件 inode。建立页表项，但不立即加载文件数据（按需调页）。
  3. 按需调页（Page Fault）：
    当进程首次访问映射区域时：内核分配物理页框，从文件读取对应数据到页缓存。更新页表，建立虚拟地址到物理页的映射。

(2) 写时复制（COW）与私有映射

  • 读操作：直接共享页缓存的只读页。
  • 写操作：
    1. 内核复制目标页到新物理页。
    2. 修改新页，标记为私有，原文件页保持不变。
    3. 更新页表指向新页。

(3) 同步与回写

  • 共享映射：脏页由内核线程（如 pdflush）异步回写到磁盘。可调用 msync() 强制同步。
  • 私有映射：写操作不影响文件，无需回写。

(4) 解除映射（munmap()）

释放线性区，清除页表项。私有映射的修改丢弃，共享映射的脏页回写（若未同步）。

直接I/O传送

我们已经看到，在Linux2.6版本中，通过文件系统与通过引用基本块设备文件上的块，甚至与通过建立文件内存映射访问一个普通文件之间没有什么本质的差异。

但是，还是有一些非常复杂的程序（自缓存的应用程序，self-caching_application）更愿意具有控I?O数据传送机制的全部权力。例如，考虑高性能数据库服务器：它们大都实现了白已的高速缓存机制，以充分挖掘对数据库独特的查询机制。

对于这些类型的程序，内核页高速缓存毫无帮助；相反，因为以下原因它可能是有害的：

  • 很多页框浪费在复制已在RAM中的磁盘数据上（在用户级磁盘高速缓存中）
  • 处理页高速缓存和预读的多余指令降低了read（）和write（）系统调用的执行效率，也降低了与文件内存映射相关的分页操作。
  • read（）和write（）系统调用不是在磁盘和用户存储器之间直接传送数据，而是分
    成两次传送：在磁盘和内核缓冲区之间和在内核缓冲区与用户存储器之间。
  • 因为必须通过中断和直接内存访问（DMA）处理块硬件设备，而且这只能在内核态完成，因此，最终需要某种内核支持来实现自缓存的应用程序。

Linux提供了绕过页高速缓存的简单方法：直接I/O传送。在每次I/O直接传送中，内核对磁盘控制器进行编程，以便在自缓存的应用程序的用户态地址空间中的页与磁盘之间直接传送数据。

我们知道，任何数据传送都是异步进行的。当数据传送正在进行时，内核可能切换当前进程，CPU可能返回到用户态，产生数据传送的进程的页可能被交换出去，等等。这对于普通I/O数据传送没有什么影响，因为它们涉及磁盘高速缓存中的页，磁盘高速缓存由内核拥有，不能被换出去，并且对内核态的所有进程都是可见的。

另一方面，直接I/O传送应当在给定进程的用户态地址空间的页内移动数据。内核必须当心这些页是由内核态的任一进程访问的，当数据传送正在进行时不能把它们交换出去。

在 Linux 中，直接 I/O（Direct I/O） 是一种绕过内核页缓存（Page Cache）的 I/O 传输机制，允许应用程序（如高性能数据库）直接控制数据在 用户空间缓冲区 和 磁盘设备 之间的传输。

特性 说明
绕过页缓存 数据直接传输到用户缓冲区，避免内核缓存的开销和冗余拷贝。
零拷贝（Zero-Copy） 通过 DMA 将磁盘数据直接写入用户空间内存（需缓冲区对齐）。
异步性 I/O 操作由硬件异步执行，进程可继续运行或切换。
适用场景 自缓存应用（如数据库）、大文件顺序访问、需要精确控制 I/O 的场景。

异步I/O

异步 I/O 是一种非阻塞的 I/O 操作模型，允许进程发起 I/O 请求后立即继续执行，无需等待数据传输完成。POSIX 标准定义了一组 AIO 接口（如 aio_read()、aio_write()），而 Linux 也提供了原生内核 AIO（如 io_submit）。

异步 I/O 的工作机制

  1. 请求提交：用户调用 aio_read() 或 io_submit()，内核将请求加入队列。
  2. 异步执行：内核启动 DMA 传输，进程继续运行。
  3. 完成通知：通过事件（如信号、回调或 io_getevents()）通知用户进程。

与同步 I/O 的对比

特性 异步 I/O 同步 I/O
阻塞行为 非阻塞，立即返回。 阻塞直到 I/O 完成。
性能 高并发，适合高吞吐场景。 简单但并发能力低。
复杂度 需处理回调或事件轮询。 逻辑直观。
