设备驱动程序模型

sysfs 文件系统

Linux 的 sysfs 文件系统 是一个虚拟文件系统，用于将内核中的设备、驱动和内核对象的层次结构以文件的形式暴露给用户空间，方便用户查看和配置内核参数。它通常挂载在 /sys 目录下，是 设备模型（Device Model） 的核心组成部分。sysfs 的主要目的是：

  • 展示设备拓扑结构：反映内核中的设备、驱动、总线等关系。
  • 提供内核对象的运行时信息：如设备状态、驱动版本、电源管理等。
  • 允许用户空间与内核交互：通过读写 /sys 下的文件动态修改内核行为（如调整设备参数）。

这里不妨介绍/sys 下的主要子目录及其用途：

目录 描述
/sys/bus 按总线类型（如 PCI、USB）分类的设备。
/sys/class 按功能分类的设备（如网卡、显卡、输入设备）。
/sys/devices 内核设备树的物理层级，包含所有已注册设备。
/sys/fs 文件系统相关的配置（如 cgroup）。
/sys/kernel 内核运行时参数（如调试选项、热插拔配置）。
/sys/module 已加载模块的信息（如参数、状态）。
/sys/power 电源管理选项（如休眠、唤醒）。

核心数据结构

kobject、kset 和 subsystem 是 设备驱动程序模型（Device Model） 的核心数据结构，用于管理内核中的对象层次关系（如设备、驱动、总线等），并通过 sysfs 文件系统向用户空间暴露这些对象的属性和状态。

kobject（内核对象）

基础对象模型，表示内核中的一个实体（如设备、驱动、总线等）。通过引用计数（kref）跟踪对象的存活状态。每个 kobject 对应 sysfs 中的一个目录。

kset（内核对象集合）

将多个 kobject 组织成一个集合（如所有 USB 设备）。支持热插拔事件（uevent）的广播。kset 本身也是一个 kobject，对应 sysfs 中的父目录。

subsystem（子系统）

  • 更高层次的抽象：表示一个内核子系统（如 PCI、USB 总线）。
  • 历史背景：在早期内核中，subsystem 是一个独立结构，但后来被简化为 kset 的封装。
  • 现代实现：通常直接用 kset 替代。

组件构成

设备

表示系统中的硬件设备（如 USB 设备、PCI 网卡）或虚拟设备（如 loopback 设备）。每个设备在内核中对应一个 struct device 结构体，包含设备的物理/逻辑信息（如地址、中断号）。

关键特性

  • 设备树：设备通过父子关系形成层次结构（如 USB 设备挂载在 USB 总线下）。
  • 唯一标识：通过 dev_t（设备号）或 struct device_id 标识。
  • sysfs 接口：设备信息在 /sys/devices/ 下展示。

驱动程序

控制设备的操作（如初始化、读写、中断处理）。每个驱动程序对应一个 struct device_driver 或更具体的结构（如 struct pci_driver）。

关键特性

  • 设备匹配：通过 id_table 或 compatible 字符串绑定设备。
  • 操作集：实现 probe()、remove()、shutdown() 等回调函数。
  • sysfs 接口：驱动信息在 /sys/bus/<bus>/drivers/ 下展示。

总线

管理设备和驱动之间的匹配与交互（如 PCI、USB、I2C 总线）。提供设备枚举、热插拔支持等功能。

关键特性

  • 设备与驱动的匹配：总线的 match() 函数决定驱动是否支持设备。
  • 热插拔事件：通过 uevent 机制通知用户空间（如 udev）。
  • sysfs 接口：总线信息在 /sys/bus/ 下展示（如 /sys/bus/pci/）。

类

按功能对设备分类（如所有输入设备归为 input 类，所有网卡归为 net 类）提供统一的用户空间接口（如 /sys/class/net/eth0）。

关键特性

  • 功能抽象：隐藏硬件差异，提供一致的访问方式（如所有显卡通过 /sys/class/graphics/fb0 操作）。
  • 自动设备节点：通过 udev 在 /dev/ 下创建设备文件（如 /dev/ttyS0）。

设备文件

设备文件（Device File） 是用户空间与硬件设备或内核模块交互的桥梁。它们以文件的形式存在于文件系统中（如 /dev 目录），但实际操作由内核中的设备驱动程序处理。 设备文件的本质：

  • 特殊文件类型：设备文件没有数据块，其索引节点（inode）存储的是 设备标识符 而非文件内容。
  • 系统调用统一性：通过 read()、write()、ioctl() 等标准文件操作接口访问设备，屏蔽硬件差异。

设备文件的分类

块设备文件

支持随机访问（如硬盘、SSD）。数据以固定大小的块（如 4KB）传输。通常用于存储设备，支持缓存和文件系统挂载。

字符设备文件

按字节流顺序访问（如键盘、串口）。不支持随机访问（或效率极低，如磁带机）。无缓存，数据直接传输。

注意：网络设备（如网卡）不通过设备文件访问，而是通过 socket() 系统调用。管理接口位于 /sys/class/net/。

设备号

  • **主设备号：标识设备类型，对应内核中的 设备驱动程序。相同主设备号的设备共享同一组驱动操作（如 /dev/sda 和 /dev/sdb 均由 SCSI 磁盘驱动处理）。
  • 次设备号：区分同一驱动程序管理的不同设备或子设备。例如，主设备号 8 的磁盘驱动可能用次设备号 0、1 表示 /dev/sda 和 /dev/sdb。

设备驱动程序

设备驱动程序是内核例程的集合，它使得硬件设备响应控制设备的编程接口，而该接口是一组规范的VFS函数集（open，reaa，seek，ioct1等等）。这些函数的实际实现由设备驱动程序全权负责。

由于每个设备都有一个唯一的I/O控制器，因此就有唯一的命令和唯一的状态信息，所以大部分IO设备都有自己的驱动程序。

设备驱动程序的种类有很多。它们在对用户态应用程序提供支持的级别上有很大的不同，也对来自硬件设备的数据采集有不同的缓冲策略。这些选择极大地影响了设备驱动程序的内部结构

字符设备驱动程序

处理字符设备相对比较容易，因为通常并不需要复杂的缓冲策略，也不涉及磁盘高速缓存。当然，字符设备在它们的需求方面有所不同：有些必须实现复杂的通信协议以驱动便件设备，而有些仅仅需要从硬件设备的一对IO端口读几个值。例如，多端口串口卡设备（一个硬件设备提供多个串口）的驱动程序比总线鼠标的设备驱动程序要复杂得多。

另一方面，块设备驱动程序本身就比字符设备驱动程序复杂得多。事实上，应用程序可以反复地要求读或写同一个数据块。此外，访问这些设备通常是很慢的。这些特性对磁盘驱动程序的结构产生了深刻的影响。

块设备的处理

扇区

  • 物理扇区：硬盘控制器读写的最小单元，通常为 512字节（现代设备可能为 4KB）。
  • 逻辑扇区：内核统一按 512字节 处理，底层驱动负责物理扇区的转换。
特性 说明
不可分割性 I/O 操作必须整扇区进行（不能读写半扇区）。
寻址方式 通过 sector_t 类型（32/64位）标识扇区下标（如 sector 2048）。
内核统一抽象 无论物理扇区大小如何，内核均按 512字节处理，驱动负责转换。

块

  • 文件系统块：文件系统管理数据的最小单元（如 ext4 默认 4KB），由 多个扇区 组成。
  • 内存页对齐：通常与内存页大小（如 4KB）匹配，方便内核缓存管理。
特性 说明
大小可配置 文件系统格式化时指定（如 mkfs.ext4 -b 4096）。
与扇区关系 1 块 = N 扇区（如 4KB 块 = 8 × 512B 扇区）。
缓存单元 页缓存（struct page）按块缓存磁盘数据。

段

  • 内存连续区：描述 物理内存中连续 的块数据，用于 DMA 传输。
  • bio_vec 结构：内核通过 bio_vec 表示段，包含物理页、偏移和长度。
特性 说明
DMA 友好 段对应物理连续的地址范围，可直接用于设备 DMA。
与 bio 的关系 一个 bio 包含多个 bio_vec（段），支持分散/聚集（Scatter-Gather）I/O。
高端内存支持 段可指向高端内存页，按需映射到内核地址空间。

通用块层

通用块层是一个内核组件，它处理来自系统中的所有块设备发出的请求。由于该层所提供的函数，内核可以容易地做到：

  • 将数据缓冲区放在高端内存仅当CPU访问其数据时，才将页框映射为内核中的线性地址空间，并在数据访问完后取消映射。
  • 通过一些附加的手段，实现一个所谓的“零一复制”模式，将磁盘数据直接存放在用户态地址空间而不是首先复制到内核内存区：事实上，内核为I/O数据传送使用的缓冲区所在的页框就映射在进程的用户态线性地址空间中。
  • 管理逻辑卷，例如由LVM（逻辑卷管理器）和RAID（廉价磁盘余阵列）使用的逻辑卷：几个磁盘分区，即使位于不同的块设备中，也可以被看作是一个单一的分
    区。
  • 发挥大部分新磁盘控制器的高级特性，例如大主板磁盘高速缓存、增强的DMA性能、IO传送请求的相关调度等等。

I/O调度程序

在 Linux 内核中，I/O 调度程序（I/O Scheduler） 是块设备层（Block Layer）的核心组件，负责优化磁盘 I/O 请求的执行顺序，以减少磁头移动（寻道时间）并提升吞吐量。以下是其工作原理和关键机制的详细解析：

1. I/O 调度的核心目标

  • 合并相邻请求：将多个连续的扇区访问合并为单个 I/O 操作。
  • 排序请求：按磁头移动的最优顺序（如电梯算法）重新排列请求。
  • 延迟执行：通过暂存请求，寻找更优的合并或排序机会。

2. 请求队列（Request Queue）

每个块设备（如 /dev/sda）维护一个独立的请求队列（struct request_queue），存储待处理的 I/O 请求（struct request）。

3. I/O 调度的工作流程

(1) 请求的产生

当用户或内核发起 I/O 操作（如 read()）时：

  1. VFS 将请求封装为 struct bio（Block I/O）。
  2. 通用块层 将 bio 插入请求队列，触发 I/O 调度器。
(2) 调度器的处理

调度器对请求进行以下操作：

  • 合并（Merge）：
    若新请求的扇区与队列中某个请求连续，则合并为一个请求。
  • 排序（Sort）：
    按调度算法（如 CFQ、Deadline）重新排列请求，减少磁头移动。
  • 插入队列：
    若无法合并，将新请求加入队列。
(3) 驱动处理请求
  • 策略例程（Strategy Routine）：
    驱动通过 request_fn 从队列获取请求，发送命令给磁盘控制器。
  • 中断处理：
    磁盘完成 I/O 后触发中断，驱动继续处理下一个请求。

4. 主要 I/O 调度算法

Linux 支持多种调度算法，可通过 echo SCHEDULER > /sys/block/sda/queue/scheduler 切换。

(1) CFQ

为每个进程分配独立的 I/O 队列，按时间片公平调度。适用场景：
多任务桌面环境（默认算法，Linux 5.0 前）。

(2) Deadline（截止时间）

为读/写请求设置截止时间，优先处理可能超时的请求。优势：
避免写请求饿死，适合数据库等高负载场景。

(3) Noop（No Operation）

简单 FIFO 队列，不排序不合并。适用场景：
SSD（无需磁头寻道优化）或虚拟化环境（底层已优化）。

(4) BFQ（Budget Fair Queuing）

基于 CFQ 改进，更精细的带宽分配。优势：
适合实时性要求高的应用（如多媒体处理）。

块设备驱动程序

块设备

块设备驱动的多设备管理能力

一个驱动管理多个物理设备： 例如，一个 IDE 驱动程序可以同时控制多个 IDE 硬盘（如 /dev/hda、/dev/hdb）。每个硬盘是一个独立的 物理块设备，但由同一个驱动程序处理。

  • 驱动代码复用：驱动程序通过 次设备号 区分不同设备（如 hda 主设备号 3，次设备号 0；hdb 主设备号 3，次设备号 1）。

逻辑设备（分区）的抽象：每个物理磁盘可以划分为多个分区（如 /dev/sda1、/dev/sda2），每个分区被视为独立的 逻辑块设备：

  • 分区表的作用：内核读取分区表后，为每个分区注册独立的 gendisk 结构。
  • 次设备号分配：主设备号标识驱动，次设备号标识分区（如 8:0 是 /dev/sda，8:1 是 /dev/sda1）。
块设备驱动与 VFS 的交互

VFS 系统调用的处理：当用户空间通过 设备文件（如 /dev/sda1）发起系统调用（如 open()、read()、write()）时：

  1. VFS 层：将文件操作路由到块设备驱动的默认操作表 def_blk_fops。
  2. 驱动层：驱动通过 struct block_device_operations 实现具体功能：
    • open()：初始化设备或分区。
    • read_iter()/write_iter()：处理读写请求（最终转为 Bio 结构提交到请求队列）。
  3. 硬件交互：驱动通过 DMA 或 PIO 与磁盘控制器通信。
关键的数据结构
  • **struct block_device**：代表内核中的块设备实例
  • **struct gendisk**：描述物理磁盘的属性（如容量、分区表）
  • **struct request_queue**：管理待处理I/O请求的队列
  • **struct bio**：描述一个块I/O操作的基本单元

访问块设备

当用户通过不同路径（如 /dev/sda1 和 /mnt/disk/partition）访问 同一块设备（相同主/次设备号）时：

  • VFS 视角：这两个路径是独立的文件，拥有不同的 dentry 和 inode。
  • 物理设备视角：它们实际指向同一个硬件设备（如磁盘分区）。

内核需要确保：

  1. 避免重复初始化：同一设备只创建一个 struct block_device 描述符。
  2. 正确引用计数：跟踪设备的打开次数，直到最后一个用户关闭后才释放资源。
内核的解决方案

(1) struct block_device 的全局管理

  • bdev 文件系统：内核通过虚拟文件系统 bdev 统一管理所有块设备的 block_device 对象。
  • 哈希表查找：使用设备号（dev_t）作为键，在全局哈希表 bdev_hashtable 中快速查找是否已存在对应的 block_device。

(2) 关键函数：blkdev_get_by_dev()

当打开块设备文件时，内核调用此函数：

工作流程：

  1. 检查哈希表是否已存在 dev 对应的 block_device。
  2. 若存在：增加引用计数。返回现有对象。
  3. 若不存在：创建新的 block_device。初始化设备（如调用驱动的 open() 方法）。添加到哈希表。

打开块设备文件

打开块设备文件 是访问物理存储设备（如磁盘、分区）或逻辑设备（如 LVM 卷）的关键步骤。无论是挂载文件系统、激活交换分区，还是用户直接操作设备文件（如 open("/dev/sda1")），内核均遵循统一的处理流程。

当内核需要打开块设备文件时，依次执行以下操作：

  1. 路径解析：通过 VFS 找到设备文件对应的 inode。
  2. 设备描述符管理：获取或创建 struct block_device 描述符。
  3. 初始化访问上下文：设置文件操作方法和硬件准备。
  4. 返回文件描述符：供后续读写操作使用。
