系统调用与信号

系统调用

为了把系统调用号与相应的服务例程关联起来，内核利用了一个系统调用分派表。这个表存放在sys_call_table数组中，有NR_syscalls个表项（在Linux2.6.11内核中是289）：第个n表项包含系统调用号为n的服务例程的地址

进入和退出系统调用

当用户态程序（如本地应用）需要请求内核服务（如读写文件、创建进程等），必须通过系统调用切换到内核态。Linux 提供了两种主要的进入方式：

传统方式：int 0x80（软中断）通过触发一个软中断（中断号 0x80），CPU 切换到内核态并执行系统调用处理程序。在早期 Linux 内核中是唯一方式。兼容性强，但性能较低（因为涉及中断上下文切换）。

现代方式：sysenter（快速系统调用）Intel Pentium II 引入的指令，直接跳转到内核预设的入口点，无需中断,速度更快（避免了中断开销）。需要 CPU 和内核同时支持。

系统调用的退出方式:内核完成系统调用后，需要切换回用户态，同样有两种方式：

  • 传统方式：iret（中断返回）从中断返回，恢复用户态上下文。兼容性强，但性能较低。
  • 现代方式：sysexit（快速返回）与 sysenter 配对使用，快速返回到用户态。仅适用于 sysenter 进入的系统调用。需要 CPU 支持。

参数传递

普通 C 函数的参数传递是通过栈（Stack）传递（如 x86 的 push 指令）。其特点在于：

  • 调用者（Caller）将参数压入当前栈（用户态栈或内核态栈）。
  • 被调函数（Callee）从栈中读取参数。

而系统调用的参数传递是通过寄存器传递，再由内核复制到内核栈。原因在于用户态不能直接访问内核栈，因此无法直接跨栈传递。而内核需要确保用户态数据的安全性和隔离性。

为什么内核不直接跨栈复制参数？

(1) 用户态栈和内核态栈是隔离的

用户态程序无法直接访问内核栈（内存隔离是操作系统的核心安全机制）。如果允许直接跨栈复制，可能引发安全问题（如用户态伪造内核栈数据）。

(2) 同时操作两个栈复杂度高

若内核尝试直接从用户栈读取数据：需检查用户栈地址是否合法（防止恶意指针）。需处理页面错误（用户栈可能被换出到磁盘）。而通过寄存器传递更简单、高效。

(3) 与异常处理机制保持一致

系统调用的底层实现（如 int 0x80/sysenter）和硬件异常（如缺页中断）使用相同的寄存器保存机制。内核用统一的 SAVE_ALL 宏保存寄存器上下文，简化了处理逻辑。

为了使用寄存器传递参数必须满足：

  • 每个参数 ≤ 32 位（寄存器宽度）。
  • 参数数量 ≤ 6 个（x86 通用寄存器数量有限）。如果超过 6 个参数，用一个寄存器（如 ebx）指向用户态内存块（结构体或数组）。内核会安全地从用户态内存复制数据（通过 copy_from_user()）。

内核如何获取参数？

  1. 用户态通过 int 0x80/sysenter 触发系统调用。

验证参数

Linux系统调用中的参数验证是内核在执行系统调用服务例程前对用户空间传入参数进行的一系列安全检查机制，旨在确保系统稳定性、安全性和可靠性。当用户态程序通过软中断、syscall指令等方式发起系统调用时，传递的参数指针指向用户空间内存，内核必须验证这些参数的有效性后才能使用，否则可能导致内核崩溃、权限提升或信息泄漏等严重问题。

参数验证的核心内容包括指针有效性检查、数据类型验证和权限审查。指针有效性检查通过access_ok()宏验证用户空间指针是否属于当前进程的合法地址范围且对齐正确，避免内核直接解引用用户指针导致页面错误。数据类型验证确保参数值在合理范围内，如文件描述符是否为有效值、标志位是否包含未知位、缓冲区大小是否超出限制等。权限审查通过capable()函数检查进程是否具备执行相应操作的特权，如CAP_SYS_ADMIN等。

具体的验证机制采用逐层防御策略：在系统调用入口处进行基础验证，在子系统层面进行语义验证，在驱动层面进行硬件相关验证。现代Linux还引入了SMAP（Supervisor Mode Access Prevention） 和SMEP（Supervisor Mode Execution Prevention） 硬件特性，防止内核意外访问用户空间数据或执行用户空间代码。对于复杂结构体参数，内核使用copy_from_user()、get_user()等专用函数安全地复制数据到内核空间后再进行处理，确保整个系统调用执行过程在可控的安全边界内进行。

信号

Linux中的信号是一种异步进程间通信和事件通知机制，主要用于在特定事件发生时向进程发送简短消息以改变其正常执行流程。 信号的核心作用包括进程控制、异常通知和简单通信：

  • 当用户按下Ctrl+C时，内核向 foreground 进程发送SIGINT信号请求终止执行；
  • 当子进程退出时，内核向父进程发送SIGCHLD信号通知状态变化；
  • 当进程执行非法操作（如访问无效内存）时，内核发送SIGSEGV信号使其崩溃并生成core dump。信号分为标准信号（32种）和实时信号（32种），前者为不可靠信号可能丢失，后者支持排队确保可靠送达。

每个信号具有三种处理方式：默认行为（SIG_DFL）、忽略信号（SIG_IGN）或自定义信号处理函数。进程通过signal()或sigaction()系统调用注册处理函数，当信号抵达时，内核将暂停进程当前执行、切换到信号处理函数、执行完毕后再恢复原有流程，这种中断-响应机制类似于硬件中断的软件实现。信号还用于实现作业控制，如SIGSTOP和SIGCONT分别用于暂停和恢复进程执行；系统管理员可通过kill命令向指定进程发送信号实现进程管理；在多线程环境中，信号处理变得更加复杂，可配置为发送到特定线程或整个进程。尽管信号机制因历史原因存在一些设计缺陷（如信号掩码竞争条件、处理函数可重入性问题），但它仍然是Linux系统中实现异步事件处理的基础设施。

产生信号

Linux中信号的产生来源多样，主要包括硬件异常、终端交互、软件条件触发和主动请求四大类别。 当进程执行非法操作（如访问无效内存地址、执行非法指令、算术运算溢出）时，CPU会触发硬件异常，内核随即捕获这些异常并转换为相应的信号发送给进程，例如访问无效内存产生SIGSEGV信号，除零错误产生SIGFPE信号。终端驱动程序在检测到特殊控制字符（如Ctrl+C产生SIGINT、Ctrl+\产生SIGQUIT、Ctrl+Z产生SIGTSTP）或终端断开连接时（产生SIGHUP）也会自动生成信号。

软件条件触发的信号由内核在特定系统状态变化时产生：当子进程终止时向父进程发送SIGCHLD，当进程设置的定时器（alarm/setitimer）超时时发送SIGALRM，当进程尝试对已关闭的管道进行读写时发送SIGPIPE，当进程超出CPU时间限制（RLIMIT_CPU）或文件大小限制（RLIMIT_FSIZE）时分别发送SIGXCPU和SIGXFSZ。主动请求产生的信号则通过系统调用接口实现：kill()系统调用允许进程向自身或其他有权限的进程发送任意信号，raise()用于向当前进程发送信号，tgkill()可向特定线程发送信号，sigqueue()支持附带额外数据的实时信号发送。此外，某些特殊系统调用如abort()产生SIGABRT、pthread_kill()产生线程定向信号，构成了完整的信号产生体系，使Linux能够灵活处理各种异步事件和异常情况。

传递信号

Linux中信号的传递是内核将产生的信号投递到目标进程并触发相应处理的过程，这一机制涉及复杂的状态管理和权限控制。 当信号产生后，内核不会立即将其传递给目标进程，而是先挂起在该进程的未决信号集（pending signal set）中，等待合适的执行时机。内核在完成系统调用、处理硬件中断返回用户空间前，或进程从睡眠状态被唤醒时，会检查当前进程的未决信号集，若存在非阻塞信号则准备传递。

信号传递的核心在于上下文切换：内核首先保存进程当前的执行上下文（包括寄存器状态、程序计数器等），然后根据信号处理方式决定下一步动作。若进程将信号处理设置为忽略（SIG_IGN），内核直接清除该信号的未决状态；若设置为默认动作（SIG_DFL），内核执行预定义行为（如终止、停止、继续或忽略）；若注册了自定义信号处理函数，内核则临时切换到用户空间的信号处理函数执行。在切换到处理函数前，内核会在用户栈上构建新的调用帧，确保处理函数执行完毕后能通过sigreturn()系统调用恢复原始执行上下文。

信号传递的复杂性体现在多个方面：

  • 进程可通过信号掩码（signal mask）阻塞特定信号，被阻塞的信号将保持未决状态直至解除阻塞；
  • 实时信号与非实时信号在传递时存在排队差异，实时信号支持按顺序传递多个相同信号，而非实时信号可能合并；
  • 多线程环境中，信号可配置为传递到特定线程或整个进程，这由线程组的信号共享策略决定；
  • 内核还必须处理权限验证，确保发送者具备向目标进程传递信号的权限（相同用户或root权限）。整个传递过程在保证系统安全性的前提下，实现了异步事件的及时响应和处理。

与信号处理相关的系统调用

系统调用 主要功能描述 常用场景
signal() 为指定信号设置处理函数（简单、古老，可移植性差）。 早期的简单信号处理，如捕获 SIGINT (Ctrl+C)。
sigaction() 为指定信号设置处理行为（功能强大、现代，推荐使用）。可精确控制信号处理时的行为。 现代程序中进行可靠的信号处理，需要控制信号掩码、标志等。
sigprocmask() 用于检查或修改进程的信号掩码（即当前被阻塞的信号集合）。 在关键代码段临时阻塞某些信号，防止其中断。
sigpending() 检查当前进程被阻塞且正在等待处理的信号集合（未决信号集）。 诊断或检查是否有信号在等待传递。
sigsuspend() 临时用一个指定的信号掩码替换当前掩码，并挂起进程，直到一个信号被捕获。 原子性地解除对某个信号的阻塞并等待其发生，避免竞争条件。
kill() 向指定进程（或进程组）发送一个信号。 终止进程（SIGTERM, SIGKILL），或进行进程间简单通信。
tgkill() 向指定进程中的特定线程发送信号。 多线程程序中精确地向某个线程发送信号。
sigqueue() 向指定进程发送一个信号，并可以附带一个整型值或指针数据。 主要用于实时信号，实现带简单数据的进程间通信。
alarm() 设置一个实时定时器，在指定秒数后向进程发送 SIGALRM 信号。 实现简单的超时机制。
pause() 挂起调用进程，直到任何信号被捕获。 等待一个异步事件的发生。
abort() 向进程自身发送 SIGABRT 信号，通常会导致进程异常终止并生成 core dump。 在检测到严重程序错误时（如断言失败）主动终止程序。
sigreturn() 不应直接调用。在用户态信号处理函数执行完毕后，由 glibc 封装并调用此系统调用，以恢复进程被信号中断前的上下文。 信号处理机制内部的关键环节，由库函数自动处理。
