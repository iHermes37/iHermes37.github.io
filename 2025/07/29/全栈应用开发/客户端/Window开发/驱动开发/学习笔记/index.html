Windows过滤平台

Windows过滤平台（WFP） 是Windows操作系统提供的一套网络数据包过滤机制，旨在提供比NDIS更高层次的网络流量控制和管理。WFP不仅支持应用层（Layer 7）协议过滤，还能支持更低层次（Layer 3/4）的网络流量过滤。

  • NDIS协议驱动 更侧重于硬件与操作系统协议栈之间的低层交互。
  • WFP 是基于更高层次的过滤平台，提供更加灵活且复杂的流量控制和管理机制，适用于网络安全、流量监控等需求。

NDIS协议驱动

NDIS (Network Driver Interface Specification) 是微软为网络驱动程序定义的标准接口，用于在操作系统内核中提供网络设备和协议栈之间的通信。NDIS协议驱动通常用于网络适配器、协议栈及网络过滤等功能的实现。

中断基础知识

中断描述符表(IDT)，是中断处理机制中最重要的一个数据结构。每个中断向量或异常在系统中都有对应的服务例程，那么IDT就是用来记录中断处理程序入口地址的线性表。

中断描述符表(IDT)中最多可容纳256个门描述符，每个中断门描述符占8个字节，中断门描述符中含相应中断向量或异常的服务例程地址。由于中断门描述符长度固定，而每个中断或异常都有系统分配的向量号，该号与中断门描述符在IDT中的位置一致。那么根据中断向量号很快就能在IDT中找到服务例程地址。

中断处理过程

处理器处理中断的过程与使用call指令调用某函数的过程相似。在响应中断的时候，处理器以中断向量号为索引找到IDT中的中断门描述符。每个中断门都有相应的运行在当前执行任务上下文中的服务例程。在中断门描述符中的段选择子指向GDT或LDT中的一个执行代码段，偏移则为中断处理例程的入口地址。

64位模式下的中断处理机制

在64位模式下，中断处理机制与非64位模式有着以下的不同：

  • IDT中指向的所有中断处理程序都使用64位代码。
  • 中断压栈的长度固定为64位，处理器使用8个字节、0扩充的存储方式。
  • 栈指针无条件地被压入栈，而在传统模式下，压栈是根据当前特权等级(CPL)有条件地入栈。
  • 如果CPL有变化，则新的SS被置为空。
  • iret指令有所变化。
  • 使用新的中断栈切换机制。
  • 中断栈帧的对齐方式不同

开发者可以使用int 3软中断来实现断点，而实际上，3号中断的确是被用来作为软件调试断点的。不知道读者有没有遇到过这样的情形：当想实时调试某款收费软件时，可能会发现WinDbg附加进去之后无论如何都断不下来，没法继续往下调试。这是什么原因呢？这是因为一些软件为了防止被调试分析，专门添加了一些防调试的技术，这样可以在一定程度上保护软件的代码逻辑以及数据不被外泄。防调试的方法有很多，这些方法并非本节讨论的重点，本节只为读者介绍通过IDT Hook来实现防调试。

权限级别的切换

实际上，要通过中断门或者调用门，才能从R3级进入R0级（在出现sysenter/syscall指令之前），而且进入的地址是填写在“门”记录中的。而这些门记录，都写在系统内存区（只有R0级才能访问）。这样作为R3级的代码，虽然可以切换到R0级，但是只能切换到操作系统已经规定的那些地址。

这样一来，操作系统就可以事先填写一组调用门，并保证切换后的地址都是操作系统的“系统调用”，来防止普通用户随意切换到自己编写的R0级的代码。

Windows内核挂钩

内核挂钩的种类极多，数得出的估计有以下几类：系统服务描述符表挂钩(SSDT Hook)、函数导出表挂钩(Export Table Hook)、中断挂钩（包括IDT Hook和IOAPIC Hook）、分发函数挂钩(Dispatch Function Hook)、系统调用入口挂钩(Syscall Entry Hook)、内联挂钩(Inline Hook)、调试挂钩(Debug Hook)。此外，肯定有其他更多不知名的挂钩方式。但总而言之，以上这些挂钩可以分为如下两类。

硬件挂钩：是利用硬件特性进行的挂钩。比如中断挂钩(IDT&IOAPIC)、系统调用入口挂钩、调试挂钩。中断挂钩利用修改IDT或者IOAPIC设置来进行中断服务函数的挂钩。而这些都是硬件特性，任何操作系统都可以用这样的方式来进行挂钩。

软件挂钩：是利用操作系统内核或者其他软件的特性进行的挂钩。比如系统服务描述符表挂钩。虽然各种操作系统都有系统服务描述表或者其类似物，但是这并不是一个硬件规范，所以在Windows内核上做的系统服务描述符表挂钩是依赖Windows的特性而存在的。此外，内联挂钩是直接修改软件中的可执行代码而“嵌入”的挂钩，因此也是软件挂钩。

系统服务描述符表挂钩

系统服务描述符表是Windows内核中的一个数据结构，这个结构保存了内核导出的一系列供用户态调用的函数地址。

JakubGlisz/GetSSDT: Finding SSDT address on windows 10 64bit

系统服务描述符表的地址可从内核导出的一个结构KeServiceDescriptorTable中获得，它主要记录了系统服务描述符表的基址和提供的系统服务函数的数量。

通过修改系统服务描述符表中的函数指针，安全软件就可以用自己提供的另一个版本的ZwCreateFile来替换掉系统中原来的那个。这种替换行为就是一种挂钩。

在Windows XP系统的机器上。在64位的Windows中有了补丁卫哨之后也并不是不能改，只是改过之后，Windows事后会进行检查，发现有改动就会对系统直接蓝屏

当然，木马也可以用同样的方法来保护自己。如果考虑到安全软件可能会打开某个目录来检查目录下是否有一个带有特征码的木马，就可以通过挂钩ZwCreateFile来察觉到这个问题（要搜索某一个目录的话，势必要先打开这个目录）。

之后可以再进行一连串的挂钩，比如挂钩ZwQueryDirectoryFile。每当有人要查询一个目录下所有的文件和子目录时，这个函数就会被调用返回。木马可以在挂钩了这个函数之后，在处理中首先调用原始版本的ZwQueryDirectoryFile获得返回的文件列表，从中小心地删除掉要保护的那个文件的信息，这样就造成了一种这个文件被隐藏起来的效果。也可以用来防止被木马扫描程序扫描到。当然，这种手段因为太过陈旧，现在已经无济于事了。

函数导出表挂钩

除了系统服务描述符表里的函数指针之外，Windows内核中还有大量的函数。这些函数和上一节中介绍的NtCreateFile之类的函数之间并没有本质区别，只不过没有放到系统服务描述符表里。总而言之，内核函数可以分为以下几类。

第一类是上一节描述过的系统服务。如果一个内核函数作为一个系统服务，那么就可以被用户态程序从系统服务接口直接调用，当然也可以从内核中直接调用。这类函数理论上是最开放的，但是令人惊奇的是，也有一部分微软未公开。

第二类是在WDK的头文件中公开出来可以供调用的函数，在帮助文档中可以找到详细的资料。比如IoCreateFile和IoCallDriver。IoCreateFile是之前的NtCreateFile会调用的一个更底层的用来打开文件的函数。

而IoCallDriver是Windows的一个非常重要的函数，在这里可以过滤到所有的请求(IRP)，以及是发送给哪个设备的。这个调用对Windows内核而言就像是人体的血管或者神经一样，负责输送最重要的信息

在没有补丁卫哨的情况下，任何能够找到地址的函数都可以实现挂钩。问题只在于如何找到函数的地址，以及如何挂钩比较简单。上面的系统服务都是用MmGetSystemRoutineAddress找到地址，然后通过修改系统服务描述符表中的一个函数指针很简单地完成了挂钩。现在开始使用IoCallDriver来介绍其他种类内核函数的挂钩方法。

前面说过IoCreateDriver函数很重要，能够捕获到所有请求的发送过程。这就可以搜集到很多信息，比如谁读/写了文件、谁动用了磁盘等，一般这些情况无一不是通过发送请求来表达的（请注意，例外也是存在的）。这样就有人想到通过挂钩IoCallDriver来开发一个“监控器”，这个监控器能够记下所有请求的发送过程，有助于分析系统的情况。这样的工具的确存在，比如IRPTrace软件。

Windows回调机制

回调机制为驱动程序提供了一种通用的方法来发送和接收某类通告，这些通告可以是系统某个部件的状态发生变化而产生的通告，也可以是开发者自定义的某个条件因为条件成立而产生的通告。

回调对象

下面介绍回调的工作过程。当系统或者开发者为驱动程序提供（发送）某一类通告时，需要创建一个回调对象A，如果驱动程序接收这类通告，则需要打开回调对象A，成功打开回调对象A之后，驱动程序还需要把一个回调例程“注册”到回调对象A上。通过这些操作后，回调对象A和驱动程序的回调例程被关联起来，当系统或者开发者发送通告时，就可以通过回调对象A发起通告，最后回调对象A关联的回调例程就会被调用。

保护进程

具体的一个内核对象实际上对应了一块内核态的内存，这块内存描述了这个对象的具体信息。

不同系统之间这个结构的定义可能存在差异，所以在使用这个结构之前，一定要把不同系统之间的差异分析清楚。

试想一下，如果有一个恶意程序，在安全软件启动之前创建了相同名字的内核对象，或者说恶意程序打开安全软件进程创建的内核对象，然后修改对象的状态（或修改内存映射文件的内容）等，这样的一系列操作都会破坏安全软件的运行的正确性。所以，自身安全也是安全软件的一个重要功能。习惯上把上述恶意程序的行为称为“对象劫持”，把对安全软件自身安全的加固和保护称为“自保护”。

保护内核对象

处理对象的打开

要操作一个内核对象，首先要打开这个内核对象（或者通过句柄复制，后面讨论），所以保护一个对象最简单的方法就是防止这个对象被恶意进程打开。

如何防止一个内核对象不被其他进程打开？答案是挂钩打开内核对象的相关函数，本书第23章已经介绍了系统服务描述符表挂钩以及内联挂钩。在挂钩的处理函数中，判断当前需要打开的内核对象名字是否是需要保护的内核对象名字，如果是，再判断打开该内核对象的进程是否是安全（即受信任）的进程，如果是非安全的进程打开受保护的内核对象，则需要在钩子处理函数中阻止这个操作。

不同的内核对象，打开对象所使用的API不同，对于事件对象，打开内核对象使用NtOpenEvent函数；而对于内存映射文件，打开内核对象使用NtOpenSection函数；针对保护不同的内核对象，需要挂钩不同的函数。

Windows系统提供了一种被称为“句柄复制”的操作。简单来说，通过句柄复制可以把一个进程句柄表中的句柄复制到另一个进程的句柄表中

从上面三个思路来看，思路一是在具体的行为操作上进行拦截，而思路二和思路三是从“源头”上进行拦截。

从获取句柄的途径上看，除了前面介绍的打开操作以及复制操作外，还存在一个途径就是通过“继承”获取句柄。一般来说，继承发生在父子进程间

恶意程序可以通过属性列表的方式，在创建子进程时，指定从安全进程中继承句柄，然后通过继承的句柄来恶意操作内核对象

进程的保护

进程对象也是一种内核对象，对进程的各种操作也需要通过句柄来进行，和保护其他内核对象一样，保护进程对象的方法也是通过在获取进程句柄时进行拦截和保护的。

进程对象也是一种内核对象，对进程的各种操作也需要通过句柄来进行，和保护其他内核对象一样，保护进程对象的方法也是通过在获取进程句柄时进行拦截和保护的。从前面的介绍可知，获取进程句柄也有三大途径：一是通过打开的方式获取；二是通过复制句柄的方式获取；三是通过继承句柄的方式获取。
