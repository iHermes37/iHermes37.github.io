共识机制-POS

权益证明

验证者选举

准入控制

这个函数 ProcessDeposits 是以太坊信标链（Beacon Chain）中处理验证者存款（Validator Deposits）的核心逻辑，属于权益证明（PoS）共识机制的关键组成部分。它的主要功能是批量处理来自以太坊 1.0 存款合约的质押交易，将符合条件的质押者加入信标链的验证者队列。

ProcessDeposit 是信标链处理单个验证者存款的核心函数，负责：

  1. 验证存款合法性（Merkle 证明、签名等）。
  2. 更新 Eth1 存款索引（确保顺序处理）。
  3. 应用存款到状态（新增验证者或更新余额）。

核心作用：将合法的 Eth1 存款转换为信标链验证者状态变更，是 PoS 准入控制的核心环节。

ApplyDeposit 是信标链处理存款的最终执行层，负责将验证者存款数据写入区块链状态，具体分为：

  • 新增验证者（首次质押）
  • 追加余额（已有验证者增持）

AddValidatorToRegistry 是信标链中将新验证者正式注册到状态的核心函数，完成以下操作：

  1. 初始化验证者对象：根据存款数据生成验证者实例。
  2. 更新状态数组：将验证者追加到信标链的全局状态中。
  3. 处理 Altair 硬分叉字段：为升级后的网络初始化新增字段。

激活验证者

ProcessRegistryUpdates 是信标链中管理验证者生命周期状态转换的核心函数，负责处理以下三类验证者状态变更：

  1. 加入激活队列：满足条件的质押者进入激活等待列表。
  2. 驱逐低余额验证者：质押余额不足的活跃验证者被强制退出。
  3. 激活验证者：等待期满的验证者正式加入共识网络。

区块提议

选举机制

每个时隙（Slot，12秒）通过 RANDAO + LMD-GHOST 随机选择一个验证者作为提议者。

ComputeProposerIndex 是信标链中选举区块提议者（Block Proposer）的核心算法，其通过以下步骤实现：

  1. 遍历活跃验证者列表：按伪随机顺序检查每个验证者。
  2. 权重比较：根据验证者的 EffectiveBalance 计算选择概率，与随机数比较决定是否选中。
  3. 版本兼容：支持 Electra 硬分叉前后的不同参数（如 MaxEffectiveBalanceElectra）。

职责

在 validator.ProposeBlock 函数中，区块提议者的职责被分解为以下步骤：

  1. 准备阶段：
    • 检查当前 slot 有效性（跳过 genesis slot）。
    • 加锁防止重复提案（async.NewMultilock）。
  2. 生成 Randao Reveal：
    • 签署 epoch 相关的随机数，用于后续区块请求（signRandaoReveal）。
  3. 获取区块数据：
    • 向信标节点请求区块模板（包含交易、投票等）（BeaconBlock 请求）。
  4. 签名与构建：
    • 对区块进行签名（signBlock），并构建完整签名区块（BuildSignedBeaconBlock）。
  5. 防罚没检查：
    • 验证提案是否违反罚没规则（SlashableProposalCheck）。
  6. 广播区块：
    • 提交签名区块到网络（ProposeBeaconBlock）。

GetPayload 是共识层（信标链）从执行层（如 Geth）获取已打包交易数据（ExecutionPayload）的核心方法，其工作流程如下：

  1. 发起 RPC 请求：通过 Engine API 调用执行层客户端的 engine_getPayloadV1。
  2. 接收预构建区块：获取执行层生成的交易列表、状态根等数据。
  3. 返回格式化响应：封装为 GetPayloadResponse 供共识层使用。

罚没

  1. 罚没检测：
    • CheckProposerSlashing / CheckAttesterSlashing（检测双重提案或投票）。
  2. 惩罚执行：
    • ProcessSlashings / SlashValidator（扣除余额并强制退出）。
  3. 怠惰惩罚：
    • ProcessInactivityScores / ProcessInactivityPenalties（惩罚长期离线验证者）。
  4. 举报机制：
    • SubmitProposerSlashing / SubmitAttesterSlashing（链上提交证据）。

功能概述

  1. 作用
    在每次 epoch 转换时，计算并更新验证者的余额，包括：
    • Attester（参与见证）的奖励与惩罚：根据验证者是否及时提交正确的见证（attestations）。
    • Proposer（区块提议者）的奖励：对成功打包区块的验证者给予奖励。
    • 惩罚：对恶意行为（如双重投票）或怠惰（未参与）的验证者扣除余额。
  2. 触发时机
    在 ProcessEpoch 中被调用（每个 epoch 一次，约 6.4 分钟）。

奖励

1. 提议者（Proposer）奖励逻辑

文件路径：beacon-chain/core/epoch/precompute/proposer.go

关键函数：proposerRewards
功能：计算区块提议者因打包交易和见证（attestations）所获得的奖励。

逻辑说明：

  • 提议者的奖励与打包的 有效见证数量 成正比。
  • BaseReward 由验证者的有效余额和全网质押量动态计算（见 baseReward 函数）。

2. 验证者（Attester）奖励逻辑

文件路径：beacon-chain/core/epoch/precompute/attester.go

关键函数：attesterRewards
功能：计算验证者因正确提交见证（attestations）所获得的奖励或惩罚。

逻辑说明：

  • 奖励条件：验证者需在 source、target、head 三个检查点正确投票（通过 CorrectlyVoted 标记）。
  • 惩罚条件：活跃但未投票的验证者会被扣除 BaseReward。
  • 动态权重：奖励按验证者的 有效余额占比 分配（EffectiveBalance / TotalStake）。

3. 基础奖励计算

文件路径：beacon-chain/core/epoch/precompute/balance.go

关键函数：baseReward
公式：
[
\text{BaseReward} = \frac{\text{EffectiveBalance} \times \text{BaseRewardFactor}}{\sqrt{\text{TotalStake}}}
]
代码实现：

参数：

  • BaseRewardFactor：协议常数（Altair 硬分叉后为 64）。
  • EffectiveBalance：验证者的有效质押余额（上限 32 ETH）。

4. 奖励分配的入口函数

文件路径：beacon-chain/core/epoch/precompute/rewards_penalties.go

关键函数：ProcessRewardsAndPenaltiesPrecompute
流程：

  1. 调用 attesterRewards 和 proposerRewards 计算奖励/惩罚。
  2. 更新验证者余额（见问题中给出的代码）。

5. 状态更新

文件路径：beacon-chain/state/state.go

关键函数：SetBalances
作用：将计算后的余额写入区块链状态。

总结：奖励分配流程

  1. 提议者奖励：
    • 打包区块 → 获得 BaseReward * IncludedAttestations。
  2. 验证者奖励：
    • 正确投票 → 获得按质押比例分配的 BaseReward。
    • 未投票 → 扣除 BaseReward（惩罚）。
  3. 动态计算：
    • BaseReward 与全网质押量平方根成反比，抑制过度通胀。

示例场景

假设：

  • 验证者 A 的有效余额为 32 ETH，全网质押量为 10,000 ETH。
  • 提议者 B 打包了 10 个见证。

计算：

  1. BaseReward = (32 * 64) / sqrt(10000) = 20.48 Gwei
  2. 验证者 A 奖励：若正确投票，获得 20.48 * 32/10000 ≈ 0.0655 ETH。
  3. 提议者 B 奖励：20.48 * 10 = 204.8 Gwei。

  • 惩罚逻辑：恶意行为（如双重投票）的罚没（Slashing）见 beacon-chain/core/slashing。

投票

委员会分配

1. 核心委员会分配函数：ComputeCommittee

(1) 功能

  • 输入：
    • indices：当前活跃的验证者索引列表。
    • seed：随机种子（由信标链状态生成，确保不可预测）。
    • index 和 count：指定要提取的委员会编号和总委员会数。
  • 输出：
    • 经过洗牌（Shuffle）后，分配给特定委员会的验证者索引子集。

在以太坊信标链的共识机制中，委员会分配（Committee Assignment）和区块提议者（Proposer）的选举是两个独立的过程，但通过协作共同完成区块验证和共识。以下是关键解析：

1. 委员会与提议者的角色区别

角色 职责 选举方式
提议者 打包并广播区块 每个 Slot 随机选 1 个验证者（基于 EffectiveBalance 和 RANDAO 种子）。
委员会成员 对区块进行投票（Attestation） 每个 Epoch 随机分组（基于 ComputeCommittee 洗牌算法）。

2. 委员会分配是否包含提议者？

(1) 默认情况下不包含

  • 委员会分配（ComputeCommittee）仅用于选择投票验证者（Attesters），而提议者是单独选举的。
  • 源码逻辑：
    • 提议者通过 GetProposerIndex 选举（依赖 RANDAO 和有效余额）。
    • 委员会成员通过 ComputeCommittee 分配（依赖洗牌种子 attesterSeed）。

(2) 动态重叠可能性

  • 自然随机性：
    由于两者均基于随机算法，提议者可能恰好被分配到当前 Slot 的某个委员会中，但这是概率性事件，非设计强制。

  • 统计分布：
    若全网有 10 万验证者，提议者被分配到同一 Slot 委员会的概率约为：

    委员会大小（64）活跃验证者数（100,000）≈0.064%活跃验证者数（100,000）委员会大小（64）≈0.064%
